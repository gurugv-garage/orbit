## Symptom (what "slow" looked like)

The control logic was correct early (40+ unit/sim tests green within the first pass), yet
the behaviour kept failing on the real dock. Each round looked like: tweak → rebuild →
OTA (~90 s) → stand in front → read the log → draw a conclusion → discover the conclusion
was wrong → tweak again. The user repeatedly had to correct "it works" claims that were
made off logs while the dock visibly did nothing useful.

## The five iteration-cost drivers (ranked by how much time each ate)

### 1. Diagnosing on the WRONG signal — the single biggest cost
For a long stretch I debugged the **controller** while the actual fault was **upstream and
intermittent**: the on-device detector was bursty (saw the face a few frames, blind for
seconds), and several "failures" were simply **the user not actually in the camera's view
/ at a bad angle**. I attributed those to control bugs and "fixed" things that weren't
broken. The loop only broke when I added **ground-truth instrumentation** (WS detection
telemetry: framesIn / detector-passes / hits / lastFace, + an on-device indicator) — which
immediately separated "camera stalled" vs "gate stuck" vs "detector blind" vs "nobody was
there." **Cost:** the majority of the wasted rounds. **Lesson:** for an embodied task,
build the telemetry that distinguishes layers BEFORE tuning anything. Don't tune blind.

### 2. Trusting the log over the lived behaviour (eroded user trust)
I repeatedly read the station log tick-by-tick and announced "it works" while the user
watched the dock do nothing — or follow the wrong thing. The log showed *a* face being
tracked; I assumed it was the user. It wasn't. This wasn't just wasted time, it spent
**trust** — the user had to say "so how can I trust it" before I switched to verifying from
what they could see. **Lesson:** the arbiter is the lived behaviour + ground-truth
instrumentation the user can also see, NOT a hopeful log line. State uncertainty plainly;
don't declare success from a proxy.

### 3. Sim green ≠ hardware works (false confidence)
40+ unit/sim tests passed throughout and never caught the real bug: the controller's
overshoot, which lived in its **interaction with real detection lag** (it grew its step on
a stale error and flung the head ~56° off the person). Sim modelled instant, clean
detection; the bug needed laggy, bursty detection to appear. Every "tests pass, should be
good" was false confidence that delayed the hardware run that would actually reveal it.
**Lesson:** sim proves LOGIC; only the device proves BEHAVIOUR. Budget the hardware run as
part of "done," and don't let green tests substitute for it.

### 4. The deploy loop made every iteration expensive (~90 s + restart)
Each controller change that touched the phone needed: bump versionCode → Gradle build →
OTA build → announce → ~90 s silent download+install+restart → reconnect → re-test. With
8 cycles that's a large fixed tax, and it punished the "tweak-and-see" style that drivers
1–3 forced me into. Two compounding annoyances: (a) the parent **brain session would
idle-close and kill the task** mid-test (looked like a crash; wasn't); (b) the dock's own
ambient conversation kept **yielding the body** (the yield/cooldown firing correctly, but
interrupting tests). **Lesson:** keep control logic SERVER-SIDE where possible (a station
restart is seconds, not a 90 s OTA) — most of the real iteration was indeed server-side and
cheap; the expensive cycles were the phone-side cadence/indicator changes. Minimise
phone-round-trips; batch phone changes.

### 5. Chasing precision the machine can't deliver (premature sophistication)
Early controllers assumed a usable degrees↔pixels mapping and "push harder if the error
isn't shrinking" (proportional gains, then an adaptive grow/shrink step). On flaky servos +
laggy detection that logic actively *caused* the overshoot. The fix was to STOP trying to
be precise: small fixed step toward the error's SIGN, then settle for a fresh look — a
deliberately dumber, robust loop. **Lesson:** for a physical actuator, start with the
simplest robust-to-flakiness loop; add sophistication only if a measured need appears.
Sophistication built on an untrustworthy signal is negative value.

## What actually broke the logjam (the method that worked)

1. **Telemetry that attributes the fault to a layer** (WS detection counters) — turned
   "it doesn't see me" from a theory into a measured fact.
2. **An on-device indicator the user can watch** (LOCK/TRACK/WAIT/SEARCH) — made the
   lived behaviour legible to both of us, so "works/doesn't" stopped being a debate.
3. **Judging AGGREGATE over a soak, not single runs** — flaky hardware means one run
   proves nothing; `ff-validate.mjs` over a 3-min soak gave real pass/fail.
4. **Listening to the user's framing** — "these are physical machines, they're flaky";
   "trust the indicator not the logs"; "lock and wait, don't move on." Each reframed the
   problem correctly faster than my own theorising did.

## The checklist this buys us (do these FIRST on the next embodied behaviour)

- [ ] Build layer-attributing **telemetry** (per-stage counters) before tuning the controller.
- [ ] Add a **user-visible indicator** of the behaviour's live state; treat it as the arbiter.
- [ ] Keep control logic **server-side**; only round-trip the phone when unavoidable, and batch it.
- [ ] Start with the **simplest robust loop** (sign + small fixed step + settle); no precision
      until a measured need exists.
- [ ] Judge on a **soak aggregate**, never a single flaky run; never claim "works" from a log.
- [ ] Expect, and design around, **bursty/laggy perception** from the start (hold-through-gaps).
