# Dock device-control — the dock acts on its own phone (Shizuku)

How the dock gains the ability to **act on its own Android phone** — change
settings, launch/drive other apps, take screenshots, simulate input — driven by
the LLM brain. Rides the conversational brain's tool-RPC path
([decision-traces/server-brain-impl.md](decision-traces/server-brain-impl.md))
and is one instance of the brain-extension doctrine
([decision-traces/server-brain-selfmod.md](decision-traces/server-brain-selfmod.md)).

> **Status — a proposed design, not yet built.** Nothing below is wired. This
> doc records the mechanism choice (and the three rejected alternatives), the
> safety model, and the as-designed wire path, so the build can follow it. The
> tool will be **off by default** behind a config gate — handing an LLM a shell
> on a real device is a deliberate switch, not a default.

> **Scope rule:** the actuation surface is **Shizuku**, a third-party privileged
> daemon, not anything orbit invents. The design work is *exposing* Shizuku
> behind orbit's existing tool-RPC seam + a phone-side allowlist, not adding new
> transport. The new tool reuses the **exact** path `set_face` already uses
> (`rpc.call` → `agent`/`tool-call` frame → phone → `tool-result`).

## 0. The problem, and why "run shell in the app" doesn't solve it

Goal: a desk companion that *does things* on its phone — not just face + voice.

The dock app is an **unprivileged, sideloaded app in its own UID sandbox**.
That sandbox is the whole point of Android security, and it means:

- `Runtime.exec("…")` **from the app** runs with only the app's own permissions.
  It can `ls`, `cat /proc/…`, run toybox — nothing it couldn't already do via
  Kotlin APIs, and **nothing** that touches other apps, system settings, input,
  or screenshots. Useless for "control the device".
- To act on the *device* you must execute at a higher privilege than the app
  has. There are exactly three ways to get there.

## 1. Mechanism: the three ways to get device privilege (and the pick)

| Mechanism | Power | Cost / catch |
|---|---|---|
| **Shizuku** ✅ | A service started at **adb/shell UID (2000)** that the app calls into. Can run `input`/`am`/`pm`/`settings`/`wm`/`screencap`/`dumpsys`/`cmd` — i.e. anything `adb shell` can. | Needs a **one-time start per boot** via adb (unless rooted). Fine for a tethered dock phone. |
| Root (Magisk) | Full `su`. Survives reboot, auto-starts. | Requires rooting the phone, breaks Play Integrity, fragile across updates. Only for a dedicated/throwaway dock. |
| Accessibility service | Read the screen UI tree + perform taps/gestures. Survives reboot, no adb. | **Cannot** run `pm`/`am`/`settings` — no shell. Insufficient for "control the device". |
| (in-app `Runtime.exec`) | App UID only. | Rejected — see §0. |

**Picked: Shizuku.** It gives adb-level power without root, and the dock phone is
tethered + we already adb into it from skills, so the per-boot arm is a non-issue.

### 1a. Shizuku API reality (verified vs `RikkaApps/Shizuku-API`)

Shizuku hands the app a **binder service at shell UID — not a literal persistent
shell.** Two facts that shape the build:

- **Permission flow:** `Shizuku.pingBinder()` (is the daemon up?) →
  `checkSelfPermission()` → `requestPermission(reqCode)` +
  `addRequestPermissionResultListener(...)`.
- **Running a command:** `Shizuku.newProcess(String[] cmd, String[] env, String dir)`
  exists and runs at shell UID, but it is `@hide` → **must be called by
  reflection**. It is the pragmatic path for "run a command". The
  officially-blessed alternative is a **UserService** exposing an AIDL interface
  (no reflection, supports long-lived privileged calls) — heavier; **deferred**
  as the fallback if `newProcess` reflection proves brittle across Android
  versions.

## 2. Surface: a station brain tool, not an on-phone capability

Per [CLAUDE.md](../CLAUDE.md), **the brain lives in orbit-station**, the phone
is the FACE + (now) the actuator. So device-control is exposed as a **station
brain tool** that RPCs *down* to the phone — identical in shape to `set_face`.
The phone never decides to run a command on its own; the LLM (in the station)
calls the tool, the phone executes and reports back.

### 2a. As-designed wire path (reuses the existing tool-RPC plumbing)

```
LLM → tools.ts  run_device_command.execute()
    → rpc.call({dock, cap:'device-control', name, args})          [brain/rpc.ts RpcBroker]
    → bus publish {topic:'agent', kind:'tool-call', toAddr:{dock,component}}
    → hub delivers to the phone peer serving cap 'device-control'  [docks/directory.ts resolveCap]
    → StationLink.handleInbound → RemoteBrain.onAgentFrame('tool-call')
    → RemoteBrain.onToolCall()  dispatch by name
    → ShizukuShell.run(cmd)   (allowlist gate → Shizuku.newProcess via reflection)
    → publishCritical('agent','tool-result',{reqId,content,isError})
    → RpcBroker matches reqId → resolves → tool returns output to the LLM → model narrates
```

No new transport, no new routing: `resolveCap` is generic, so the only new wire
fact is a new **capability tag `device-control`** the phone advertises in its
`hello` `caps` — and it advertises it **only when Shizuku is actually armed**, so
the station won't route the tool to a phone that can't serve it.

## 3. Safety model: allowlist-gated arbitrary shell

The tool is **one free-form `command` string** (flexible — the LLM can compose
`input`, `screencap`, etc.), but the **phone enforces a binary allowlist** before
executing — the brake on the LLM:

- **Allow** (curated): `input`, `am`, `pm`, `settings`, `wm`, `screencap`,
  `dumpsys`, `cmd`, `svc`, `monkey`, `content`, `getprop`.
- **Reject** everything else, and (start strict) reject shell metacharacters
  (`;`, `&&`, `|`, backticks) unless explicitly allowlisted — a rejected command
  returns an `error: …` result, never executes.
- Output is captured (stdout+stderr+exit code), truncated to a sane cap (~4 KB).

Two further brakes:

- **Config gate** (`brainDeviceControl`, default **off**): the tool is only in
  the brain's tool array when enabled, consistent with the `brain*`
  config-profile pattern.
- **Arming gate**: if Shizuku isn't armed, the cap isn't advertised and/or
  `ShizukuShell.run` returns `error: shizuku not armed`.

## 4. Arming: one-time-per-boot start over adb

Shizuku needs a start each boot (unless rooted). Reuse the skills' existing
"assume the single connected adb device" convention
(`.claude/skills/stress-dock-pipeline/bin/lib.sh`) — no new device discovery:

1. **One-time, manual:** install the Shizuku app on the dock phone.
2. **Per boot:** run Shizuku's start script over adb —
   `adb shell sh /storage/emulated/0/Android/data/moe.shizuku.privileged.api/start.sh`
   (exact path is printed by the Shizuku app). Wrap in a small helper the dock
   workflow calls on connect/boot.
3. **First-grant (debug):** a debug-only `DebugTestReceiver` action can kick
   `Shizuku.requestPermission` for hands-free first approval.

## 5. Build outline (where each piece slots in)

**Station** (`orbit-station/server/src/modules/brain/`):
- `schemas.ts` — `runDeviceCommandSchema` (one `command` string + optional
  `timeoutMs`) modeled on `setFaceSchema`.
- `tools.ts` — a `tool('run_device_command', …)` entry modeled on `set_face`
  (tools.ts:404-416), config-gated by `brainDeviceControl`. Uses existing
  `deps.rpc` / `deps.dock` / `deps.getTurnContext` (no new deps), `cap:'device-control'`.

**Phone** (`node-dock/app`):
- Gradle: `shizuku-api` + `shizuku-provider` (`libs.versions.toml` + `build.gradle.kts`).
- `AndroidManifest.xml`: the `rikka.shizuku.ShizukuProvider` `<provider>`.
- **new** `…/dock/device/ShizukuShell.kt`: `isReady()`, allowlist gate, `run()`
  via `Shizuku.newProcess` reflection, permission request/listener. (Its own
  holder — `DockTools` has no `Context`.)
- `RemoteBrain.onToolCall()` (RemoteBrain.kt:562-612): a `"run_device_command"`
  branch → `ShizukuShell.run`; result via the existing `tool-result` publish.
- `StationLink.kt:196`: conditionally append `"device-control"` to `caps`.
- `DebugTestReceiver.kt` (debug): an arm/`DEVCTL` action for phone-only testing.

## 6. Verification (end-to-end, when built)

1. Arm Shizuku; confirm `ShizukuShell.isReady()`.
2. Phone-only smoke: `adb … DEVCTL -e command "screencap -p /sdcard/_x.png"` →
   `adb pull` — proves shell-UID execution.
3. Allowlist: a denied command (e.g. `rm -rf …`) → `error:` result, no execution.
4. Routing: `GET /api/docks` shows `device-control` in caps iff armed.
5. Full loop: enable `brainDeviceControl`; drive a turn ("take a screenshot of
   yourself"); confirm in the obs trace tool-call → tool-result → narration, and
   the artifact on the phone.
6. Builds: `cd orbit-station && npm run build`; `cd node-dock/app &&
   ./gradlew :app:installDebug && ./gradlew test`.

## 7. Decision log

- **Shizuku over root/accessibility/in-app-exec.** In-app exec can't leave the
  app sandbox; accessibility can't run `pm`/`am`/`settings`; root breaks Play
  Integrity and is fragile. Shizuku = adb-level power, no root, and the dock is
  tethered so per-boot arm is cheap.
- **Station brain tool, not an on-phone capability.** Keeps the brain in the
  station (CLAUDE.md) and reuses the `set_face` tool-RPC path verbatim — zero new
  transport. Phone is actuator only; it never self-initiates a command.
- **Allowlist-gated arbitrary shell over a fixed action menu.** One `command`
  string keeps the LLM flexible; the phone-side binary allowlist + config gate +
  arming gate are the brakes. A fixed `take_screenshot/set_volume/…` menu was
  considered safer but too rigid for an exploratory companion.
- **`newProcess` (reflection) now, UserService (AIDL) later.** `newProcess` is
  the minimal path to "run a command"; the AIDL UserService is the clean,
  blessed path and the documented fallback if reflection gets brittle.
- **Off by default (`brainDeviceControl`).** Device shell for an LLM is opt-in,
  matching the `brain*` config-profile doctrine.
