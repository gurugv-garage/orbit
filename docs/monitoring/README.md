# Monitoring — the ~daily health read

A routine an LLM (or a human) runs against the STORED data to answer: is the dock
still behaving, is anything regressing, and is anything costing what it shouldn't.

**This runs no live tests and drives no hardware.** It reads observability turns,
`conv_events`, perception snapshots, and cost rollups — so it is safe at any hour,
and re-running it for a past window reproduces that window (modulo retention:
conv_events 14d, request ring ~1wk, clips/frames 200MB each).

To DRIVE the dock (real audio, injected utterances, replays) see
[debugability.md](../debugability.md). To understand any signal below, see
[observability.md](../observability.md).

## How to run

```bash
node docs/monitoring/collect.mjs --hours 24                 # last 24h, dock-redmi
node docs/monitoring/collect.mjs --hours 24 --dock dock-x   # another dock
node docs/monitoring/collect.mjs --hours 24 --to 1784700000000  # a past window
```

It prints one JSON blob. When a number needs evidence behind it, run a probe:

```bash
node docs/monitoring/probe.mjs suppressed-barges   # did suppressed onsets have real speech?
node docs/monitoring/probe.mjs dropped-speech      # re-transcribe dropped clips (needs :8078)
node docs/monitoring/probe.mjs endpoint-lag        # do barges endpoint during the reply?
node docs/monitoring/probe.mjs tts-delay           # LLM-end → speech-start
node docs/monitoring/probe.mjs followup-chains     # dock talking to room noise
```

The run then has three jobs:

1. **Score** each check below against its threshold (§ Bug watchlist).
2. **Compare** against the last few entries in [RESULTS.md](RESULTS.md) — a
   metric moving is often more informative than its absolute value.
3. **Append** a dated entry to RESULTS.md in the documented format, including
   anything you investigated and what you concluded.

For anything that trips a threshold, pull the evidence before judging: the
incident bundle for the window (`GET /api/observability/incident?dock&from&to&format=md`)
and, for hearing issues, the actual audio — every dropped clip is kept at
`GET /api/perception/utterance-audio/<dock>/<audioStartMs>`.

### Look beyond the checklist

The tables below are the things we already know to watch — they are a floor, not
a ceiling, and they will always lag the system. **Spend part of every run
looking for something nobody listed yet.** The richest sources:

- **Read a few turns end to end**, not just the aggregates — pick the slowest
  turn, a silent one, and one random conversation, and follow them through the
  Timeline. Most bugs found so far were noticed this way, not by a metric.
- **Ask why any number is *oddly round or oddly stable*** — a metric that never
  moves is often a code path that never runs (that is how dead code and
  hardcoded provenance strings were found).
- **Correlate across categories** — cost spikes with turn counts, drops with
  time of day, silence with trigger kind. The interesting findings usually live
  in a ratio, not a count.
- **Trust a transcript that reads wrong.** Garbled STT during barge ("Wait,
  start by M to Rickson") is a real signal about audio quality even when no
  threshold fires.
- When you find something new, write a probe for it in `probe.mjs` if it will
  be asked twice, and add a row to the watchlist with a threshold you can
  defend.

## What to look at, by category

### 1. Hearing — did it hear people correctly?
- `hearing.dropsVoicedFraction` — clips discarded below the voiced floor. Some
  is normal (room tone); a spike means either a noisy day or a gate problem.
- `hearing.dropsWhileSpeaking` — the barge-risk band (strict 35% floor applies).
  These are the ones most likely to be a real interruption thrown away.
- `hearing.empty` — VAD endpointed audio but STT returned nothing. A few is
  normal (coughs, doors); a spike suggests mic/AEC trouble or a sick sidecar.
- `hearing.unreachable` / `decodeFail` — should be **0**. Non-zero = the dock was
  deaf or the audio decoder was failing; both are user-visible outages.
- `hearing.sttLagMsP50/P90` — "you stopped talking → transcript ready". Drives
  how fast anything downstream can react.

### 2. Attention — did it respond to the right things?
- `admission.ranTurn` vs `skipNotAddressed` — the ratio IS the addressed-gate
  behaviour. Lots of skips is usually correct (room chatter), but a day of
  near-zero `ranTurn` with people present means it stopped responding.
- `admission.skipStale` — queued speech that expired before it could run. Each
  one is someone who spoke and got nothing back.
- `admission.via` — the trigger mix. A sudden jump in `followup-window` or
  `busy-drain` is the followup-chain shape (the dock talking to room noise);
  a jump in `mood:*` means it is initiating a lot on its own.

### 3. Interruptions — could people stop it?
- `barge.skipSelfMotion` vs `barge.holds` — how often the servo-noise guard
  suppressed a pause versus allowed one. **Skips exceeding holds means most
  interruptions never paused the reply.**
- `barge.yielded` — sustained interruptions that aborted the reply without a
  clean stop word.
- `barge.resumedNoWords` — barges that turned out to be noise and correctly
  resumed (added 2026-07-23; should be common in a noisy room).
- `barge.stopDismiss` / `stopPause` — explicit voice stops that worked.

### 4. Turn quality — were the replies any good?
- `turnQuality.erroredTurns`, `toolErrors` — should be **0**.
- `turnQuality.silentTurns` — finished cleanly but said nothing. A few is by
  design (overheard framing); many means it has gone mute.
- `turnQuality.unfinishedTurns` — turns with no terminal state; usually a crash
  or a restart mid-turn.
- `turnQuality.ttftMs*`, `firstSpeechMs*` — responsiveness. `firstSpeechMs`
  is what the user actually feels.
- `stoppedReasons` — how replies ended early (see the ⏹ badges in obs).

### 5. Cost — is anything running away?
- `cost.totalUsd`, `usdPerTurn` — the headline. Compare to the last runs, not
  to an absolute ideal.
- `cost.byUsecase` — which role is spending. `Conversation` dominating is
  normal; a perception role (`audio-enricher`, `summary`, …) climbing is the
  shape of the 2026-07-13 enricher incident ($4.4/day from a wrong model +
  thinking left on).
- `cost.cachePct` — prompt-cache hit rate. Structurally it can be high (~96%
  observed); Google's implicit cache is best-effort, so treat sustained 0%
  across a busy day as a regression in prompt stability, not a bad hour.
- `cost.last7Days` — the trend line; a step change is the thing to catch.

## Bug watchlist — thresholds and what they mean

Thresholds are **triage triggers, not verdicts**: crossing one means *go look*,
not "there is a bug". Under the project's bug bar, a bug needs a reproduction
and a stable number, so a tripped threshold becomes an investigation with
evidence attached, recorded in RESULTS.md.

| Check | Healthy | Investigate when | Why it matters |
|---|---|---|---|
| `hearing.unreachable` | 0 | ≥1 | the dock was DEAF; users get silence |
| `hearing.decodeFail` | 0 | ≥1 | audio silently vanishing before the VAD |
| `turnQuality.erroredTurns` | 0 | ≥1 | an LLM/tool failure reached the user |
| `turnQuality.toolErrors` | 0 | ≥2 | a capability is broken |
| `turnQuality.unfinishedTurns` | 0-1 | ≥3 | crashes/restarts mid-turn |
| `admission.skipStale` | <5% of heard | >10% of heard | people spoke and got nothing |
| `barge.skipSelfMotion` vs `holds` | skips < holds | skips > holds, or `probe.mjs suppressed-barges` >30% with speech | **CONFIRMED bug (2026-07-23)**: gestures suppress the barge pause — 71% of suppressed onsets had real speech |
| `hearing.dropsWhileSpeaking` | low, and empty on inspection | any that transcribe to real words | a real interruption was thrown away |
| `turnQuality.silentTurns` | <10% of turns | >25% of turns | the dock has gone quiet on people |
| `turnQuality.firstSpeechMsP50` | <5s | >7s, or +50% vs recent runs | replies feel sluggish |
| `hearing.sttLagMsP90` | <1s | >2s | STT is falling behind; barge/stop react late |
| `cost.usdPerTurn` | stable | +50% vs the last runs | a model/prompt change is costing money |
| `cost.byUsecase` non-Conversation | small | any role >25% of spend | background work is outspending the user |
| `cost.cachePct` | >30% on a busy day | 0% across a busy day | prompt prefix stability regressed |

### Known-open issues to keep watching
Track these each run even when they don't trip a threshold; they are the
suspicions with evidence but no fix yet.

- **Self-motion mute swallowing barge onsets — CONFIRMED, unfixed.** The 1.8s
  guard re-arms with every gesture and the dock gestures throughout a reply, so
  the barge pause is unavailable for much of any animated turn. Measured
  2026-07-23: 17 skips vs 14 holds in 24h, and **12 of 17 (71%) of the
  suppressed onsets had real speech within 8s** — including "Wait, I am
  interrupting you right now. Please pause" and "Actually hold on". Those users
  were talked over. Re-measure with `probe.mjs suppressed-barges`; the fix
  direction is scaling the mute to the actual move (a head tick and a full pan
  are not equally loud) or bypassing it when the onset is well above
  `ONSET_RMS`.
- **Voiced-fraction gate vs quiet far-field speech** — the two-tier floor
  (35% speaking / 10% idle) fixed the worst of it. Periodically re-run the
  clip sweep: transcribe the kept dropped clips and count real words
  (2026-07-23 baseline: 16% of drops carried words at the old flat floor).
- **TTS start delay** — ~470ms median LLM-end → speech-start, p90 1.7s
  unexplained. Watch `firstSpeechMs` minus `ttftMs`.
- **Echo into STT** — garbled transcripts during barge ("Wait, start by M to
  Rickson"). Not fixed by the endpointing work; an AEC/audio-quality issue.

## Writing the run up

Append to [RESULTS.md](RESULTS.md), newest first, using the template at the top
of that file. Rules that keep the log useful:

- **Record the numbers even when everything is fine** — the trend is the
  product; a quiet day is data.
- **Separate observation from conclusion.** "83 skip:not-addressed" is an
  observation; "the addressed gate is too strict" is a claim that needs
  evidence (clips, a repro, a rate).
- **Link the evidence**: turn ids, clip timestamps, and the incident-bundle
  window you looked at, so a later run can re-open exactly what you saw.
- **Note anything that changed in the system** (a deploy, a threshold change, a
  sideload) — most metric jumps have a cause in the commit log.
