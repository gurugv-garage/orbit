# Comms stability — architecture review & hardening plan

> Status: **in progress** — findings from a full review of the comms backbone
> (station WS hub + bus, phone StationLink/RemoteBrain, ESP32 station_link,
> WebRTC/media, HTTP/REST surface), 2026-07-06. Nothing here is fixed yet;
> this doc is the worklist. Items get checked off / moved to decision-traces
> as they land.

The platform's comms are one WebSocket hub (`/ws`) for all peers (phone app,
ESP32 body, browser UI, task processes) + an in-process bus, WebRTC for bulk
media, and REST for console/config/OTA. The topology is right and several
subsystems are genuinely solid (see [What's already solid](#whats-already-solid)).
This doc lists what's broken or fragile, ordered by when it bites: today on
the LAN, when more devices join, and when the station leaves the LAN.

## Verdict in one paragraph

Three things are broken today: the hub's backpressure guard protects the
wrong traffic class (broadcasts shed, directed frames buffer unboundedly);
the ESP32 silently drops any inbound WS frame over ~1 KB (no fragment
reassembly); and device identity/dedup only works for the two hardcoded
`kind`s, so every future device type escapes the newest-wins displacement
that keeps the roster sane. Separately, the whole platform runs with zero
authentication — a conscious trusted-LAN posture, but it is load-bearing in
more places than obvious (OTA build/install endpoints, capability tenancy,
task role), and it hard-blocks the remote-deployment plan.

## A. Broken today (LAN, single dock)

### A1. Directed frames bypass backpressure — the exact case that matters

The hub sheds broadcast frames when a peer's `bufferedAmount` exceeds 1 MB,
but **directed** frames (`to`/`toAddr`) always send:
`server/src/core/websocket-gateway.ts:148-151`. Motion `command`s, `tool-call`s and task
digests are all directed — so a stalled-but-not-dead ESP32 or phone
accumulates unbounded directed sends in Node heap. The 2 s ping / 3-miss
watchdog reaps dead sockets, but not slow ones.

**Fix shape:** per-peer send policy keyed on traffic class (see B4):
telemetry sheds, state coalesces latest-wins, critical delivers-or-errors,
hard cap terminates the peer.

### A2. ESP32 drops any inbound frame > ~1024 B, silently

`on_ws_event` parses `ev->data_ptr/data_len` directly and ignores
`payload_len`/`payload_offset`/`fin`
(`node-dock/body-firmware/dock_body_v0/src/station_link.c:369-372`); the
esp_websocket_client buffer defaults to 1024 B. Larger frames arrive
fragmented, fail `cJSON_ParseWithLength`, and vanish — no log, no error
frame. Everything sent today happens to be small, so this is a landmine:
the first larger presence list or config snapshot disappears untraceably.

**Fix shape:** reassemble using `payload_len`/`payload_offset` (bounded
accumulation buffer, e.g. 8 KB, drop + log beyond), or at absolute minimum
log loudly when `payload_offset > 0`.

### A3. Body's `config` subscription is dead code

The body subscribes to the `config` topic (`station_link.c:225`) but never
sends the `interest` frame the config module requires
(`server/src/modules/config/index.ts:71-74`) and has no `config` handler —
every config frame is dropped at `station_link.c:303`. Either implement
(interest + handler) or delete the subscription so it stops misleading.

### A4. Wi-Fi wedges permanently after 20 failed retries

After 20 consecutive failures the firmware sets `WIFI_FAIL_BIT` and stops
retrying forever — "parks and waits for power-cycle"
(`node-dock/body-firmware/dock_body_v0/src/wifi_sta.c:26-30,72-79`). A
router outage that outlasts the budget = dead body until someone pulls the
plug. **Fix shape:** a timer that re-arms the retry budget every few
minutes.

### A5. No mid-motion failsafe on link loss

On Wi-Fi/WS drop, `bl_motion_tick` keeps interpolating to the last
commanded target — a 30 s sweep in flight runs to completion with no brain
able to intervene (`bodylink_motion.c:303-328`; neither disconnect path
touches motion). Low stakes for a servo neck; wrong default for the rover
or anything with reach. **Decide the policy now:** freeze-at-current-pose
on disconnect (behind a profile flag if the hold-to-target behaviour is
wanted for short moves).

### A6. Phone can't detect a half-open socket

StationLink's only disconnect signal is the incoming-frame loop ending
(`node-dock/app/.../station/StationLink.kt:244`) — no read-idle watchdog,
no missed-heartbeat check. On a silent network drop, `_connected` stays
true and a mid-turn failure is rescued only by the 75 s watchdog
(`RemoteBrain.kt:764`). **Fix shape:** the station pings every 2 s, so
inbound traffic is near-constant — "no inbound frame for ~10 s → force
reconnect" closes this cheaply. Also reset conv-window/interim state on
disconnect (`RemoteBrain.kt:93-110` lingers stale today).

### A7. Fire-and-forget failures on the phone's critical path

`startTurn` fails loudly when `publishCritical` returns false, but
`tool-result`, `speech-status`, `turn-cancel` and confirm-resolution all
ignore the boolean (`RemoteBrain.kt:615-620`, `341`, `317`, `632`). A
dropped tool-result silently stalls the station's tool call until its 10 s
RPC timeout. **Fix shape:** check the result everywhere; retry-once or log.

### A8. `PATCH /api/config` hangs the client on malformed JSON

`JSON.parse(await readBody(req))` with no try/catch
(`server/src/modules/config/index.ts:105`), and `core/http.ts` doesn't wrap
`route()` — a bad body means no response ever. Perception already has the
safe `parseBody` pattern (`perception/index.ts:1326-1330`); reuse it.

## B. Multi-device blockers (more ESP32s / phones / new hardware kinds)

### B1. Identity/dedup breaks for any kind outside a 2-entry map ★ most important foundation fix

Newest-wins displacement — the mechanism guaranteeing at most one live peer
per `(dock, component)` (`hub.ts:330-341`) — only works when the peer's
`kind` maps to a component, and `KIND_COMPONENT` is hardcoded with exactly
two entries (`server/src/core/protocol.ts:165-171`). Any new device kind
gets **no component**: stale ghosts are never displaced on reconnect,
duplicates accumulate, and `resolveCap` picks whichever the roster's
`find()` hits first.

**Fix shape:** devices declare `component` in `hello` (map kept as legacy
fallback); hub warns/rejects a hello with no resolvable component.

### B2. Capability conflicts resolve by luck

`resolveCap` is first-match by iteration order
(`server/src/modules/docks/directory.ts:85-89`), no conflict detection. Two
peers on one dock both claiming `servo` (realistic with multiple ESP32s)
silently split commands nondeterministically. **Fix shape:** detect + log
the conflict at hello time; deterministic tie-break (newest wins, or
explicit priority in hello).

### B3. No protocol version, no wire validation

No version field anywhere in the handshake; the only inbound gate is
"object with a string `t`" (`protocol.ts:148-150`). `subscribe` topics and
`publish` payloads are unchecked — a peer can subscribe/publish to any
topic including `station`/`agent`/`tasks` (`hub.ts:352-356, 394`). zod is a
dependency but unused on the wire. Lockstep upgrades work with two devices;
a fleet on OTA can't. **Fix shape:** `proto: 1` in `hello`/`welcome` now
(costs nothing); topic-membership check on subscribe/publish; zod schemas
at the hub edge for the frames that drive actuators (bodylink, brain, ota).

### B4. No traffic-class concept on the one WS

The single socket carries latency-critical frames (`speak`, `turn-status`,
motion commands), bulk (the base64 turn JPEG when the WebRTC stream is
down — `RemoteBrain.kt:472-479`), and telemetry, with no prioritization; a
big frame head-of-line-blocks critical ones. The macro split is already
right (bulk media on WebRTC; the phone's telemetry-outbox vs
critical-direct split is the correct instinct — `StationLink.kt:154-157,
318-327`). **Generalize it:** tag topics with a class (critical / state /
telemetry / bulk); the hub's per-peer send policy keys off it. This one
mechanism also fixes A1 and gives every future device type a defined
contract. Also: move the fallback turn-JPEG to an HTTP POST off the
critical WS path.

## C. Security posture (trusted-LAN today; hard-blocks remote deployment)

No authentication anywhere:

- no token/origin check on `/ws` (`hub.ts:100`) — `hello` fields (`role`,
  `dock`, `caps`) are trusted, so impersonating a dock component (claim
  `caps:['servo']`, receive body commands) is trivial on the LAN;
- none on any REST route — including `POST /api/ota/:target/build` and
  `/install-adb` (`server/src/modules/ota/index.ts:264-343`), i.e. anyone
  on the network can trigger builds and adb installs;
- `role:'task'` is loopback-by-convention only, not enforced (the loopback
  check in `directory.ts:42-57` is a persistence heuristic, not a
  boundary);
- `readBody` is unbounded in all three copies (`docks`, `config`, `ota`
  modules); WS `maxPayload` sits at the ws default 100 MB, parsed on the
  event loop; no rate limiting anywhere;
- OTA is sha256-only — integrity, not authenticity; artifacts + offers are
  unauthenticated end to end.

Defensible on a home LAN. **Minimum bar before any off-LAN exposure:**
shared secret in `hello` + bearer header on mutating REST (fits the
existing `.env` pattern), loopback enforcement for `role:'task'`, caps on
`readBody`/`maxPayload` (64 KB–1 MB is generous), OTA signing.

## D. WebRTC: LAN-only by construction

No STUN/TURN anywhere: SFU defaults to host candidates
(`server/src/modules/media/sfu.ts:56-58`, `STUN_URL` env unset), phone uses
an empty ICE config (`MediaStreamer.kt:127`, also the TTS loopback
`WebRtcAudio.kt:389`), and browser viewers construct `RTCPeerConnection()`
with **no config plumbing at all** (`web/src/modules/LiveTile.tsx:36`,
`PerceptionStudio.tsx:450`). Off-LAN: signaling connects, media never
flows. Budget for the remote move: a TURN server + browser/phone ICE config
plumbing (server side has the `STUN_URL` escape hatch; clients have none).

Smaller SFU nits: viewer-map key is `browserId|streamId` split on `|` with
no escaping (`sfu.ts:223, 266`); the in-process tap has no backpressure —
Opus decode for STT runs synchronously on the event loop per packet
(`perception/processors/stt-watch.ts:221-246`) — fine at one dock, the
scaling ceiling for several streaming docks.

## E. Smaller items

- **bodylink DESIGN.md has drifted badly from the firmware** — body is a
  station client now (not a WS server); the "no state stream" rule (§1.4)
  is violated by design (2 s `bodylink/state` + on-change); no
  `error`/`BAD_MESSAGE` frames are ever emitted (§1.7/§5.4); velocity-cap
  default 3000 in firmware vs 4000 documented (`bodylink_motion.c:25,33`);
  `applied` payload shape differs (§3.2). One doc pass to stop §3 from
  misleading — station/firmware disagreement on motion numbers has had real
  symptoms before (velocity-cap collapse-to-wiggle incident).
- **Web UI polls where push exists** — worst: Brain's 1.5 s instance poll
  (`web/src/modules/Brain.tsx:1412`), PerceptionStudio's four concurrent
  loops (`PerceptionStudio.tsx:262,366,377,396`). Cosmetic at this scale.
- **Config push is fire-and-forget** (no ack/retry;
  `config/index.ts:41-54`) — recovered only by snapshot-on-interest at
  reconnect. Acceptable; just know the snapshot is the real consistency
  mechanism, so a client that subscribes without announcing interest (A3)
  gets nothing.
- ESP32 emits no `error` frames for malformed/unknown input — silent no-op
  (`station_link.c:303,321,372`). Fold into the A2 fix.

## What's already solid (keep these patterns)

- Address-not-socket identity with newest-wins displacement
  (`hub.ts:326-341`) — state binds to `(dock, component)`, reconnects need
  no re-addressing.
- Both RPC brokers: correlation ids, bounded timeouts, peer-left rejection,
  never-hang (`brain/rpc.ts`, `brain/tasks/capabilities.ts`).
- ESP32 bounded-send + separate TX lock, so a half-open socket can't wedge
  the client or starve the pong watchdog (`station_link.c:48-54`,
  `sdkconfig.defaults` `WS_CLIENT_SEPARATE_TX_LOCK`).
- OTA proof-of-life rollback gate: new firmware must rejoin Wi-Fi **and**
  the station before the slot commits (`station_ota.c:214-230`).
- Preempt/idempotent motion model — command floods are free
  (`bodylink_motion.c:230-264`); control task (prio 5) outranks network
  (prio 4).
- Layered liveness: hub 2 s ping / ~6 s reap; task 10 s silence self-exit.
- Phone: telemetry outbox (DROP_OLDEST) vs critical-direct split; OTA
  auto-apply with inline SHA-256 verify and an app-lifetime download scope.

## Suggested sequencing

1. **WebSocketGateway: directed-frame backpressure + traffic classes** (A1 + B4) — one
   mechanism, protects every current and future peer.
2. **Identity/component fix + capability-conflict detection** (B1 + B2) —
   cheapest now, hardest to retrofit once multiple boards exist.
3. **ESP32 trio**: fragment reassembly/guard (A2), Wi-Fi retry re-arm (A4),
   disconnect motion policy (A5). Plus A3 (dead config sub) while in there.
4. **Protocol version field + hub-edge validation** (B3); phone half-open
   watchdog (A6) + critical-send checks (A7); config parse fix (A8).
5. **Auth minimum + ICE/TURN plumbing** (C + D), gated on the
   remote-deployment decision.
