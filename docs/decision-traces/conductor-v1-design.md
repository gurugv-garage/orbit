## 1. What v1 is (one paragraph)

A **per-dock Conductor**: a cheap, session-independent loop that reads a small **config of
tuning variables**, and for each **conducted thing** decides ŌĆö by structured rules over those
tunings + cheap reads of the world (presence, conversation state, the snapshot stream) ŌĆö
whether it should be **armed**, and when armed, whether to **run or stop**. It never
runs an always-on LLM. Body contention is already solved (the lease). v1 conducts exactly two
things ŌĆö **faceFollow** (a *task*: a periodic look-around window, lease-arbitrated body) and
**wakeUp** (a *behaviour*: always-on "hey orbit" ŌåÆ "did you call me?", hardcoded in the brain,
no body) ŌĆö a deliberate contrast that proves the conductor governs both a spawned task and an
in-place behaviour.

## 2. The four things it adds (unchanged from the vision ┬¦4 ŌĆö restated for v1)

1. **Session-independent lifetime** ŌĆö runs dock-scoped, including when nobody's talking.
2. **Declarative arming** ŌĆö *armed ŌēĀ running.* A tuning-driven rule starts the thing only when
   the world warrants, stops it when it doesn't.
3. **A stable named handle for an intent** ŌĆö you toggle "faceFollow", not the churning task
   instances under it.
4. **A governable surface** ŌĆö enable/disable/reconfigure conducted things (REST + console),
   plus a manual **override** (Run now / Stop / Auto) that pins a thing on/off past its rule.

## 3. The model ŌĆö three small pieces

```
  Conductor (per dock)            ŌåÉ the cheap loop; ticks ~1 Hz
    reads: ConductorConfig         ŌåÉ the TUNING VARIABLES (per dock)
    reads: world (cheap)           ŌåÉ presence, convState, lease, ALL tasks, own state
    for each Conducted thing:
        decide(tunings, world) ŌåÆ DESIRED: off | running
        enact(desired) ŌĆö per the thing's KIND (below)

  Conducted (a registered descriptor ŌĆö NOT new runtime)
    { name, kind: 'task' | 'behaviour', taskName?, priority?, instrumentedAt?,
      decide(tunings, world) ŌåÆ desired, defaults }
      // kind:'task'      ŌåÆ taskName: spawn/stop that supervisor task (e.g. faceFollow)
      // kind:'behaviour' ŌåÆ instrumentedAt: where it's hardcoded; enact toggles that
      //                    in-place hook on/off (e.g. wakeUp in the brain)

  ConductorConfig (tuning variables, per dock, live-editable)
    { <name>: { enabled, ...knobs } }   e.g. faceFollow.activateAfterMs, wakeUp.phrase
```

**Two KINDS ŌĆö task vs behaviour.** A conducted thing is one of:
- **`'task'`** ŌĆö a *generic spawnable process*. Its work is a supervisor task (faceFollow ŌåÆ
  the `face-follow` task, lease-arbitrated). `enact` = supervisor `start`/`stop` by `taskName`.
- **`'behaviour'`** ŌĆö a *reaction hardcoded in-place at design time*. Its work is a tiny hook
  woven directly into the relevant code path (wakeUp ŌåÆ a wake-phrase check inside the brain's
  addressed-final handler). `enact` = toggle that in-place hook on/off via a narrow callback.
  No process, no re-plumbing of signals the station already has. **Its descriptor records
  `instrumentedAt`** ŌĆö the file + function where it's woven ŌĆö so the UI/docs can always point
  to exactly where the behaviour lives (e.g. `brain/index.ts: matchesWake() in onAddressedFinal`).

This keeps wakeUp honest (a one-line check shouldn't be a process) while faceFollow stays a
real task. The **governance is identical** ŌĆö both are "a named intent the conductor turns
on/off + tunes by a rule," and both get the manual override (┬¦7). What actually differs between
the two kinds ŌĆö and what *doesn't* ŌĆö is ┬¦3b, written deliberately loose so we don't paint
ourselves into a definition.

**`decide` is the only thing-specific logic.** A **pure function**
`(tunings, world) ŌåÆ 'off' | 'running'` ŌĆö arithmetic / rules over the tunings + cheap world
reads. No I/O, no LLM, unit-testable like the gate. This is the **seam your "code logic later"
sits at**: today `decide` reads number/bool tunings; a complex thing can later carry a
richer `decide` (still a cheap pure fn) without changing the conductor.

### 3a. What `world` contains ŌĆö the conductor's full picture (built now, used lightly)

The conductor is aware of **more than presence**. `world` is assembled cheaply each
re-evaluation and passed to every `decide`:

```
  world = {
    present: bool, lastPresenceMs: number,        // perception (presence + the snapshot stream)
    identity: string | null,                       // recognized person, if any
    listening: bool, turnActive: bool,             // conversation state
    bodyHolder: { holder, priority } | null,       // the lease ŌĆö who drives the body now
    tasks: [                                        // EVERY running instance on this dockŌĆ”
      { name, instanceId, initiator: 'user'|'brain'|'self', ageMs }
    ],
    self: { /* the conductor's own maintained state: last decisions, since-when */ },
    now: number,
  }
```

The **task picture is deliberately rich**: which tasks run, **who started each** (a user via
the brain, the brain autonomously, or the conductor/`self`), and **how long** (`ageMs`).
v1 uses this *lightly* (e.g. faceFollow stays off while a live turn is active), but the signal
is THERE so a thing's `decide` ŌĆö and the v2 learning task ŌĆö can reason over it (e.g. "a
USER-initiated body task is running ŌåÆ don't arm faceFollow"; later "task X has run 5 min,
likely stuck"). Building the input shape now means v2 adds policy, not plumbing.

### 3b. Behaviour vs task ŌĆö the one real difference

The honest, load-bearing difference is one axis: **a task is DECOUPLED, a behaviour is
INSTRUMENTED.** faceFollow is a task (a standalone "follow a face" capability, indifferent to
who armed it); wakeUp is a behaviour (woven into the brain's `onAddressedFinal`, on a stream
that only exists in-process there). Everything else ŌĆö process/no-process, generic/bespoke ŌĆö is
a *consequence*, not the definition, so we don't bake it in. The full reasoning, why we keep it
thin, and the door to collapsing the two into one concept later, are in
[task-vs-behaviour.md](../task-vs-behaviour.md).

## 4. The two v1 conducted things (the proving pair)

Chosen as a **deliberate contrast**: one is a **task** that drives the BODY (lease-arbitrated),
one is a **behaviour** that drives the CONVERSATION (no body, hardcoded in the brain) ŌĆö so v1
proves the conductor governs *both* a spawned task and an in-place behaviour.

### faceFollow ŌĆö a periodic "look around" window (TASK)
The "idle 5 min ŌåÆ activate, run 15 min ŌåÆ sleep" intent. **Idle = no conversation** (no active
turn / not listening) ŌĆö presence does NOT gate it (it can search for someone when it activates).

```
  decide(t, world):
    if !t.enabled: return 'off'
    if world.turnActive || world.listening: return 'off'   // never during a conversation
    // ACTIVE WINDOW state (held in the conductor's self-state per thing):
    //  - not active + conversation-idle Ōēź t.activateAfterMs (5 min)  ŌåÆ ACTIVATE (start a window)
    //  - active + window age Ōēź t.runForMs (15 min)                   ŌåÆ DEACTIVATE (sleep)
    //  - after deactivating, won't reactivate until another full idle stretch
    return active ? 'running' : 'off'
```
`kind:'task'`, `taskName:'face-follow'`, `priority:30`. Runs the `face-follow` task at lease
priority 30 while the window is open. The 5-min idle clock + 15-min run cap live in the
conductor's per-thing `self` state (so `decide` stays pure ŌĆö it's handed
`idleMs`/`windowAgeMs` in `world`/`self`).

### wakeUp ŌĆö always-on "hey orbit" (BEHAVIOUR, hardcoded in the brain, no body)
**`instrumentedAt`: `brain/index.ts: matchesWake() in onAddressedFinal ŌåÆ session.wake()`.**
```
  decide(t, world):  return t.enabled ? 'running' : 'off'   // always active when enabled
```
When the dock is **NOT listening** and a final STT transcript matches the wake phrase
("hey orbit"), wakeUp **speaks "did you call me?" and opens the listening window** (`tapOpen`).
No body, no lease.

**Why this is a BEHAVIOUR, not a task (resolved in the build).** Final transcripts arrive via
`TranscriptApi.onFinal`, which is **single-consumer** (the brain registers it once) ŌĆö the STT
stream is consumed in-process by the brain and is *not* exposed to spawned task processes.
Rather than add a transcript fan-out + a whole process to host a one-line phrase check, wakeUp
is woven **directly into the brain's existing addressed-final handler** ŌĆö that's the definition
of a behaviour. The conductor holds the tunings (phrase/prompt/enabled) and toggles the hook via
the brain's `WakeApi.setWakeConfig`; the *match* (`matchesWake`) lives where the finals already
are. As built:
  - `brain/index.ts: matchesWake(text, phrase)` ŌĆö the lenient phrase match (unit-tested in
    `brain/wake.test.ts`).
  - In `onAddressedFinal`: when the dock is **not listening** and the final matches the wake
    phrase, call `session.wake(prompt)` ŌåÆ `tapOpen()` + enqueue an autonomous turn that speaks
    the prompt verbatim.
  - The conductor's `setBehaviour('wakeUp', on, tunings)` effect calls `setWake(dock, ŌĆ”)` to
    arm/disarm + push phrase/prompt. Off by default until the conductor arms it.

This does NOT overlap the brain's "addressed decision" (which decides whether a heard utterance
becomes a turn *once a window is open* ŌĆö a different thing from *waking from idle on a phrase*).
The old on-phone Porcupine/Silero wake pipeline is **dead** (disabled / broken since the
server-STT cutover; cleanup tracked separately) ŌĆö see the verified note below.

**Verified (2026-06-26, in code) ŌĆö the dead on-phone wake path:** the dock waking on a phrase
did NOT work before this, because of a **dormant legacy on-phone wake pipeline** left over

lives now). Cleanup of the dead on-phone path is tracked separately so it doesn't entangle the

Between them these prove every mechanic: a **task** that's **conditionally windowed** by time
+ conversation state (faceFollow), a **behaviour** that's **always-on** when enabled (wakeUp),
start/stop a spawned task, a body task mediated by the lease, a hardcoded non-body behaviour,
the override, and "off when disabled" ŌĆö all tuned live.

## 5. Tuning variables (v1) ŌĆö the "learned later" surface

Per-dock config block `conductor.<dock>` (lives in the existing config registry, so it's
already push-on-change + console-editable):

```
  faceFollow.enabled         bool   default true
  faceFollow.activateAfterMs number default 300000   (5 min no-conversation ŌåÆ activate a window)
  faceFollow.runForMs        number default 900000   (15 min, then deactivate until idle again)
  wakeUp.enabled             bool   default true
  wakeUp.phrase              text   default "hey orbit"   (the wake phrase to match in STT)
  wakeUp.prompt              text   default "did you call me?"
  tickMs                     number default 1000     (conductor cadence)
```
These are the knobs in the **Conductor tab** ŌĆö editable in real time (the conductor reads the
live config each tick, so a change applies on the next tick).

**Why this is the right "start simple":**
- It's just **data** ŌĆö sensible defaults, console-editable, hot-applied (the conductor reads
  the live config each tick). No learning needed to ship.
- **The learning door (v2, NOT now):** an episodic *reasoning task* the conductor spawns
  reviews history (who's around, when, what was useful) and **writes back** these same tunings
  (e.g. flips `faceFollow.enabled` per time-of-day, raises `idleReset.afterMs`). The conductor
  reads them next tick ŌĆö *learned conducting with zero core change.* v1 just makes sure the
  tunings are the single point everything flows through.
- **The code-logic door (later):** a tuning that's normally a value can become a small policy
  fn injected into `decide` for a complex behaviour ŌĆö the `decide` signature already allows it.

## 6. Lifecycle + the cheap loop (the actual algorithm)

The decision logic is one function, `reconcile(dock)` ŌĆö assemble `world`, ask each conducted
thing `decide`, and start/stop to match. It is **re-evaluated by two triggers** (same
`reconcile`, so they can't disagree):

- **CADENCE** ŌĆö a ~1Hz tick. Catches anything events would miss + time-based rules
  (idle-reset's `afterMs`). **v1 ships cadence-only** ŌĆö it already reacts within ~1s. A manual
  override also calls `reconcile` immediately so Run-now/Stop apply without waiting a tick.
- **EVENT** *(named here, DEFERRED to v2)* ŌĆö re-`reconcile` immediately on a decision-changing
  event (a task starts/stops, presence changes, a turn begins/ends) for instant
  responsiveness. v1 doesn't wire events; the loop is written so adding them later is just
  "call `reconcile` from these subscriptions" ŌĆö no logic change.

```
  reconcile(dock):                                // cadence now; event-or-cadence in v2
    cfg   = config.conductor[dock]                // live tunings
    world = assembleWorld(dock)                   // ┬¦3a ŌĆö presence, conv, lease, ALL tasks (+initiator,age), self
    for c in conducted:
        desired = override(c.name) ?? c.decide(cfg[c.name], world)  // override pins on/off; else pure decide
        if c.kind === 'task':                     // a spawned process
            running = isTaskRunning(c.taskName)
            if desired==='running' && !running: startTask(c.taskName)   // arm ŌåÆ run
            if desired!=='running' &&  running: stopTask(c.taskName)     // disarm ŌåÆ off
        else: // c.kind === 'behaviour'           // a hardcoded in-place hook
            setBehaviour(c.name, desired==='running', tunings)           // toggle the in-place hook
    log a [cond] line ONLY when a thing's desired/running transitions
```

- **Override** (the Run now / Stop / Auto buttons): a per-`(dock,name)` pin ŌĆö `'run'` forces
  desired `running`, `'off'` forces `off`, `'auto'` clears the pin (back to `decide`). It wins
  over `decide` for both kinds.
- **Idempotent + self-correcting:** each reconcile matches desired vs actual, so a missed
  start/stop heals on the next tick (same discipline as the lease/heartbeat). A faceFollow task
  started under an open session that then closes simply dies and is restarted next tick ŌĆö that's
  how a session-scoped task becomes a session-INDEPENDENT conducted thing with no new machinery.
- **No always-on LLM** (vision ┬¦5 held). The conductor is rules + cheap reads.
- **Body arbitration is NOT the conductor's job** ŌĆö the lease already handles "two things
  want the body." The conductor decides *whether a thing runs at all*; the lease decides
  *who drives the body right now*. Clean separation.

## 7. Where it lives + the surface (as built)

- **Code:** `server/src/modules/conductor/` ŌĆö the pure core (`conducted.ts`: the `Conducted`
  descriptors + `decide` fns + `CONDUCTED` registry; `reconcile.ts`: the `reconcile` + `Effects`)
  and the I/O glue (`index.ts`: `conductorModule`, the ~1 Hz loop, `assembleWorld`, REST). It
  depends on the brain's `ConductorAccess` (convMode + task start/stop/list) + `WakeApi`
  (the wakeUp behaviour's hook), the bodylink lease holder, and config ŌĆö all existing.
- **REST:** `GET /api/conductor/:dock` ŌåÆ `{ dock, conducted: [{ name, kind, desired, running,
  tunings, override, instrumentedAt, taskName }] }` ┬Ę `POST /api/conductor/:dock/:name/(run|stop|auto)`
  (the manual override). Tunings are edited through the existing config (`POST /api/config`
  `{ conductor: {ŌĆ”} }`).
- **Console:** the **Conductor tab** ŌĆö a card per conducted thing with a clear **task / behaviour**
  badge, live state (off/arming/running), the **`instrumentedAt`** line for a behaviour (where
  it's woven in code) or the `taskName` for a task, the **Run now / Stop / Auto** buttons, the
  enabled toggle + the knobs. (Debuggability was the faceFollow lesson; make the decisions visible.)
- **Log:** `[cond] <dock> <name>: <from>ŌåÆ<to> (why)` on every transition ŌĆö the ground-truth
  trace for testing (like `[ff-event]`).

## 9. Open / deferred (kept OUT of v1 deliberately)

- **Learned write-back** (the reasoning task that tunes the variables) ŌĆö v2. v1 ships fixed
  defaults + console editing; the tunings are structured so v2 is a pure add.
- **`decide` as injected code-logic** ŌĆö the signature allows it; no behaviour needs it yet.
- **The "behaves" plug-in contract** (independently authored/loaded policy modules, vision
  ┬¦5a/┬¦10) ŌĆö v1 hardcodes the two behaviour descriptors in the registry. Pluggable loading is
  a later generalization once we have >2 and a reason.
- **Event-driven re-evaluation** (┬¦6) ŌĆö v1 is cadence-only (~1s reactive); event triggers
  (task start/stop, presence change, turn start/end ŌåÆ immediate `reconcile`) are a v2 add the
  loop is already shaped for.
- **ResourceMonitor** (station-level throttle) ŌĆö vision ┬¦6, last.
- **Trigger discipline escalation** ŌĆö v1 triggers are strictly cheap rules; no escalation path.
