## 1. The two memory tiers (the framing that settled it)

The dock has memory at **two tiers**, and naming them is what made the design click:

| Tier | Is | Backing | Lifetime |
|---|---|---|---|
| **Short-term memory** | the **perception stream** — snapshots (👁/🎙/👤/😮/🤖) | in-memory ring | seconds–minutes; drops the past |
| **Long-term memory** | the **durable belief store** | sqlite (`.data/orbit.db`) | persists; supersede-not-delete |

The **long-term memory curator** is the thing that *organizes and maintains long-term
memory from various sources.* It is **not** the perception processing — it's the layer
that decides what, from the short-term tier (and elsewhere), deserves to become or stay
a long-term belief, and keeps those beliefs honest.

This reframing matters: an earlier cut called it just "the curator" and scoped it to
*maintain-only* (revise/forget existing beliefs). Treating snapshots as a memory tier
(real-time memory), not as "raw input," makes **promotion from short→long-term** part
of the *same* job — tending the memory hierarchy — rather than a separate concern.

---

## 2. Two operations, one purpose

The curator tends long-term memory via **two operations** (one module, one console,
one "tend memory" purpose — but distinct contracts because their blast radius differs):

```
short-term (stream) ──consolidate──▶ NEW durable beliefs ──reconcile──▶ kept honest
   sources (§3)         (CREATE)         (lineage, derived)   (REVISE / FORGET)
                                                                  ▲
                              existing long-term beliefs ─────────┘
```

- **consolidate** — read a window of a chosen **source** (§3), let the LLM judge what's
  worth durable memory, write the salient few as **derived** beliefs with **lineage**
  back to what they came from. This is **create** — the dangerous side (it can fabricate
  from a noisy feed), so it carries uncertainty as **confidence** rather than pretending
  to be certain, and leans on reconcile + source-choice (§4) to stay sane.
- **reconcile** — review existing beliefs, **revise** drifted/contradictory ones
  (supersede, history kept) and **forget** the stale. **Bounded**: it can only touch ids
  it was shown — a hallucinated id never reaches the store (the id-guard, kept from the
  first cut + tested).

Why two operations, not one prompt: they read **different sources** (a stream vs the
belief store) and have **different risk** (unbounded create vs bounded edit). Keeping
them separate keeps each prompt + each safety contract clean. They compose in one pass:
consolidate first, reconcile the result.

## 3. Sources are a CHOICE, not a restriction

The curator does **not** own its inputs — it **selects** them. For a given signal it
reads the *best available* representation, and that set grows over time (a future
cleanup/event-extraction stream just becomes another selectable source).

Concretely today, a **speech snapshot already carries both** representations (the
background diarization patches the record in place — [stt-watch.ts]):

- `payload.text` → **diarized** (clean, speaker-labelled) — the default read.
- `payload.sttText` → **raw Parakeet** — available *for more context* when needed.
- `payload.bgModel` → whether it's been upgraded yet.

So "source selection" is mostly *which fields/kinds the curator reads*, not separate
pipelines. **Default: diarized speech. Allowed: corroborate with raw, pull other kinds.**
No restriction — it's a choice depending on what's needed, and it's expected to evolve.

**Vision will follow the same shape (planned).** Just as speech has a cheap live path
(Parakeet) + a refined background stream (diarized) that the curator prefers, **vision
will get a refined KEYFRAME stream** — salient frames emitted on change, not the raw
free-running 5s VLM windows. The curator will consolidate from *that* refined visual
source (cleaner than raw VLM, and the natural answer to "what was seen worth
remembering"). It also relieves the ring (see §8c): the live VLM drip is the main
non-activity-gated filler today; a change-gated keyframe stream doesn't pile up the same
way. Not built yet; noted so the source-seam already anticipates it.

## 4. Hallucination: minimize, don't pretend to eliminate

Stance (explicit): **hallucination is a fact of life.** We don't gate it perfectly up
front; we **minimize** it and let it wash out:

1. **Pick the best source** (diarized over raw) — the first and cheapest defense, and
   the reason source-choice is first-class (§3).
2. **Carry uncertainty as confidence**, don't assert. A derived belief lands at modest
   confidence; reconcile + future corroboration raise or prune it.
3. **Lineage on every derived belief** — `LineageEdge` back to the snapshot(s) it came
   from, so any belief is auditable ("why do I believe this").
4. **Reconcile is the cleanup pass** — contradictions/drift get fixed over time, so
   consolidate doesn't have to be perfect.

**Few levers, not dozens.** A deliberate constraint: a small set of sane config knobs
(source choice, an interval, a window, maybe one salience threshold) — NOT a per-signal
knob farm that's impossible to manage. Confidence + reconcile do the fine-tuning that
config otherwise would.

## 5. The hard part: event-time alignment

Consolidating from multiple sources is hard not because of *what* to keep but because
each source is on its **own timeline and latency**:

- live STT lands ~200 ms; the **diarized** version lands seconds–minutes later and
  *replaces* it; vision windows lag ~5 s; identity is ~2 s state; a future stream might
  emit long after the moment it describes.

So the curator must reason about **when something actually happened** (event-time), not
when the record *arrived*. A diarized utterance spoken at 10:02 but landing at 10:05 must
be consolidated against the identity/vision that were true **at 10:02** — else you
remember "Guru said X while no one was in frame."

The primitives already exist: snapshots carry `from`/`to` (IST event-time), and the
store has **`stateAt(kind, iso)`** = "the value of state stream `kind` as of time T"
([perception-pipeline.md] state-vs-event note). Consolidate reads the speech window and,
per utterance, uses `stateAt('identity', utterance.from)` to attach *who was present when
it was said*. **Alignment = query each source as-of the event-time, not the latest.**

---

## 6. Salience: let the LLM decide

Consolidate does **not** pre-categorize into facts/events/presence buckets. It hands the
LLM the (event-time-aligned) window and asks it to judge what is worth durable memory,
guided by the conservative stance (§4). Simpler prompt, fewer brittle rules, and it
generalizes as sources grow. The LLM returns candidate beliefs (claim + type + subject +
confidence); we attach lineage and write them `derivation:'derived'`.

## 6a. Cadence — the two ops run on different clocks

A blind wall-clock timer is wrong for **both** ops, for opposite reasons, so they get
different triggers matched to their nature:

Speech is **bursty** — occasional trickles AND sudden floods — and memory must keep up
**reliably + efficiently** at both ends. A blind timer fails both (floods → giant
redundant prompts + the ring drops un-consolidated speech; trickles → never hit the
threshold, stranded). So consolidate is **load-aware** around a per-dock **watermark**
(last-consolidated event-time); "pending" = speech after it.

| Op | Trigger (`cadence.ts`, pure) | Why |
|---|---|---|
| **consolidate** | **flood** (`pending ≥ K`) **OR age** (`oldest pending ≥ maxAge`) **OR quiet** (`no speech for quietMs` & pending>0) | flood batches the rush; **age** is the safety net (overrides the floor so the ring can't drop un-consolidated speech); **quiet** sweeps the end of an exchange so a lone sentence isn't stranded. |
| **reconcile** | **slow interval** (~30 min) | belief drift is slow; no need to chase the stream. |

**Bounded batches (flood-safe).** A pass takes only an **oldest, capped chunk** of
pending (`batchSize = min(pending, maxBatch)`) and advances the watermark over **exactly
what it sent** — so a 50-utterance flood drains over several small passes (not one giant
prompt), each utterance consolidated **exactly once** (no re-send, no duplicates), and an
LLM error **doesn't advance the watermark** (the span retries — no data loss). Tested:
the 50→3-batch drain asserts exactly-once.

**The watermark is RESTART-SAFE — derived from belief lineage, not a stored cursor.**
The in-memory watermark gives exactly-once *within a run*; across a restart it would
reset to `''` and re-consolidate the ring into **duplicate beliefs**. Fix: on first
sight of a dock, the watermark is **seeded** from `MemoryStore.latestConsolidatedSource`
= the newest source event-time among that dock's derived-belief lineage edges. **The
durable beliefs ARE the checkpoint** — "where I'm at" is derived from what's actually in
long-term memory, so it can't drift from a side-record. Tested: a fresh curator seeded
from lineage processes only the post-watermark utterance, never re-sends consolidated
ones. (Caveat: a span the LLM judged "not salient" leaves no lineage, so it's
re-examined after a restart — safe, just a little wasted; acceptable for v1.)

**Patient under contention** — consolidate is the lowest-priority LLM consumer (the
diarized re-transcription + live STT share the Metal GPU, §8 of perception-pipeline);
memory can wait, transcription can't. The floor + bounded batch keep it from thrashing.

All decisions are **pure functions** (`decideConsolidate`, `batchSize`, `shouldReconcile`)
— exhaustively unit-testable without Gemini/sqlite. **Reliable + testable now.**

Default knobs (few, per §4 — **first guesses, tuned-by-eye until a real flood is
observed**): `K=5`, `maxAge=5 min`, `quiet=45 s`, `maxBatch=20`, `floor=30 s`; reconcile
`30 min` / min `4` beliefs. All env-overridable; NOT a per-signal knob farm.

## 7. Structure (the rewrite)

Ground-up rewrite (the first `curator.ts` is reference for the proven idioms — the
loop, JSON tolerance, the id-guard, the console handle — not the thing extended). The
loop SHAPE is kept (it mirrors `auto-summarizer.ts` and is tested); the STRUCTURE is
rebuilt for two-ops + sources + alignment instead of maintain-only.

- the **loop + console handle** — runs enabled operations per dock on a cadence;
  enable/disable toggle; a recent-passes feed (per pass: op, reviewed, created/revised/
  forgot, skipped-reason, the changes) + run-now. (kept from the first cut.)
- **reconcile** — the first cut's logic, extracted: beliefs → verdict → validated
  (id-guarded) revise/forget plan → store.
- **consolidate** — new: source window → event-time-aligned context → LLM salience →
  derived beliefs (lineage + confidence) → store.
- **sources** — the selectable-source seam (diarized speech now; extensible).

Pure decisions/parses/plans stay unit-testable without Gemini/sqlite (injected effects),
as before. In-process (reaches the `MemoryStore` + the `SnapshotStore` ring directly).

**Config scope = ALL DOCKS (global).** The curator is a single station-wide loop; its
knobs (`CuratorConfig`) gate every dock's consolidate/reconcile identically — there is
**no per-dock override**. This matches every other perception-console knob today (gate,
diarization, vision instruction, sidecars are all global). The console shows an **"all
docks"** scope badge so this is explicit; the seam leaves room for a future per-dock
override (none needed yet). The knobs are **live-tunable** — held in a mutable
`CuratorConfig` the loop reads each pass (not load-time consts), so a console edit applies
on the next tick **without a restart**; env values seed the defaults. `CONFIG_META`
carries each knob's bounds + a note (drives validation AND the UI field hints).

## 8a. v1 vs. what-it-needs-to-be (the honest delta)

Built **reliable + testable now**, shaped toward the target, with the fragile parts
flagged for iterative build-out rather than faked-solid:

| Aspect | v1 (built now — reliable + testable) | What it needs to be (the delta — iterate) |
|---|---|---|
| **Cadence** | **load-aware** (§6a): watermark + flood/age/quiet triggers + bounded batches + exactly-once. Solid + tested (incl. a 50-utterance flood drain). | knob VALUES (K/maxAge/quiet/maxBatch) are first guesses — tune against a real flood |
| **Source** | **diarized speech window** (`payload.text`), default | corroborate with raw `sttText`; pull vision/identity text as context — evolving (§3) |
| **Event-time alignment** | **simple**: per utterance, attach `stateAt('identity', from)` — who was present when said | **FRAGILE → iterate.** Full multi-source alignment across lagging streams (vision ~5 s, future late-emitting sources), reconciling arrival-vs-event time robustly (§5). v1 does the one cheap, correct attachment; the general aligner is later. |
| **Salience** | LLM judges the window (§6), conservative bar; confidence **clamped** so a one-window distill can't over-assert | richer corroboration / multi-pass salience — later |
| **Subject (who)** | used WHEN present-at is available, **not forced** (identity unstable → forcing = wrong names) | anchor every belief to a person/speaker once identity is stable (§8b — the biggest recall-quality gap) |
| **De-dup vs existing** | "already known" handed in + prompt asks to merge-one-topic; still leaks | smarter semantic de-dup; reconcile prunes the leakage meanwhile (§9) |

**Why this split:** the cadence + the id-guarded plan + the single `stateAt` attachment
are *deterministic and unit-testable*. Multi-source timeline alignment is *reasoning
about latency* — genuinely fragile, easy to get subtly wrong, and best grown against
real captured data. So v1 ships the correct-small version and the delta is explicit, not
hidden.

### 8b. First live run — what it confirmed, what it exposed

First end-to-end run against **real room conversation** (STT sidecar up, ~37 utterances,
a genuine flood):

**Confirmed (the mechanism — works):** speech → diarized → consolidate read the pending
span → LLM judged salience → created durable `derived` beliefs; a second pass picked up
the *next* span (different content), watermark advanced, **no re-consolidation**. The
flood/exactly-once behaviour holds live, not just in the unit test. Captured genuinely
useful beliefs ("intends to make creations open and free", "wants to open a shop to sell
a 3D model", "goal of creating a robot pet").

**Exposed (quality — tuning, now partly addressed):**
- **No subject on most beliefs** ("is willing to live in a one-bedroom house" — *who?*).
  **Decision: do NOT force subject yet** — vision/identity isn't stable, so demanding
  "who" would attach *wrong* names. Prompt now uses present-at WHEN available, omits
  otherwise. Revisit when identity is stable. *(This is the biggest known limitation for
  recall quality, deliberately deferred to the identity work.)*
- **Over-confident** — the model returned 0.8 on thin fragments. **Fixed:** confidence is
  now **clamped** (`CONSOLIDATE_CONFIDENCE_MAX`, default 0.6) — a one-window distill can't
  assert near-certainty; corroboration via reconcile can raise it over time.
- **Fragmenty / near-duplicate beliefs** — several tiny overlapping beliefs from one
  topic. **Improved (prompt):** require complete standalone sentences + merge-one-topic;
  **still leaks** — full semantic de-dup/merge is reconcile's job and isn't strong yet.
- **Occasional misread** ("believes human clones are being made" — over-read of a
  fragment). The grounding-guard proves *traceability*, not *meaning*; accepted per §4,
  mitigated by the conservative prompt + clamp + reconcile. Inherent to a noisy feed.

### 8c. Ring residency = the durability boundary (and which stream fills it)

The source the curator consolidates from is the **snapshot ring** — a bounded (~1000),
in-memory, **capacity-evicted** FIFO. Implications worth stating plainly:

- **The watermark gives exactly-once, NOT at-least-once.** It guarantees nothing is
  *re-consolidated*, but if a speech record is **evicted from the ring before being
  consolidated, it is permanently lost** (the ring has no disk backing).
- **Catch-up is gated on ring-residency, not curator uptime.** If the curator is down/idle
  a long time, on restart it consolidates **everything still in the ring** after its
  watermark (age-trigger fires, drains in bounded batches, oldest-first). Anything the
  ring already evicted is gone. **So a bigger ring CAP directly widens the catch-up
  window** — purely a ring config change; the curator needs no change to benefit.
- **What fills the ring is NOT uniform with "activity":** identity/emotion (2 s timer but
  **change-gated** → ~nothing in a static scene), speech (**VAD-gated** → only utterances),
  bodymotion (**command-gated**) all stay quiet when the room is. The **exception is
  vision** — a free-running ~5 s loop that commits a window **whenever the stream is
  published, regardless of activity.** So:
  - **quiet room, camera on** → ring fills at ~vision's rate (~12/min) → ~80 min of buffer
    → speech is safe for a long time. **Low activity is the SAFE case.**
  - **busy room** → identity+emotion+speech pile on top → ring churns in minutes → the
    very speech you most want can be evicted before consolidation. **High activity is the
    eviction-RISK case** (counter-intuitive but true).
- **The vision fix is the planned keyframe stream (§3), not the curator.** A change-gated
  refined keyframe source replaces the free-running VLM drip as both the curator's visual
  input *and* the main ring filler — relieving pressure the same way diarization did for
  speech. (Also a perception efficiency win: vision today re-describes an empty room on a
  clock.) Until then, the cheap lever is ring CAP + the age/floor cadence.

## 9. Open points

- **Corroboration with raw** (§3) — the *mechanism* for "also check `sttText` when the
  diarized read is thin" is a choice the consolidate prompt/code can make; the exact
  trigger is left to evolve (not a config lever yet).
- **De-dup against existing beliefs** — consolidate should avoid re-creating a belief the
  store already holds (recall the subject first, hand the LLM "what you already know" so
  it only adds the genuinely new). First cut may over-create; reconcile prunes dups.
- **Cross-process write visibility** — a task writing via `this.memory` (separate sqlite
  connection) isn't seen live by the in-process curator until its next read (memory.md).
  Fine today; an event seam if it ever matters.
- **Subject attribution** (the big one, deferred) — beliefs mostly lack a "who" because
  identity/vision isn't stable. Once it is, anchor each belief to the present person /
  speaker so recall ("what does Guru believe about robots?") works. Tracked in §8b.
- **NEXT STEP — surfacing this to the user-facing agent.** We now have a real corpus of
  derived beliefs to play with. The open question is *how the dock brain consumes them*:
  it already has `recall_memory` (pull on demand), but the richer question is whether
  consolidated beliefs should also **proactively shape grounding** (the per-turn context),
  and how recall ranks derived (lower-confidence, possibly noisy) beliefs vs. explicitly
  recorded ones. This is the next design thread, not part of the curator itself.
