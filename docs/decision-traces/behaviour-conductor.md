## 1. The problem that started it

"Have the dock **find faces in the room and follow them if they move.**" Then,
immediately: *don't code for one use case* — model the general thing, because there
will be many (daydream over memories, helpfully barge into a conversation, make and
share artifacts, monitor resources, …). The question became: **what is the general
construct for "the dock continuously watches what's happening and decides to do
something about it" — and how does it relate to what already exists?**

This doc is the answer we converged on, and — just as important — **why the obvious
answers were wrong.**

---

## 2. What already existed (and why it wasn't enough)

Two pieces of the codebase already do a version of "watch → decide → act":

- **The attention gate** (`modules/perception/attention/`, built, **disabled by
  default**) — a debounced watcher reads the snapshot ring, a **pure** judge
  (`evaluateGate`) decides RAISE/QUIET, and on RAISE it calls
  `enqueueAutonomousTurn` (the brain). It is *one* "watch→decide→act" loop hardwired
  to *one* action (raise a self-thought). See [perception-to-brain.md](../perception-to-brain.md).
- **Tasks** (`modules/brain/tasks/` + `tasks/`, built) — long-running background
  jobs, each its **own OS process**, LLM-authored (`write_task`), supervised
  (lifecycle, checkpoints, crash isolation). A task **already has its own LLM**:
  `this.agent` is a ready-to-use pi Agent (the dock's model + key, lazy-loaded), and
  `this.ask()` is the one-shot. (tasks.md was corrected to document this + the
  capability layer. NOTE: memory is NOT a capability — it's direct `this.memory`, §4a.)

So "watch→decide→act" and "a capable, reasoning, supervised doer" both exist. The
gate is the *pattern*; the task is the *substrate*. Neither, alone, is the general
construct — but together they nearly are.

## 3. The dead ends (recorded so we don't relitigate them)

We tried, and rejected, several framings. Each was wrong for a specific reason worth
keeping.

| Framing tried | Why it was rejected |
|---|---|
| **"Build a face-follow feature."** | The user explicitly wanted the general construct, not the one case. faceFollow is an *example*, not the design. |
| **Generalize the gate into a "reactor registry" of N gates.** | Right instinct (the gate *is* one instance of the pattern), but it stops at "escalate to the brain." It has no model for **acting on the body**, no **lifetime** of its own, and no **governance**. Too thin. |
| **"Behaviours are in-process objects" (vs. tasks being processes).** | The claimed distinction — "behaviours need in-process access, tasks don't" — **collapsed under scrutiny**. Tasks will need the perception stream too (tasks.md §9 says so). And a few-Hz control loop survives a sub-ms localhost WS round-trip fine. *Neither* latency nor capability justified in-process. The real differences were lifetime and triggering, not the substrate. |
| **"A behaviour is just a long-running task."** | A task is **session-scoped** (ends when the conversational session ends — tasks.md §7) and **imperatively launched** (`run_task`). The things we want — faceFollow, daydream, resource-monitor — must run **when no one is talking**, must be **armed-but-not-running** until the world warrants, and must be **governable as a named set**. A task can't own "when do I run" (that's the thing being spawned) and can't outlive a session. So a behaviour is **not** just a label for a long task. |
| **"Behaviour = a separate concept with its own supervisor/authoring."** | Duplicates ~800 lines of tested task machinery (supervisor, lifecycle, `write_task`, scaffold, typecheck) to express a difference that is really "started by a trigger, not by `run_task`." Wasteful. |
| **One always-on reasoning agent per dock.** | "The conductor reasons every cycle over memory+stream" = a **second brain per dock**, running continuously, costing tokens continuously. Defeats the whole "cheap things run often, expensive things run rarely" economy. Rejected in favour of the cheap conductor (§5). |

The path through the dead ends produced the model in §4.

## 4. The model we landed on: a per-dock conductor that conducts tasks

**One conductor instance per dock.** It is **not** capable itself — it is a thin,
session-independent **conductor**. All capability lives in the **tasks** it runs.

```
 ResourceMonitor            station-level — watches the HOST (CPU/GPU/RAM/sidecars),
        │  (may throttle)    can tell conductors to back off (see §6). NOT per-dock.
        ▼
 per-dock CONDUCTOR         cheap governor — rules + config + pluggable policies +
        │                    cheap reads of memory & the perception stream.
        │  (authors/spawns/   Decides WHICH tasks to run. Reasons only by SPAWNING a
        │   stops/configures) reasoning task — never an always-on LLM of its own (§5).
        ▼
 TASKS                      the universal capable doers — process + this.agent +
                            wire capabilities (perception, ACTUATOR-LEASE) + DIRECT
                            access (this.agent, this.memory — shared code/.env).
                            faceFollow loop, barge-in, make-and-share, …
                            (memory CURATION is NOT here — it's pipeline work; §4a)
```

**Behaviours wrap tasks.** A "behaviour" (faceFollow, daydream, …) is **what the
conductor chooses to do** by authoring/triggering/managing one-or-more tasks —
not a separate construct with its own runtime. `write_task`/`run_task`/`stop_task`
(the brain's existing task tools) are the conductor's toolkit. A single behaviour
can manage **multiple** tasks (faceFollow → one task per tracked person;
create-artifacts → a research task + a video task).

### What the conductor construct adds (its entire value — hold it to this)

We interrogated "what is the value-add at all?" hard. A task alone *almost* suffices.
The conductor earns its existence by adding **exactly four things tasks
structurally lack** — and nothing else:

1. **Session-independent lifetime** — tasks die with the conversational session; the
   conductor is **dock-scoped** and runs when no one is talking.
2. **Declarative arming** — *armed ≠ running.* A cheap trigger spawns the actual
   work only when the world warrants, and tears it down after. "When do I run" can't
   live inside the thing being spawned.
3. **A stable named handle for an intent** — the intent ("follow faces") may be 0..N
   churning tasks; you toggle the **intent**, not the anonymous tasks.
4. **A governable surface** — enable/disable/reconfigure standing intents (this is
   what the former "BehaviourAdjuster" idea becomes — the conductor managing
   itself; §6).

**The discipline:** the moment an conductor does *capability work* instead of just
*arming / triggering / governing tasks*, it has overstepped — that work belongs in a
task. Keep it thin and it earns its place; let it fatten and it is a redundant second
task system.

### 4a. The purpose-rule — what is a behaviour vs. what is the pipeline

The sharpest boundary, and the one that decides where a piece of work lives:

> **Producing / maintaining memory + perception ARTIFACTS is the perception
> PIPELINE's job. A behaviour's / task's main goal is to ACT** — it *operates on* the
> artifacts the pipeline produced, and may touch memory **as a side effect**, but
> creating/maintaining an artifact is never *why it exists*.

The test: **is creating a memory/perception artifact the MAIN GOAL, or a side effect?**
Main goal → pipeline. Side effect of an action → behaviour/task.

- The pipeline produces a growing surface of **perception artifacts** (snapshots,
  summaries, layered memories). **memory** is the durable, subject-anchored,
  lineage-tracked subset. New use case → maybe new pipeline processing. Producing
  artifacts is the point.
- A behaviour/task's goal is an **action**: move (faceFollow), speak (barge-in), share
  (post to Slack), alert/adjust (resource monitor). It *reads* the artifact surface; a
  write-back (e.g. "I recorded that I interjected") is incidental.

**This rule re-homed a thing we'd built wrong.** We first built `daydream` as a *task*
that curates memory (revise stale/contradictory beliefs). But its main goal IS
maintaining memory artifacts — no action in the world — so by the rule it is **pipeline
work, not a behaviour.** It became the **long-term memory curator**
(`modules/perception/memory/longterm/`, an in-process pipeline loop — now with both a
*consolidate* (create) and *reconcile* (maintain) op; see
[long-term-memory-curator.md](long-term-memory-curator.md)). The deletion is the rule
doing its job: catching a "behaviour" whose real goal was artifact upkeep. (We also
dropped `create-artifact`-as-summary; "make-and-**share**" is a real action task, but the
memory-summary half was pipeline work wearing a task costume.)

Memory access from a task/behaviour survives — as the **direct** `this.memory`
affordance (NOT a wire capability; memory is reconstructible from shared code + `.env`,
see [tasks.md](../tasks.md) "Direct vs. the wire"). It is exactly the "touch memory as a
side effect" the rule permits a
behaviour/task to use.

## 5. Cheap conductor, not a second brain (the cost decision)

Decision (deliberate, cost-driven): **the conductor's own loop never runs an
always-on LLM.** It is rules + config + pluggable policy modules (§5a) + cheap reads
of memory and the snapshot stream. It decides *which tasks to run* by policy.

When **judgment** is genuinely needed — "is this conversation worth interjecting?",
"how should these memories be revised?" — the conductor **spawns a reasoning
task** whose `this.agent` does the thinking and can write back configs/memory. So
reasoning is **episodic, visible, lifecycle-managed, and cost-attributed** (it shows
up in the Cost tab as a task), never a hidden continuous drain.

"Changes its behaviour over time" therefore means: a reasoning task (a daydream-style
job) periodically reflects and **rewrites the conductor's configs** — adaptation
is a discrete, auditable act, not free-running thought.

> This *is* the old "pyramid" (cheap tier gates the expensive tier), restated without
> the jargon: **the conductor is the cheap governor; the expensive thinking is
> tasks it conducts.** See §7 for the pyramid's retirement.

### 5a. Configs are not enough — pluggable policies

Some conductor logic is too complicated for configuration knobs alone (faceFollow's
search/track/re-acquire state machine; barge-in's relevance heuristics). So the
conductor's decision layer is **config + pluggable policy modules** — a "behaves"
can be **written independently and plugged in/out**, not only tuned by config. Two
tiers of authoring, mirroring tasks' `packaged/` vs `generated/`:

- **Config-tunable behaviours** — thresholds, on/off, cadences. Changed live (like the
  gate's `enabled` toggle, like `brainTask*` config keys).
- **Pluggable policy modules** ("behaves") — committed or LLM-authored code units the
  conductor loads to make a non-trivial decision. Same arm/fire/govern contract,
  but the *decide* step is real code, swappable without touching the conductor
  core. (Open design point — §9.)

Keep policy modules **cheap by contract** (arithmetic / state machines / rules). A
policy that wants to *reason* doesn't reason inline — it tells the conductor to
**spawn a reasoning task** (§5). This is the line that keeps the conductor cheap.

## 6. ResourceMonitor is station-level, above the conductors

Decision: **ResourceMonitor is NOT a per-dock behaviour.** Host resources (CPU, RAM,
the shared Metal GPU — the [§8 "don't stack models"](../perception-pipeline.md)
contention, swap, sidecar health) are **about the whole station/host**, not about
what's in front of one dock's camera. Folding it into the per-dock conductor would
run it N times monitoring the same shared host and wrongly couple host-stability to
perception.

So ResourceMonitor sits **above** the conductors: it watches the host and can
**throttle** conductors (e.g. "back off spawning tasks — the GPU is saturated").
It is a layer, not a peer.

## 7. The pyramid is retired as a named pillar

[perception-pipeline.md §9](../perception-pipeline.md) framed a "tiered escalation
pyramid" (T1 streams → T2 fuse → T3 brain) as a forward plan. Its tier-2/3 escalation
framework was **never built** (only T1 + the cheap-rules gate exist). We are
**deleting the pyramid as a top-level abstraction** and keeping only its one true
idea, now expressed concretely:

> **A cheap trigger wakes an expensive task.** The conductor (cheap) decides;
> tasks (expensive, some of which reason) do.

## 8. faceFollow — the first build, chosen to ground the keystone

faceFollow needs **no intelligence** (it's pure sense→arithmetic→act), so it cleanly
isolates the one genuinely hard, undesigned, physically-risky piece: the **actuator
lease**. Build it first to prove the keystone before any framework rests on it.

- **Sense** — subscribe to identity snapshots (`payload.faces[].box`, normalized
  coords). The 2 s identity cadence is too slow for a follow loop, so faceFollow needs
  a faster box source (a higher-rate detect, boxes only) — one of the substrate
  additions (§9).
- **Decide** — box-center error → `foot` (pan, ±90°) + `neck` (tilt, −60…+35°) deltas,
  with deadband + slew limit. A **search/track/re-acquire state machine**: visible+
  centered → idle; visible+off-center → align; not visible → sweep **from last-known
  position outward** (search meaningfully, not blind). This is the first "behaves"
  policy module (§5a).
- **Act** — through the **leased** actuator facade, NOT raw `MotionExecutor`.
- **Loop closure** — every move also feeds `bodymotion.pushCommand` so identity's
  ego-motion guard ignores the self-induced view change (mechanism already built).

The body has **2 DOF (pan+tilt), no locomotion** — "follow" = head-tracking, not
pursuit. A person who leaves pan range can only be searched-for, not chased.
(Locomotion is node-rover, a different device.)

The faceFollow prototype's real job is to exercise the **lease lifecycle**, not the
control math: *acquire → stream corrections → crash the task → confirm the body
releases and the brain regains control.* If that is clean, the substrate is sound.

## 9. What's genuinely new engineering (everything else is reuse)

The model keeps the new work small and concentrated:

1. **The actuator-lease facade** — THE KEYSTONE. `MotionExecutor` is today the body's
   single master, last-write-wins. Letting tasks drive it reintroduces the
   multi-master problem the brain doc killed *by construction*. The facade must be a
   **leased, preemptible, liveness-checked arbiter**: ownership with a short TTL
   (a dead task releases the body in ~one leasetick, not the 10 s WS-watchdog window);
   **the brain always preempts** (a user turn yanks ownership back); a rate clamp.
   This is a **physical-safety boundary**, not just a correctness one — sharpened by
   the decision that tasks get a full agent loop (an autonomous agent with a physical
   actuator). Solve it **once at the executor**, for every caller (brain, tasks).
2. **Event-driven task entry + perception-subscribe facade** — so a control-loop task
   (faceFollow) is `onSnapshot(...)`, not a non-interruptible `while(true){…sleep}`
   loop. The task harness's `sleep` is time-driven; reflexes are event-driven.
3. **The conductor** — per-dock, session-independent, cheap rules+config loop with
   the task toolkit and read access to memory + stream. Thin by construction (§4).
4. **The "behaves" plug-in mechanism** (§5a) — how policy modules are authored,
   loaded, swapped. **Open design point.**
5. **ResourceMonitor** (§6) — station-level host watcher that can throttle
   conductors. **Open design point.**

Reused as-is: the task process substrate, `this.agent`, `write_task`/scaffold/
typecheck, the supervisor lifecycle, config-registry + REST + console toggles, the
memory store (`recall/remember/revise/forget`), the snapshot ring + `subscribe`.

## 10. Open design points (not yet decided)

- **The "behaves" plug-in contract** (§5a) — module shape, how the conductor loads/
  unloads them, LLM-authoring guardrails (a body-touching policy needs a higher trust
  bar than a reminder; the lease TTL + rate clamp are the safety net).
- **Lease arbitration policy details** (§9.1) — TTL value, preemption rules, what
  happens to a half-finished follow when the brain preempts.
- **Trigger discipline** — are conductor triggers strictly cheap-rules (and the
  fired task may decide "do nothing", e.g. barge-in spawns a judge-task that usually
  stays quiet), or may a trigger escalate? Leaning strict (cheap trigger, task
  decides) to keep the conductor thin.
- **Faster face-box source** for faceFollow (§8) — drop identity period for tracking,
  vs. a separate box-only detector. Decide after measuring jitter on the real dock.
- **Conductor authoring** — is the conductor itself a fixed station component,
  or partly LLM-authored? (It's thin enough to be fixed; the *behaves* it loads are
  the authored part.)

## 11. Build order

1. **faceFollow** as a task (event-entry + lease) — grounds the keystone (§8).
2. Extract the **lease facade** cleanly once faceFollow's crash/preempt loop is clean.
3. The **conductor** shell (per-dock, armed, governable) — retrofit the attention
   gate as its first cheap trigger.
4. A **reasoning behaviour** (daydream: memory revise + agent) — grounds the spawn-a-
   reasoning-task path (§5) and the memory-mutate facade.
5. **HelpfullyBargeIn / CreateArtifacts** — grounds the trigger-discipline (§10) and
   external tools (Slack).
6. **ResourceMonitor** (§6) — station-level, last.
