## 1. The problem that started it

The dock already surfaces things the brain decides to *say*. But a growing class of
events isn't a thing the brain says — it's a thing the **user** needs to see and act
on, that arrives while the dock is busy or the user is away:

- a background **task finishes** while the agent is mid-conversation;
- a **Slack message** arrives that should be shown (not just auto-answered);
- future: a reminder fires, an artifact is ready, someone at the door.

These share a shape the current system has no home for: **they persist until the user
deals with them.** They survive a restart. They may carry rich media (the Slack
message's text, an image, a voice clip). The user should be able to *click* one. And
the dock should be *able* to announce it — but announcing is optional, not the point.

The user's own framing, which we adopt: this is **two different things**, and the
mistake would be to build one.

## 2. The decision that frames everything: two subsystems, not one

There are two kinds of "deferred message," and they want opposite lifecycles:

| | **Brain message** (auto-consumed) | **Notification** (user-facing) |
|---|---|---|
| Who reads it | the **system** (the brain drains it) | the **user** (clicks/dismisses) |
| Lifecycle | ephemeral — speak it, gone | **durable** — persists until acknowledged |
| Survives restart? | no (coalesces/expires) | **yes** |
| Examples | task-signal → autonomous turn, perception nudge | Slack message, "your task finished — here's the result," reminder fired |
| **Already built?** | **yes** — the `#autoQueue` autonomous-turn queue | **no — this doc** |

The brain-message path is the **autonomous-turn queue**
(`session.ts` `#autoQueue`, drained when idle, coalesces within 60 ms, expires in
≤120 s — see [perception-to-brain.md](perception-to-brain.md)). It is exactly right
for what it does and we leave it **untouched**.

> **DEAD END (recorded so we don't relitigate it): "one store with a type flag."**
> Tempting — one table, a `mode: announce | inbox` column. Rejected because the two
> lifecycles share almost nothing: the autonomous queue is in-memory, coalescing,
> expiring, and already wired to the brain's idle drain; the inbox is persistent,
> per-item-acknowledged, and wired to the dock UI. Folding them gives a schema that's
> half-ignored on every row and a module that owns two unrelated state machines. The
> user called this directly: *"separate out the two things."* We did.

The two subsystems **meet at exactly one seam**: a notification may, optionally, ask
the brain to announce it — which it does by **enqueuing one autonomous turn** (the
existing path). The inbox produces; the autonomous queue consumes; neither owns the
other.

## 3. Scope honesty: what "rich media" can actually mean today

The user wants notifications to carry **text, image, audio/voice, video** and an
**option to speak it out.** The dock's current rendering surface (verified against
`node-dock/app`) constrains this hard, and the plan must be honest about it:

| Media | Can the dock render it now? | Delivery |
|---|---|---|
| **text** | ✅ yes — subtitle band, event log, badges | frame payload (native) |
| **image** | ✅ yes — already accepts base64 (`imageBase64` in turn-requests) | base64 in frame, or asset URL (§6) |
| **speak it out** | ✅ yes — **but only via one autonomous turn**, *not* a raw `speak` frame (see correction below) | enqueue one autonomous turn (§4b) |
| **audio clip** (pre-recorded) | ❌ **not wired** — phone only plays synthesized TTS | needs a new playback path |
| **video** | ❌ **not wired** — only the live WebRTC SFU stream exists | needs a new playback path |

> **CORRECTION (from §10 pressure test).** An earlier draft of this table claimed a
> notification could speak by sending a standalone `speak` frame with a synthetic
> `turnId`, because `DockTts` holds no turn state. **That is wrong.** `onSpeak`
> ([RemoteBrain.kt:521](../../node-dock/app/app/src/main/kotlin/dev/orbit/dock/agent/RemoteBrain.kt#L521))
> **drops any `speak` frame whose `turnId` ≠ `currentTurnId`**, and `currentTurnId`
> is only set when a real turn starts (user turn, or an *adopted* autonomous turn). A
> raw notification speak frame would be silently dropped. The **only** way to make the
> dock speak is to enqueue an autonomous turn (§4b) — which is what the design already
> does. The shortcut never existed; the design is unaffected.

So the build splits cleanly: **text + image are frame-native and cheap**; **speak is
free but routes through one autonomous turn** (not a raw frame); **audio/video are a
separate, later capability** that needs a real asset-delivery and playback path on the
phone. We design the schema to *carry* an attachment list so audio/video slot in later
without a migration, but Stage 1–2 ship only what the dock can already show. We do
**not** pretend the dock plays audio today.

## 4. The model: a per-dock inbox the brain can write, the dock renders, the user clears

```
producers ─────────────▶  NotificationStore (per dock, durable)  ──┐
  • task finish/notify        add() · list() · ack() · dismiss()    │
  • slack inbound                                                   │ directed frame
  • brain (a tool)                                                  │ (mirror task-digest)
  • future: reminders                                               ▼
                                                          dock phone: badge → card → tap
                                                                   │           │
                                                          (optional speak)  ack/dismiss
                                                                   │           │
                                                          one autonomous   POST back
                                                             turn (§2 seam)  → ack()
```

**Per-dock, like everything else** (tasks, memory, sessions are all dock-scoped).
A notification belongs to a dock; cross-dock fan-out is explicitly out of scope.

**The store is the subsystem.** A small module (`modules/notifications/`) owning:
- a **per-dock durable list** (persistence: §5),
- a tiny **API** — `add`, `list`, `ack`, `dismiss`, `get`,
- a **directed push to the dock** that mirrors `task-digest` exactly (§4a),
- a **REST mount** `/api/notifications/:dock` for the read-only console view,
- one **bus topic** `notifications`.

### 4a. Delivery to the dock copies `task-digest` verbatim

`task-digest` is the proven analog: a directed `agent`-topic frame to
`(dock, component:"phone")`, re-pushed every 3 s as a failproof, rendered into a HUD
StateFlow on the phone. We copy its shape:

```jsonc
// station → phone, directed
{ "topic": "agent", "kind": "notif-digest",
  "toAddr": { "dock": "anne-bot", "component": "phone" },
  "payload": {
    "unread": 2,
    "items": [ {
      "id": "n-ab12", "kind": "slack" | "task" | "reminder" | "info",
      "title": "Slack from Guru", "body": "ping when you're free",
      "image": null,            // base64 or asset URL, optional
      "createdAt": 1750000000000,
      "state": "unread" | "seen" | "acked",
      "speak": false,           // did/should the dock announce it
      "actions": [ { "id": "open", "label": "Open" } ]  // optional click targets
    } ]
  } }
```

Re-push: same `setInterval(..., DIGEST_SWEEP_MS)` sweep that already pushes
`task-digest`, and on every `add`/`ack`/`dismiss` and on `hello`. This gives
free crash-recovery: a reconnecting dock gets the current inbox without any replay
protocol.

**Phone side** mirrors `onTaskDigest`: a new `onNotifDigest` updates a
`_notifications` StateFlow; a new Compose affordance renders it. The user's "bell with
a badge" lives here — a small badge on the dock face showing `unread`, tappable into a
card stack. Tapping a card → the phone publishes an `agent`-topic frame up (`notif-ack`
/ `notif-action`) → the store marks it `acked`/`dismissed` → next digest reflects it.

> **Why a digest and not per-item push?** Same reason task-digest is a digest:
> idempotent, self-healing, no per-message ACK protocol, survives reconnects for free.
> The list *is* the state; we push the whole (bounded) list.

### 4b. The optional "speak it out" is the §2 seam, nothing more

A notification with `speak: true` doesn't get a special audio path. The store enqueues
**one autonomous turn** ("you have a Slack from Guru: …") via the existing
`enqueueAutonomousTurn`. That turn's `speak` frames go through the normal TTS path. So
"announce this notification" reuses the brain-message subsystem rather than
duplicating it — the inbox *produces into* the autonomous queue. This is the only
coupling between the two subsystems, and it's one-directional.

## 5. Persistence: JSON-per-dock, not SQLite

Two conventions exist: tasks write `checkpoint.json` per instance; perception memory
uses SQLite. The inbox is a **short, bounded, append-and-mutate list per dock** —
closer to a task checkpoint than to a queryable belief store. We use a single
`notifications.json` per dock under the brain data root:

```
.data/notifications/<dock>/inbox.json   // [{id, kind, title, body, image, actions,
                                         //   createdAt, state, speak}], newest first,
                                         //   capped (e.g. 200), expired/dismissed pruned
```

Rationale: no query patterns justify SQLite (we always load the whole small list to
build a digest anyway); JSON read/write-sync matches the tasks supervisor's
best-effort persistence and keeps the module dependency-free. If the inbox ever grows
query needs (search, cross-dock, retention analytics) it can graduate to the perception
SQLite pattern behind the same API — the `NotificationStore` interface is the seam.

> **Concurrency (must-fix, from §10).** Two producers can call `add()` within the same
> tick (a task finishing while a Slack message arrives). The proven tmp+rename write is
> crash-safe but **not** concurrency-safe — read-modify-write from two callers loses one
> `add()` (last-write-wins). The fix is structural, not a lock: the `NotificationStore`
> is a **singleton that owns the in-memory list**; every `add/ack/dismiss` mutates that
> in-memory array and then persists serially. No two file writes race because there's
> one owner. This is how it should be built regardless; recorded here so it isn't missed.

## 6. Open design points (decide before the stage they block)

1. **Image delivery: base64 vs. asset URL.** Base64-in-frame is zero-infra and fine
   for small images; large/animated media bloats every 3 s digest. Decision deferred
   to Stage 2 (first image producer). Likely: small thumbnails base64 inline; anything
   bigger gets a station static-asset endpoint (`/assets/notif/<id>`) and the frame
   carries the URL. **Blocks: image notifications.**
2. **Audio/video playback on the dock.** No path exists (§3). This is a genuine new
   capability — phone-side player + an asset endpoint. **Out of scope for Stage 1–3;**
   the schema's `attachments` field reserves room. **Blocks: voice/video notifications.**
3. **Acknowledgement semantics.** Three states (`unread → seen → acked`) or two
   (`unread → done`)? Does "seen" mean rendered, or scrolled-to? Start with two
   (`unread`/`acked`); add `seen` only if the UI needs a read-receipt. **Blocks: phone
   ack frame.**
4. **Retention/expiry.** Per-kind TTL? Hard cap + LRU prune? Start: cap 200, drop
   `acked` older than N days, never auto-drop `unread`. **Blocks: store prune logic.**
5. **De-dup / coalesce.** Should five Slack messages from one person collapse to one
   notification? Probably yes (a `coalesceKey` like the autonomous queue's), but start
   without it and add if it's noisy. **Blocks: nothing — additive.**

## 7. Web console: read-only debug view, not a feature

Per the scope decision, the console gets a **read-only** view only — info/debug, not a
user surface. It copies `Tasks.tsx` verbatim: poll `GET /api/notifications/:dock`
every 3 s, list items with kind/title/state/age, no actions beyond maybe a manual
"clear" for debugging. No bell, no toast, no notification center in the browser. The
dock is the product surface; the console just lets us see what the store holds.

## 8. Build order

Each stage is independently shippable and ends at something visible.

- **Stage 1 — store + dock badge + dismiss (the vertical slice).**
  `NotificationStore` (JSON persistence, `add/list/ack/dismiss`), the `notif-digest`
  push mirroring `task-digest` (incl. 3 s sweep + hello), phone `onNotifDigest` +
  a minimal badge-and-card affordance, phone `notif-ack` up → `dismiss`. **One
  producer to prove it:** task-finish writes an `info` notification. Text only.
  *Done when:* finishing a task leaves a tappable badge on the dock that clears on tap
  and survives a station restart.

- **Stage 2 — Slack as a real producer + optional speak.**
  Route inbound Slack mentions/DMs (today: canned "coming soon") into `add()` instead.
  Wire `speak: true` → one autonomous turn (§4b). Add the read-only console view (§7).
  Settle open point #1 (image delivery) if Slack carries an image.
  *Done when:* a Slack DM appears as a dock notification, optionally announced, and
  shows in the console.

- **Stage 3 — richer kinds + actions.**
  `actions` click targets (e.g. "Open" → opens a brain turn with context), reminder
  producer, coalesce (#5) if noisy, retention tuning (#4).

- **Later (own capability) — audio/video.**
  The phone-side player + asset endpoint (#2). Separate effort; schema already
  reserves `attachments`.

## 9. What's genuinely new vs. reuse

**Reuse (most of it):** the `agent`-topic directed-frame contract, the `task-digest`
push/sweep/hello pattern, the phone's StateFlow-into-HUD render pattern, the
`enqueueAutonomousTurn` seam for speaking, the `Tasks.tsx` console poll pattern, the
JSON-per-dock persistence convention.

**New (the actual work):** the `NotificationStore` + its module/REST mount/bus topic;
the `notif-digest` / `notif-ack` frame kinds; one new phone Compose affordance (badge +
card stack) and its `onNotifDigest`/ack wiring; the producer call-sites (task finish,
Slack inbound). Audio/video playback is new but deferred.

## 10. Pressure test before committing (what we verified, what broke)

Before building, we verified the plan's load-bearing claims against the actual code
rather than trusting the design prose. Verdicts:

- **"Speak a notification with a raw `speak` frame + synthetic `turnId`." → FALSE.**
  `onSpeak` drops any frame whose `turnId` ≠ `currentTurnId`
  ([RemoteBrain.kt:521](../../node-dock/app/app/src/main/kotlin/dev/orbit/dock/agent/RemoteBrain.kt#L521)),
  and `currentTurnId` is only set when a real turn starts. The shortcut never existed.
  **Impact: none on the design** — §4b already routes speak through one autonomous turn,
  which is the only working path. The §3 table was corrected.
- **"Delivery copies `task-digest` (directed frame, 3 s sweep, free crash-recovery)." → TRUE,**
  with one caveat: the sweep is unconditional and directed frames **bypass the hub's
  1 MB backpressure shed** ([websocket-gateway.ts:146](../../orbit-station/server/src/core/websocket-gateway.ts#L146)).
  task-digest is tiny (4 string fields, no images), so this is free **today**. A
  notification digest carrying **base64 images re-pushed every 3 s** is a real cost
  cliff (~165 KB/s for 10 images). Mitigation already deferred in §6.1 (images via asset
  URL, not inline). Stage 1 is text-only, so it's a non-issue to start.
- **Persistence + producer wiring → SOUND, with the concurrency gap noted in §5.**
  The data-root + tmp-rename pattern is proven; the clean task-finish hook is right after
  `enqueueAutonomousTurn` in `onTaskSignal`
  ([brain/index.ts:120](../../orbit-station/server/src/modules/brain/index.ts#L120)).
  The only real gap was concurrent writers, fixed structurally (single-owner store, §5).

**Conclusion: no POC needed.** The one uncertainty (speak-out-of-turn) resolved *toward*
the design. The two residual risks (image cost cliff, concurrent writes) both have known
fixes that fold into the stages that need them. The plan is buildable as written — it is
parked by **priority**, not by unresolved risk.

## 11. Parked sub-idea: "a UI for the transient brain messages"

Separate from the durable inbox above, we explored a smaller idea: surface the
**existing** transient autonomous-turn queue (the "brain messages") in a UI. Investigating
it produced two facts that reshape — and currently deflate — the idea:

1. **The queue is server-side, not on the client.** It's `#autoQueue` in
   `DockBrainSession`
   ([session.ts:164](../../orbit-station/server/src/modules/brain/session.ts#L164)) —
   per-dock, in-memory, bounded to 4, drained when idle. The phone has **no message
   queue**; `DockTts.pending` is only a TTS sentence buffer, and the phone is otherwise a
   pure renderer of the current turn.
2. **A queued item is a trigger-prompt, not a displayable message.** When a task finishes,
   what's enqueued is a *prompt to the brain* — e.g.
   `[background task remind-after (t-aaaa) finished] You set a reminder to drink water` —
   which then **runs an LLM turn** that *generates* the spoken text. The human-facing
   message only exists *after* the turn runs, as `speak` frames, and **nothing retains a
   list of them** (observability records turns that ran, not the queue, and not their
   final spoken text).

So "a UI of the brain messages" forks into two different, both-non-trivial features:
**(a)** show the raw queued *triggers* + drain decisions (a debug view of the mechanism —
cheap, just expose `#autoQueue` over REST), or **(b)** capture and show **what the dock
actually said** (needs a new retained per-dock log of autonomous-turn spoken output —
the thing a person would actually recognise as "recent messages"). Neither exists today.

**Parked.** If revived, decide (a) vs (b) first — they share almost no implementation.
The multimedia-on-brain-output wish (image/audio/video alongside speech) is independent of
both and shares the §3 / §6.2 media-playback constraint: images are frame-native, audio/
video need a new phone player.
