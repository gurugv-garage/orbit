## 1. What v1 is (one paragraph)

A **per-dock Orchestrator**: a cheap, session-independent loop that reads a small **config of
tuning variables**, and for each known **behaviour** decides — by structured rules over those
tunings + cheap reads of the world (presence, conversation state, the snapshot stream) —
whether it should be **armed**, and when armed, whether to **run or stop** its task. It never
runs an always-on LLM. Body contention is already solved (the lease). v1 conducts exactly two
behaviours — **faceFollow** (a body behaviour: a periodic look-around window) and **wakeUp**
(a conversation behaviour: always-on "hey orbit" → "did you call me?") — a deliberate contrast
that proves the orchestrator conducts both body and non-body behaviours.

## 2. The four things it adds (unchanged from the vision §4 — restated for v1)

1. **Session-independent lifetime** — runs dock-scoped, including when nobody's talking.
2. **Declarative arming** — *armed ≠ running.* A tuning-driven rule starts the task only when
   the world warrants, stops it when it doesn't.
3. **A stable named handle for an intent** — you toggle "followFaces", not the churning task
   instances under it.
4. **A governable surface** — enable/disable/reconfigure standing behaviours (REST + console).

## 3. The model — three small pieces

```
  Orchestrator (per dock)         ← the cheap loop; ticks ~1 Hz
    reads: OrchestratorConfig      ← the TUNING VARIABLES (per dock)
    reads: world (cheap)           ← presence, convState.listening, snapshot signals
    for each Behaviour:
        decide(tunings, world) → DESIRED: off | armed | running
        enact via the Supervisor (start/stop the task) + the Lease (priority)

  Behaviour (a registered descriptor — NOT new runtime; it wraps a task)
    { name, taskName, priority, decide(ctx) → desired, defaults }

  OrchestratorConfig (tuning variables, per dock, live-editable)
    { <behaviour>: { enabled, ...knobs } }   e.g. faceFollow.enabled, idleReset.afterMs
```

**`decide` is the only behaviour-specific logic.** It's a **pure function**
`(tunings, world) → 'off' | 'armed' | 'running'` — arithmetic / rules over the tunings + cheap
world reads. No I/O, no LLM, unit-testable like the gate. This is the **seam your "code logic
later" sits at**: today `decide` reads number/bool tunings; a complex behaviour can later carry
a richer `decide` (still a cheap pure fn) without changing the orchestrator.

### 3a. What `world` contains — the orchestrator's full picture (built now, used lightly)

The orchestrator is aware of **more than presence**. `world` is assembled cheaply each
re-evaluation and passed to every `decide`:

```
  world = {
    present: bool, lastPresenceMs: number,        // perception (presence + the snapshot stream)
    identity: string | null,                       // recognized person, if any
    listening: bool, turnActive: bool,             // conversation state
    bodyHolder: { holder, priority } | null,       // the lease — who drives the body now
    tasks: [                                        // EVERY running instance on this dock…
      { name, instanceId, initiator: 'user'|'brain'|'self', ageMs }
    ],
    self: { /* the orchestrator's own maintained state: last decisions, since-when */ },
    now: number,
  }
```

The **task picture is deliberately rich**: which tasks run, **who started each** (a user via
the brain, the brain autonomously, or the orchestrator/`self`), and **how long** (`ageMs`).
v1 uses this *lightly* (e.g. faceFollow stays off while a live turn is active), but the signal
is THERE so a behaviour's `decide` — and the v2 learning task — can reason over it (e.g. "a
USER-initiated body task is running → don't arm faceFollow"; later "task X has run 5 min,
likely stuck"). Building the input shape now means v2 adds policy, not plumbing.

## 4. The two v1 behaviours (the proving pair)

Chosen as a **deliberate contrast**: one drives the BODY (lease-arbitrated), one drives the
CONVERSATION (no body) — so v1 proves the orchestrator conducts *both* kinds of behaviour.

### faceFollow — a periodic "look around" window (body behaviour)
The "idle 5 min → activate, run 15 min → sleep" intent. **Idle = no conversation** (no active
turn / not listening) — presence does NOT gate it (it can search for someone when it activates).

```
  decide(t, world):
    if !t.enabled: return 'off'
    if world.turnActive || world.listening: return 'off'   // never during a conversation
    // ACTIVE WINDOW state (held in the orchestrator's self-state per behaviour):
    //  - not active + conversation-idle ≥ t.activateAfterMs (5 min)  → ACTIVATE (start a window)
    //  - active + window age ≥ t.runForMs (15 min)                   → DEACTIVATE (sleep)
    //  - after deactivating, won't reactivate until another full idle stretch
    return active ? 'running' : 'off'
```
Runs the `face-follow` task at lease priority 30 while the window is open. The 5-min idle
clock + 15-min run cap live in the orchestrator's per-behaviour `self` state (so `decide` stays
pure — it's handed `idleMs`/`windowAgeMs` in `world`/`self`).

### wakeUp — always-on "hey orbit" (conversation behaviour, no body)
```
  decide(t, world):  return t.enabled ? 'running' : 'off'   // always active when enabled
```
When the dock is **NOT listening** and a final STT transcript matches the wake phrase
("hey orbit"), wakeUp **speaks "did you call me?" and opens the listening window** (`tapOpen`).
No body, no lease.

**Verified (2026-06-26, in code) — what exists & why it's dead:** the dock waking on a phrase
does NOT work today, and the reason is a **dormant legacy on-phone wake pipeline** left over
from before the server-STT cutover:
  - `node-dock app/.../perception/PorcupineWakeWord.kt` — a real Porcupine detector, but
    **disabled** (no `PORCUPINE_ACCESS_KEY` in local.properties → "stays disabled, never
    fires"); built-in keyword is "jarvis", not "orbit".
  - `SileroVad.kt` — the on-phone VAD, known **broken** (LSTM state diverges, prob frozen
    ~0.0005 — see TODO/RCA notes).
  - `PerceptionPipeline.kt` instantiates both and runs them on mic frames, but **STT is now
    server-side** (`stt-watch`) and listening is **tap/palm-only** (`tapOpen` + VAD-extend).
  - `PerceptionEvent.WakeWord` survives only **repurposed** as a generic "open listening"
    signal (labels `(tap)`/`(dev)`/`(barge-in)`/`(aec-test)`), not an actual wake word.
So: there is no *working* wake-on-phrase, and the on-phone Porcupine/Silero path is dead
weight. The NEW wakeUp does the match **server-side on the STT transcript** (where audio
understanding lives now), NOT on the phone. It does NOT overlap the brain's "addressed
decision" (that decides whether a heard utterance becomes a turn *once a window is open* — a
different thing from *waking from idle on a phrase*). **Cleanup of the dead on-phone wake
pipeline is tracked separately** (it's woven into MicCapture/PerceptionPipeline/bus/tests) so
it doesn't entangle the orchestrator build. This wakeUp is the simplest server-side wake;
richer wake behaviour comes later.

**The one mechanical question (resolve in build step 1):** final transcripts arrive via
`TranscriptApi.onFinal`, which is **single-consumer** (the brain registers it once). So the
wake-phrase match can't just add a second `onFinal` listener. Two clean options — pick during
the build, don't pre-bake:
  (a) the match rides in the brain's existing final-transcript handler, GATED by the
      orchestrator's `wakeUp.enabled` (brain owns the mechanism; orchestrator governs it); or
  (b) add a small transcript fan-out so the wakeUp logic can subscribe independently.
Either way the orchestrator's wakeUp behaviour holds the tunings (phrase/prompt/enabled) and is
the on/off + governable surface; the *match* lives wherever finals already are. (Leaning (a) —
no new plumbing, and "open listening on a phrase" is naturally a conversation-layer action.)

Between them these prove every mechanic: a behaviour that's **conditionally windowed** by time
+ conversation state (faceFollow), a behaviour that's **always-on** (wakeUp), start/stop a
task, a body behaviour mediated by the lease, a non-body behaviour, and "off when disabled" —
all tuned live.

## 5. Tuning variables (v1) — the "learned later" surface

Per-dock config block `orchestrator.<dock>` (lives in the existing config registry, so it's
already push-on-change + console-editable):

```
  faceFollow.enabled         bool   default true
  faceFollow.activateAfterMs number default 300000   (5 min no-conversation → activate a window)
  faceFollow.runForMs        number default 900000   (15 min, then deactivate until idle again)
  wakeUp.enabled             bool   default true
  wakeUp.phrase              text   default "hey orbit"   (the wake phrase to match in STT)
  wakeUp.prompt              text   default "did you call me?"
  tickMs                     number default 1000     (conductor cadence)
```
These are the knobs in the **Behaviour tab** — editable in real time (the conductor reads the
live config each tick, so a change applies on the next tick).

**Why this is the right "start simple":**
- It's just **data** — sensible defaults, console-editable, hot-applied (the conductor reads
  the live config each tick). No learning needed to ship.
- **The learning door (v2, NOT now):** an episodic *reasoning task* the orchestrator spawns
  reviews history (who's around, when, what was useful) and **writes back** these same tunings
  (e.g. flips `faceFollow.enabled` per time-of-day, raises `idleReset.afterMs`). The conductor
  reads them next tick — *learned conducting with zero core change.* v1 just makes sure the
  tunings are the single point everything flows through.
- **The code-logic door (later):** a tuning that's normally a value can become a small policy
  fn injected into `decide` for a complex behaviour — the `decide` signature already allows it.

## 6. Lifecycle + the cheap loop (the actual algorithm)

The decision logic is one function, `reconcile(dock)` — assemble `world`, ask each behaviour
`decide`, and start/stop tasks to match. It is **re-evaluated by two triggers** (same
`reconcile`, so they can't disagree):

- **CADENCE** — a ~1Hz tick. Catches anything events would miss + time-based rules
  (idle-reset's `afterMs`). **v1 ships cadence-only** — it already reacts within ~1s.
- **EVENT** *(named here, DEFERRED to v2)* — re-`reconcile` immediately on a decision-changing
  event (a task starts/stops, presence changes, a turn begins/ends) for instant
  responsiveness. v1 doesn't wire events; the loop is written so adding them later is just
  "call `reconcile` from these subscriptions" — no logic change.

```
  reconcile(dock):                                // cadence now; event-or-cadence in v2
    cfg   = config.orchestrator[dock]             // live tunings
    world = assembleWorld(dock)                   // §3a — presence, conv, lease, ALL tasks (+initiator,age), self
    for b in behaviours:
        desired = b.decide(cfg[b.name], world)    // pure
        running = supervisor.list(dock).some(i => i.name === b.taskName && i.state === 'running')
        if desired==='running' && !running: supervisor.start(b.taskName, …)  // arm → run
        if desired!=='running' &&  running: supervisor.stop(that instance)    // disarm → off
    log a [orch] line ONLY when a behaviour's desired/running transitions
```

- **Idempotent + self-correcting:** each reconcile matches desired vs actual, so a missed
  start/stop heals on the next tick (same discipline as the lease/heartbeat).
- **No always-on LLM** (vision §5 held). The conductor is rules + cheap reads.
- **Body arbitration is NOT the conductor's job** — the lease already handles "two behaviours
  want the body." The conductor decides *whether a behaviour runs at all*; the lease decides
  *who drives the body right now*. Clean separation.

## 7. Where it lives + the surface

- **Code:** a new `modules/orchestrator/` (per-dock `Orchestrator` instances + a registry of
  `Behaviour` descriptors). It depends on the supervisor (start/stop), config (tunings), and
  cheap world reads (presence/convState/perceive) — all existing.
- **REST:** `GET /api/orchestrator/:dock` (each behaviour: enabled, desired, running, why) ·
  `POST /api/orchestrator/:dock/:behaviour {enabled, ...knobs}` (edit tunings).
- **Console:** a small badge/card per dock — list behaviours, live state (off/armed/running),
  a toggle + the key knobs. (Debuggability was the faceFollow lesson; make the conductor's
  decisions visible.)
- **Log:** `[orch] <dock> <behaviour> <off|armed|running> (why)` on every transition — the
  ground-truth trace for testing (like `[ff-event]`).

## 8. Build order (small steps, each verifiable)

1. **`wake-up` task** — watches the STT queue; on the wake phrase while not listening, speak
   the prompt + open the listening window. (Confirm the STT-text→station path first.) The
   non-body behaviour to conduct alongside faceFollow.
2. **`Orchestrator` + `Behaviour` registry + `decide` fns** — pure, unit-tested (decide truth
   table per behaviour incl. faceFollow's idle-window state machine; reconcile loop start/stops
   the right tasks given a fake supervisor).
3. **Tunings in the config registry** + REST.
4. **Wire it into the station** (one Orchestrator per dock, ticking ~1 Hz).
5. **Console Behaviour tab** (faceFollow + wakeUp, live state + the tunings; leave Tasks tab
   as-is + a dock-filter freebie).
6. **Hardware test:** no conversation 5 min → faceFollow window opens, it looks around/follows;
   15 min → it sleeps; a brain turn preempts mid-follow (lease); say "hey orbit" while idle →
   wakeUp speaks "did you call me?" + opens listening; toggle either `enabled=false` in the
   console → it stops. All visible in `[orch]` + `/holder`. (Activate timers will be tuned down
   for the test, then restored.)

## 9. Open / deferred (kept OUT of v1 deliberately)

- **Learned write-back** (the reasoning task that tunes the variables) — v2. v1 ships fixed
  defaults + console editing; the tunings are structured so v2 is a pure add.
- **`decide` as injected code-logic** — the signature allows it; no behaviour needs it yet.
- **The "behaves" plug-in contract** (independently authored/loaded policy modules, vision
  §5a/§10) — v1 hardcodes the two behaviour descriptors in the registry. Pluggable loading is
  a later generalization once we have >2 and a reason.
- **Event-driven re-evaluation** (§6) — v1 is cadence-only (~1s reactive); event triggers
  (task start/stop, presence change, turn start/end → immediate `reconcile`) are a v2 add the
  loop is already shaped for.
- **ResourceMonitor** (station-level throttle) — vision §6, last.
- **Trigger discipline escalation** — v1 triggers are strictly cheap rules; no escalation path.
