## 1. The behaviour, and its hard boundary

"Find faces in the room and follow them." On this body that means **head-tracking**:
keep a face centered by panning/tilting. The body has **2 DOF** —
`foot` (pan, ±90°) and `neck` (tilt, −60…+35°) — and **no locomotion**
([brain/schemas.ts] `DEGREE_LIMITS`). So:

- ✅ pan/tilt to keep a face centered; search-sweep when none; lock on when found;
  re-acquire from last-known position when lost ("search meaningfully").
- ✅ follow a *named* person (face-api recognition) or the most-salient face.
- ❌ **pursue** someone who leaves pan range or the room — can only search + wait.
  (Locomotion is node-rover, a different device.)

This is an **action** behaviour (goal = move the body), unlike the curator (goal =
maintain memory). Per the purpose-rule it's a genuine behaviour, not pipeline work.

## 3. What must be built

### 3a. A faster face-box source (medium)
Identity samples at `IDENTITY_PERIOD_MS=2000` (2 s) **and is change-debounced** — far
too slow + sparse for a tracking loop. The box *data* exists; the *cadence* doesn't.
faceFollow needs boxes at **~4–5 Hz**. Options: (a) a dedicated faster `recognizeAll`
read for tracking (cheap; competes with vision for the shared frame grabber), or (b) a
box-only fast detector tap (no recognition/expression — lighter, smoother). Start with
(a); measure jitter on the real dock before building (b).

### 3b. The controller (medium, well-understood — NOT fragile)
Pure-ish arithmetic + a state machine, testable like the curator's logic:
- **box-center error → `foot`/`neck` deltas**, with **deadband** (no twitch when ~centered)
  and **slew limit** (glide, not lurch); map normalized error → degrees within limits.
- **state machine**: `SEARCH` (sweep) → `TRACK` (center) → `REACQUIRE` (lost → sweep
  from **last-known** foot/neck outward, then full sweep) → back. Holds last-known pos.
- **target policy**: most-salient (largest×centered) or a named person (recognition);
  build salient first, named is a filter on top.
- **loop-closure**: every move ALSO calls `pushCommand` (3 above), in lockstep.

## 4. The actuator lease (the keystone)

**The problem.** `MotionExecutor` is today "the body's single master, last-write-wins."
Its callers, all unmediated: the brain `move` tool + `set_face`→`playGesture`
([brain/tools.ts]), tasks' `runSteps` ([tasks/register-capabilities.ts]), the bodylink
console ([bodylink/index.ts]), and the session lifecycle's `stop()` on turn start/cancel
([brain/session.ts]). The moment faceFollow *also* drives the body **continuously**, N
masters fight: last-write-wins becomes thrash, and a crashed follower can freeze the body
mid-pan. So we put a **lease** in front of the executor — the one new piece of real
infrastructure.

**Decision: a PRIORITY lease (not "brain always wins").** A caller acquires the body at
a **priority**; the highest active priority holds it; equal priority = last-write-wins
(today's behaviour, preserved). Chosen over "brain always preempts" because it's strictly
more general — a future safety/emergency behaviour can outrank even the brain — at the
cost of a small amount more machinery. Indicative levels (open, tune in build):

```
  emergency / safety        100   (future: e.g. "stop, about to fall")
  brain turn / explicit move  60   (a conversation gesture, a user `move` command)
  faceFollow (tracking)       30   (the reflex — yields to the brain, outranks idle)
  console / manual sliders    20
  idle / heartbeat             0
```

So: a brain turn (60) **preempts** faceFollow (30) — faceFollow yields, the brain
gestures, then faceFollow resumes when the brain releases. An emergency (100) outranks
even the brain.

**The lease contract** (what the executor grows):
- `acquire(dock, holder, priority) → lease | null` — granted iff priority ≥ the current
  holder's; returns a handle. A higher-priority acquire **preempts** the current holder
  (notifies it so it can stop cleanly).
- `setTargets/runSteps(lease, …)` — motion calls now require a valid, current lease;
  a call from a superseded lease is **rejected** (no thrash).
- `release(lease)` — give the body back; the next-highest waiter (if any) takes it.
- **LEASE TTL + liveness** — a lease **auto-expires** unless renewed (faceFollow renews
  each control tick). So a **crashed/hung follower releases the body within one TTL**
  (not frozen forever) — the safety property. Brain/console leases are short-lived
  (a gesture) so they expire naturally.
- **preempt callback** — the held loop is told "you've been preempted" so it stops its
  own ticking (faceFollow pauses; doesn't keep computing corrections into the void).

**Backward-compat:** existing callers (brain, console, tasks) wrap in a short
auto-acquire-at-their-priority + release, so today's behaviour is unchanged when
faceFollow isn't running. The lease only *bites* when two holders contend.

**Where it lives:** on/beside `MotionExecutor` in `bodylink/` — it IS the body's master,
so arbitration belongs there, mediating every caller (not bolted onto faceFollow).

## 4b. v1 AS-BUILT — a plain task, no lease (the simplest path)

Decision (deliberate, scope): **v1 faceFollow is a plain TASK** (`tasks/packaged/
face-follow/`), not the lease/orchestrator. The follow LOOP genuinely is just a task —
`recognize` (now returns the box, §3a was a 3-line capability add, NOT a new sensor) +
`move`, in a `while` loop. The **lease is deferred** — its only job is safe coexistence
with the live brain, which v1 sidesteps by *not running mid-conversation*. §4's lease
design stands as the "when coexistence matters" plan.

What shipped:
- **`recognize` returns `box {x,y,w,h}`** (option A — reuse the existing sensor, +name
  for free; a box-only fast detector is the fallback only if per-tick recognize proves
  too heavy at the rate we want — not pre-built).
- **`control.ts` (pure, tested):** `pickTarget` (named/salient + continuity LOCK),
  `nextPose` (proportional pan+tilt, deadband + slew, sign-correct), `sweepStep` (search
  from last-known outward). 16 unit tests.
- **`task.ts`:** the loop — recognize → pick → track (`nextPose`→`move`) or search-sweep
  when missing; `target?` param (named) else most-salient; checkpoints its pose.
- **NOT in v1:** the lease (§4), ego-motion loop-closure via `pushCommand` (no task
  capability for it yet; the per-tick re-recognize self-corrects a panned-past face).

### Multi-person policy (how a crowd doesn't break it)
`recognize` returns ALL faces. The risk is **oscillation** — two people → flip-flop. The
fix is **select-one + lock + deliberate switch** (`pickTarget`):
- **named** (`target` set) → follow only that recognized person, ignore others;
- else **most-salient** (largest × most-central);
- **continuity LOCK** — once chosen, keep following *that* person (matched tick-to-tick
  by box proximity), switching only when they leave OR a challenger is **decisively** more
  salient (≥1.4× — hysteresis, same anti-flicker discipline as the identity stream). v1
  locks by **box-continuity** (no recognition dependency); named-lock rides on top.

## 5. Build order (lease-first)

The lease is the risk; prove it before the control math (matches behaviour-orchestrator
§11 spirit):

1. **The lease/arbiter on MotionExecutor** — acquire/release/TTL/preempt + priorities;
   wrap existing callers compat. Unit-test the arbitration table + TTL expiry.
2. **Prove the keystone with a TRIVIAL mock follower** — a behaviour that just pans
   back-and-forth holding a lease. Verify, on the real body: acquire → stream moves →
   **kill it → body releases within TTL** → brain regains control; brain turn **preempts**
   it mid-sweep. *This is the make-or-break test.*
3. **Faster box source** (3a) — drop to ~5 Hz; measure jitter.
4. **The real controller** (3b) — error→delta, deadband, slew, the state machine,
   loop-closure. Replaces the mock follower.
5. **Named-target** + tuning on the real dock.

If step 2 is clean, the design holds; if the lease is ugly there, fix it before any
control math rests on it.

## 6. Open points

- **TTL value** — short enough that a dead follower frees the body fast (~1–2 control
  ticks, e.g. 500 ms–1 s), long enough not to expire mid-track. Tune in step 2.
- **Preempt resume** — when the brain releases, does faceFollow resume from where it was
  or re-acquire the face fresh? Probably re-acquire (the view changed during the gesture).
- **Box-source cost** — 5 Hz `recognizeAll` shares the frame grabber with vision; watch
  for GPU/CPU contention (the same shared-Metal concern as elsewhere). The box-only tap
  (3a-b) is the fallback if jitter/contention is bad.
- **Console surface** — faceFollow is a behaviour: enable toggle + a "tracking/searching"
  status + the current target, in the console (the orchestrator's governance surface).
- **The orchestrator itself** — faceFollow is the first action behaviour; whether it runs
  under a minimal behaviour-runtime now or standalone-then-extracted is a
  behaviour-orchestrator decision, not this doc's.

## 7. The `perceive` stream — on-device MLKit as the fast face source (answers §3a)

**Why.** The live test (2026-06-25) showed the station-side `recognize` (face-api on a
WebRTC-pulled frame) is the wrong source for tracking: it (a) **misses off-frontal faces**
— a full right-side sweep at face height detected no one until a frontal face appeared at
the edge — and (b) reads the **box `y` far too low** (≈0.89 for a face the console centers
at `neck=0`), making the controller dive the neck to its down-limit and stare at the floor.
Neither is a controller bug; both are signal-quality problems in the face-api path.

Meanwhile the **phone already runs MLKit face detection** (`FaceTracker.kt`, ~1 Hz,
`PERFORMANCE_MODE_FAST` + `LANDMARK_MODE_ALL` + `CLASSIFICATION_MODE_ALL` +
`enableTracking`) and emits `FaceSeen(x,y,size)` on a **phone-local** `PerceptionBus` —
consumed only for wake/presence/UI, never forwarded. MLKit is on-device (low latency),
mirror-corrected, and **good at off-frontal faces** — exactly what tracking needs. Decision
(2026-06-25, with the user): **forward the on-device signal to the station as a new
`perceive` perception stream**, and have faceFollow consume *that* instead of `recognize`.

**Scope (this round):** vision-only (audio VAD/level deferred to its own stream later).
Push **everything MLKit already computes** — zero new inference; the cost is serialization
+ one `StationLink.publish` per tick.

**Envelope** (`perceive` topic, dock → station, ~1 Hz; payload):
```
{ faces: [ {
    x, y, size,                 // NDC box center (−1..+1, mirror-corrected) + width frac
    bbox: { l, t, r, b },       // full rect, NDC
    yaw, pitch, roll,           // MLKit headEulerAngleY/X/Z — head pose ("facing me?")
    trackingId,                 // stable across frames → continuity lock (better than proximity)
    smile, leftEyeOpen, rightEyeOpen,   // MLKit probabilities (raw, pre-fusion)
    landmarks: [ { type, x, y } ]       // up to 11 points, NDC
  } ],
  emotion: { kind, confidence },        // FER+ fusion (already emitted)
  gesture: { name, palm, score } | null,// MediaPipe (already emitted)
  identity: { name, confidence } | null,// on-device (already emitted)
  zoom: { ratio, min, max }             // CameraX zoomState — current FRAMING (free, this build)
}
```

**Zoom vs Focus — two separate camera levers (2026-06-25).** They are distinct concerns:
- **Zoom = framing** (field of view). CameraX `cameraControl.setZoomRatio` +
  `cameraInfo.zoomState`, free off the handle we already hold (`bindToLifecycle`). Digital
  only on this front cam — crops+upscales the already-small (320×240) analysis frame, so it
  helps *framing/centering* a distant subject but does **not** recover real detail at distance.
- **Focus = fidelity** (dynamic resolution): run analysis/vision at low res when nothing
  needs detail (cheap/battery), raise it on demand ("read that label", enroll a face). The
  seams already exist — `FaceTracker.bind` pins analysis at 320×240 (with a comment that
  recognition reliability is bound to it), `encodeJpegBase64` caps the vision JPEG at
  320px/q70, and `captureRecognitionJpegBase64` already takes a 640×480 on-demand still. A
  `focus.level` of `rough | normal | detail` would map to concrete (analysisRes, jpegMaxEdge,
  quality) tuples — the brain/behaviours drive intent, not pixels.

**This build carries `zoom` read-state only.** The focus/resolution signal is **deferred**
until the rough/normal/detail levels are designed (raising res has real ISP/CPU cost on the
2018 phone — the contention the FaceTracker comments warn of — and recognition thresholds
were tuned at 320px, so a raise needs re-validation). The **write** sides — `set_zoom` and
`set_focus` brain tools (the first *downward* camera controls) — are both follow-ups.

**Dynamic resolution — safe to build, deferred to its own pass (2026-06-25).** Verified the
station does NOT bake in a fixed resolution: `frame-grabber.ts` parses W/H from the VP8
keyframe and `recognizer.ts` normalizes boxes by actual W/H → coords are 0..1 regardless of
res. So changing res won't break downstream geometry. The ONE real dependency is that
recognition match thresholds were *tuned* at 320px crops (a raise shifts embedding distances
→ re-validation, not a crash). Safe half (lower fidelity for cheap mode; use the EXISTING
on-demand 640 still for "detail" without renegotiating the live stream) is reliable; raising
the live analysis res is the part needing re-tuning. Deferred whole to keep this build
focused on landing the follow.

**Noise control on the stream (REQUIRED, in the forwarder).** ~1 Hz raw + a few-% box
jitter on a still face would spam the bus/console. The forwarder dedups: **change-significant
+ rate-limited + heartbeat** (same shape as the identity stream's CONFIRM/DROP hysteresis):
suppress a frame ~identical to the last sent (Δx/Δy/Δsize < ε, same trackingId + emotion +
gesture + identity); coalesce fast change to a min-interval (≤ ~5 Hz); always send
transitions (face appear/disappear, identity change, new trackingId) immediately; and send a
heartbeat at most every ~2–3 s when unchanged so the station knows the face is *still there*
(not stale). Dedup lives in the forwarder (one place), not the detector path.

**Live test 2 (2026-06-25, build 14 OTA'd to dock-redmi).** The perceive stream worked
end-to-end: rich frames (box + Euler + trackingId + landmarks + emotion + zoom) flowed, and
the **eye-midpoint anchor fixed the neck-dive** — a centered face now reads y≈0.44 (the eyes)
instead of ≈0.89 (the sagging box), so the head centers and holds level instead of diving to
the floor. faceFollow acquired, centered, and held on a still person. **New problem found:**
MLKit detection is **bursty** — it drops a still, frontal, *centered* face for multiple
seconds at a time (observed `/perceive` going blank for 7–10 s between detection bursts).
The old fixed `lostAfter` grace (~8 ticks) then expired and the head **swept away from a
person who never moved** ("it moved, I didn't"). Fix: **asymmetric grace** — when the lock
was CENTERED at drop-out, wait `lostAfterCentered` (default 4× `lostAfter`) and keep looking
*right here* before concluding they left and sweeping; a centered person doesn't teleport, so
a gap is a detection dropout, not a departure. `lock.centered` carries the "were we holding
ON them" bit; both tunable as task params. Encoded as a closed-loop test (centered drop-out →
holds still, never sweeps, re-locks on return). The deeper detection sparsity is the
remaining real limit (a faster/steadier on-device detector, or fusing the burst with a
predicted hold, is future work).

**faceFollow is a BEHAVIOUR, with cooldown-on-yield (2026-06-25).** Reframed from a one-shot
task to a **standing behaviour** (user): it never completes — find → follow → hold when still
→ sweep when lost → forever. It only **yields** the body when another mover takes it, then
**cools down** before resuming. This is the actuator-lease idea in its lightest form — observe
who's driving, defer, don't hard-lock:
- **Move-source tagging.** `MotionExecutor` now records `lastMover {tag, at}` per dock; every
  command path passes a source — `brain-turn` (the brain's move/gesture tools), `console`,
  `task:<instanceId>` (the `move` capability stamps the calling task), `station`. New read
  capability `bodyMover` returns it.
- **Yield + cooldown.** Each tick faceFollow reads `bodyMover`; a tag that isn't its own
  `task:<id>` and newer than its last move = a FOREIGN mover → it stops commanding and arms a
  `cooldownMs` window (default **60 s**, "we'll adjust"; `FF_COOLDOWN_MS` / task param). While
  cooling it issues nothing (won't fight) but keeps the lock, so it resumes on the same person.
  A foreign move during cooldown re-arms the full window. **Cold start never yields** — the
  cooldown only triggers on a real foreign move. Pure + injected-clock tested (cold-start
  immediate, yield, hold-through-cooldown, resume, re-yield).

**Adaptive on-device detection cadence (2026-06-25).** The live bottleneck was perceive-stream
sparsity (~0.3–1 Hz): MLKit ran at a fixed 1 Hz on the phone and the forwarder deduped further.
Fixed-rate constants are guesses that rot (different phone, lighting, CPU load), so instead of
re-tuning, `FaceTracker` now runs the face path at an **adaptive interval**: it measures each
run's full cost (encode + async detect + FER, via `addOnCompleteListener`) as an EMA and sets
the next gate to `cost × SLACK`, clamped ~5 Hz…2 Hz. A fast phone runs fast; a loaded/slow
phone self-limits — no per-device rate to tune. (Build 15.)

**`objects` — reserved, not built (2026-06-25).** MLKit *can* detect objects, but via a
SEPARATE model (`object-detection`: boxes + tracking + only 5 coarse categories) — a NEW
analyzer pass = real CPU/ISP cost on the 2018 phone, NOT free like the face extras. Object
*naming* is already the vision sidecar's job (Qwen-VL, open-vocabulary, far better than
MLKit's 5 labels) — just slower. The envelope reserves room for a future `objects: [...]`
array (fast on-device boxes/motion the slow sidecar can't give), but adding it is a deliberate
cost/value call for its own pass; phone stays face-only here.

**What the 11 landmarks reliably buy us** (computed server-side, so we iterate without a
re-flash): **eye-midpoint anchor** — center the head on the eyes, not the box geometric
center (the box center is what reads low; the eye midpoint is stable) — this is the direct
fix for the neck-dive; a **gaze-on-dock** boolean (yaw within ±N° AND eyes open) for a
future "follow whoever's attending to me" salient-pick; and a **relative** distance proxy
(inter-ocular px). NOT reliable from 11 coarse points: precise eye gaze, absolute distance,
viseme/talking, micro-expression — don't build behaviour on those.

**Cannot get cheaply (excluded):** full-body pose / hand skeleton joints / depth / iris-gaze
/ age — all need new models on a 2018-class phone. "body-track" here means **head
orientation** (Euler), not a body-pose model.

**Plumbing.** The high-rate face geometry is a **live latest-state** signal (faceFollow reads
"the last face frame"), NOT the heavy 1000-cap `SnapshotStore` memory ring — only
emotion/identity (kinds the ring already models) optionally also land as snapshots.
faceFollow's `recognize`-based `faces()` is replaced by a read of the per-dock latest
`perceive` faces; the box→error math is unchanged (it just gets a better box, and later the
eye-midpoint). Use of `yaw/pitch/roll` in the controller is **transport-now, use-later** (the
data flows + is visible; salient-by-facing is a follow-up).
