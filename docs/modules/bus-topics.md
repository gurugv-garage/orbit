# Bus topics

The station's spine is one in-process **`Bus`** ([`core/bus.ts`](../../orbit-station/server/src/core/bus.ts)) — a generic
`Map<topic, Set<handler>>` with `on(topic, fn)` and `publish(msg)`, nothing more.
Everything that happens — a peer connecting, a config change, a body command, a
perception result — is a `BusMessage` on a **topic**. The
[`WebSocketGateway`](websocket-gateway.md) bridges the bus to/from sockets;
modules subscribe to react.

> **Say "topic," not "bus."** There is exactly one bus and **15 topics** on it.
> "The dock's facts go on the bus" and "perception results go on the bus" are
> both true but useless — they're different *topics* (`client` vs `perception`),
> opposite directions. The bus is the pipe; the **topic** is the traffic type.
> Name the topic. (This doc exists because that distinction kept getting blurred
> — e.g. the handler once called `#onBusFact` is now `#onClientFact`, because it
> only ever handled the `client` topic.)

## The registry

Direction is from the station's point of view: **↑** peer→station (a peer
publishes over WS, the gateway lands it on the bus), **↓** station→peer (a module
publishes, the gateway sends it out as an `event` frame), **↔** both, **internal**
= module-to-module only.

| Topic | Dir | Owner / canonical consumer | Producers | Carries (kinds) |
|---|---|---|---|---|
| **`station`** | internal ↓ | everyone (docks, ota, config, media, bodylink, perception) | `WebSocketGateway` (peer lifecycle), docks, bench | `peer-joined` / `peer-left` / `peer-updated`, `dock-updated` / `dock-removed`, `presence`, `bench` |
| **`config`** | ↔ | config module | console peer + config | defaults + live changes (push-on-change to firmware/app) |
| **`bodylink`** | ↔ | bodylink module | bodylink executor (+ ESP32 peer) | `command` (↓), `state` / `applied` / `digest` (↑), `ping` |
| **`ota`** | ↔ | ota module | ota module (+ device peer) | availability offers, `progress`, `result` |
| **`media`** | ↑ | media module (SFU) | dock + browser peers | WebRTC signaling — SDP offer/answer, ICE candidates |
| **`client`** | ↑ | [`PerceptionProcessingHub`](perception-processing-hub.md) (`#onClientFact`) | dock peer | on-device facts → channel `client.<kind>` (VAD, battery, face) |
| **`agent`** | ↔ | brain (+ rpc, feedback) | dock peer (↑) + brain (↓) | ↑ `user`, `turn-request`, `hello`, `recording`; ↓ `speak`, `tool-call`, `tool-result`, `turn-status`, `brain-status`, `dock-error`, `task-digest` |
| **`perception`** | ↓ | perception module + `PerceptionState` | `PerceptionProcessingHub`, perception module | `snapshot`, `state`, `recognize-result`, `enroll-result` (directed to dock + undirected copy) |
| **`perceive`** | ↑ | perception module | dock peer | on-device MLKit face-track frames (the fast faceFollow source) |
| **`tasks`** | ↔ | brain (tasks) | task-process peer (↑) + brain (↓) | ↑ attach / status / notify / ask / finish; ↓ init / input / stop |
| **`obs`** | internal | observability module | brain + modules | `event`, `brain-debug` (Session / Turn / Step trace ingest) |
| **`conductor`** | — | conductor module | — (REST surface only, for now) | per-dock behaviour + task governance |
| **`slack`** | ↑ | slack module | Slack (Socket Mode) peer | inbound `message` / `mention` / `dm` (ingest only) |
| **`feedback`** | ↑ | feedback module | app peer | session-feedback flag → full debugging dump (MD) |
| **`*`** | — | **`WebSocketGateway` only** | — | the one wildcard subscription — see [The bridge](#the-bridge) |

Kinds are the ones observed in the tree at time of writing; a topic isn't a
closed set — a producer can send a new `kind` and consumers ignore what they
don't handle.

## Three patterns make it legible

1. **One topic per module, named after it.** Each module owns a topic matching
   its name (`config`, `bodylink`, `ota`, `media`, `perception`, `obs`, `slack`,
   `feedback`, `conductor`) and subscribes to it. The brain owns `agent` +
   `tasks`; perception also consumes `perceive` and `client`. So "which module
   handles topic X" is almost always "the module named X."
2. **`station` is the shared broadcast.** Peer presence / health / directory
   changes affect everyone, so `station` is the one topic with many
   cross-module consumers (docks, ota, config, media, bodylink, perception).
3. **`*` has exactly one subscriber.** Only the `WebSocketGateway` subscribes to
   `*` — that's the bus→socket bridge. Every other subscription is a single,
   specific topic.

## Producer split = direction

You can't see direction from the topic name alone; it comes from *who produces*:

- **Station-produced** (module → bus → out): `station`, `ota`, `perception`,
  `obs`, and the `↓` half of `bodylink` / `agent` / `tasks` / `config`.
- **Peer-produced** (peer publishes over WS → gateway → bus): `client`, `agent`
  (↑), `tasks` (↑), `perceive`, `media`, `feedback`, `slack`, and the device
  half of `bodylink` / `ota` / `config`.

A peer-produced message is stamped `source: peer.id` by the gateway; a
station-produced one is `source: 'station'`.

## The bridge

The gateway's single `bus.on('*')` turns **every** bus message into an outbound
`event` frame, delivered only to peers **subscribed to that message's topic**
([`websocket-gateway.ts`](../../orbit-station/server/src/core/websocket-gateway.ts)).
This is why a module never touches a socket: it publishes to a topic, and any
peer that `subscribe`d to that topic receives it. The reverse — a peer's inbound
`publish` frame — is landed on the bus by the same gateway, stamped with the
peer's id as `source`.

## Directed delivery (`to` / `toAddr`)

Most messages broadcast to all topic subscribers. A `BusMessage` may instead be
**directed** ([`isDirected`](../../orbit-station/server/src/core/bus.ts)):

- **`to`** — a specific peer id.
- **`toAddr`** — a `(dock, component)` address, resolved against the live roster
  at fan-out time (so reconnects / hardware swaps need nothing special on the
  send path). Preferred for module→dock pushes.

Delivery is still gated on topic subscription. In the tree today, directed
delivery is used on `perception` (results to the addressed dock), `bodylink`
(commands to a dock's body), and `tasks` (input to a specific task process). A
directed message also rides an **undirected copy** where a console needs to see
it too — e.g. perception results are published twice (directed to the dock,
undirected for the Studio + world-state aggregator).

## Adding a topic

1. Add it to the `Topic` union in
   [`core/protocol.ts`](../../orbit-station/server/src/core/protocol.ts) with a
   one-line comment (that union is the canonical list).
2. Subscribe in the owning module's `init(bus)` via `bus.on('<topic>', …)`.
3. Publish with `bus.publish({ topic, kind, payload, source, to?/toAddr? })`.
4. Add a row here.
