orbit-station · data-flow walkthrough

One moment, packet to behaviour

Follow one moment end to end: a dock's camera photons and mic audio enter over WebRTC, become perception, ground a brain turn, and return as spoken words and body motion. Two transports, keyed by the same peer id, crossing at the bus — traced from packet to behaviour.

plane 1 Control & facts

WebSocketGateway — owns the /ws server. hello / subscribe / publish / event frames. Small discrete JSON.

plane 2 Live media

Sfu + PerceptionProcessingHub — WebRTC RTP in, tapped and fanned to perception processors. Continuous.

Note the memory arm: every writer-job reads the SnapshotStore — nothing feeds the summarizer directly (speech-watch writes the record; bg-audio patches that same record in the store; rolling-summary then reads the window). The ✗ inside the rolling-summary node marks the current gap: the acoustic audioKind/salience fields ARE on the record in the store (and now persist), but summarizer.stitch() only renders the text — so the ego's summary never sees "startling"/"music" as a weighted signal. Orientation

The same peer id keys both planes

The dock app's peer id is the unifying key. On the media side the SFU calls it streamId; on the bus side every WS frame is stamped with it as source. Because they're the same string, one processor subscription can correlate a video frame with a battery-level fact from the same device.

note

WebRTC is negotiated over the WS — the SDP offer/answer and ICE candidates ride as media-topic frames through the WebSocketGateway. But once negotiated, the media itself flows as RTP into the Sfu, not through the gateway. Signaling = control plane; payload = media plane.

one bus, many topics

There is one Bus (a generic Map<topic, handler> in core/bus.ts) and 15 topics on it (full registry). The dock's facts and perception results are different topics (client vs perception), opposite directions — not "the bus." The bus is the pipe; the topic is the traffic type. Below, each is named.

Control plane · inbound

W1A peer dials /ws and the gateway bridges it to the bus

The WebSocketGateway owns the one WebSocket server. Every peer — dock phone, ESP32 body, browser console — connects here, says hello, and gets a welcome with its resolved dock/component. Two inbound frame types drive everything:

  • subscribe — the peer names the topics it wants; the gateway records them per-peer.
  • publish — the peer emits a fact; the gateway forwards it to the in-process bus, stamped with source: peer.id.
core/websocket-gateway.tsinbound publish → bus
case 'publish':
  // heartbeats also refresh links / build / rssi on the roster …
  this.#bus.publish({
    topic: f.topic, kind: f.kind, payload: f.payload,
    source: peer.id,          // ← the unifying key, stamped here
  });
  break;

From here a fact is on the bus, and any module subscribed to that topic reacts. Notably, the PerceptionProcessingHub subscribes to the client topic — so a dock's on-device VAD or battery fact enters the media plane's processors through this WS path.

Media plane · ingest

M1The SFU answers the dock's offer and taps its tracks

After the dock's SDP offer arrives (as a media control frame), Sfu.onProducerOffer builds an RTCPeerConnection keyed by streamId = dockId, answers, and subscribes to pc.onTrack. That handler is the crossroads:

media/sfu.tspc.onTrack.subscribe
if (track.kind === 'video') this.#startKeyframeRequests(...); // RTCP PLI / 2s

// hand the track to the processing tap — STT/vision/recording see
// the SAME inbound RTP the SFU forwards to browser viewers.
this.#tap?.onTrack(streamId, track.kind as MediaKind, track);

That one line is why the SFU runs server-side rather than peer-to-peer: routing media through a server means every producer's stream passes one observable point — the tap. In production, #tap is the PerceptionProcessingHub.

why PLI

The dock sends a keyframe only at connect; under packet loss the decoded P-frames smear. The SFU asks for a fresh keyframe every 2s so both viewers and the vision/face processors stay sharp.

Media plane · the router

M2What PerceptionProcessingHub does: route two feeds to a processor registry

It does not touch media samples — the processors do that. The hub is a router / multiplexer: it takes two unrelated feeds, stamps each with dock identity, and delivers exactly the right slice to each processor. It also implements MediaTap, which is just how the SFU hands it the first feed.

Takes inTransformsProduces
RTP tracks (SFU onTrack)resolves streamId → dockId; source-filters by each processor's mediaKinds / channels; manages per-stream lifecycle (start / teardown, gated by dockReady)RTP → each interested processor's onRtp

It hands each processor a StreamContext — the processor's whole outward surface: ctx.emit(result) (→ the perception topic) and ctx.publish(channel, …) (a fact → other processors — the graph edges in M3). The hub's emit takes the finished result, stamps routing fields onto it (makeResult({...r, streamId, dockId, ts})), and publishes ×2.

the hub creates no meaning

At this stage there is only audio, video, and facts — the hub has no idea what "presence" or "identity" is, and never builds a result. It runs models on nothing. The processor turns raw input into a PerceptionResult (next section); the hub only stamps identity onto it and forwards. It's a router, not an interpreter.

decoupled

It imports no other module — just a bus, a resolveDock closure (peer id → dock, live from the roster), and the dockReady gate. So media stays portable and the processor set stays swappable; they meet only through these closures.

Media plane · the processor graph

M3Not a flat fan-out — a graph, and every node has two outputs

Every processor implements one interface — StreamProcessor (perception/processor.ts): it declares mediaKinds / channels / sources, and gets onStreamStart(ctx), onRtp?, onChannelItem?, onStreamEnd. That uniformity is why the hub's registry can treat them all the same. Each is a factory returning StreamProcessor (a few return StreamProcessor & {…} to expose an extra method like captureNow).

The hub delivers each only what it declared, try/catch-wrapped so one can't kill the SFU. But they aren't independent — some read others' work — and each writes to two places: the live perception topic and the durable SnapshotStore ring.

The edges are the point. Video decode is a single shared stage (one ffmpeg per dock, owned by face-recognition via FrameGrabber) — face-recognition, vision, and identity all read its decoded JPEG rather than decoding again. That coupling is safe because everything is keyed to the same streamId. Two audio branches: parakeet-tdt is the live transcriber (every utterance); the Gemini bg-audio interpreter is a separate, optional, async upgrade fired only on a significant acoustic event — it patches the speech record with an addressed?/kind/salience/summary read (and can rewrite the transcript). And bodymotion→identity is the ego-motion guard: while the head pans, a face leaving frame is the robot turning, not the person leaving — so presence is frozen (no false "X left" / "Y arrived").

vision is change-gated, not per-frame

vision does not run the ~5 s VLM on every decoded frame. It probes ONE frame every ~1.5 s and embeds it with an in-process DINOv2 model (perception/embed.ts, onnxruntime, no sidecar). The expensive qwen call fires only when the frame's embedding has moved past a distance threshold from the last analyzed one — i.e. the scene actually changed — or on a sense-wake (the mic heard something salient) or a 5-min heartbeat. A static room costs cheap embeddings, not VLM inferences. Further gating built on the same embedder: a recent-view cache (reuse a description when the head sweeps back to a spot it just saw), self-motion suppression (defer while the head is panning), and window dedup (collapse near-identical frames in a window). So the scene results you see are change events, not a fixed cadence — the timeline in the Perception Studio labels each with why it ran (scene-change / sense-wake / heartbeat / reused) and accounts for the gated gaps. See vision-snapshot.ts.

Every processor emits the same envelope — PerceptionResult. One perception topic carries all of them, discriminated by kind; a new processor adds a kind, never a new shape or topic. The processor authors kind + payload (the meaning it just computed); the hub stamped the rest:

two sinks, two shapes

The live result (② → perception topic) is a PerceptionResult. The durable record (① → the ring) is a SnapshotRecord — a different shared shape: { dockId, source, interval:{from,to,durationMs}, payload:{ text, …extra } }, IST-stamped. One is for reacting now; the other is the memory arm's raw feed. A processor writes both.

🚪 The gates

The conditions that decide whether flow continues

Not everything perceived flows through — several gates drop, defer, or downgrade signal so the expensive stages (the VLM, an LLM turn, a self-thought) fire only when they should. Each is numbered and marked on the diagrams (🚪1–🚪3 in M3, 🚪4–🚪5 on the whole-path). What each one asks:

GateWhere (file)Passes only when… (else → )
🚪1 vision change-gate 👁 vision-snapshot vision-snapshot.ts the frame's DINOv2 embedding moved past a distance threshold (pixel Δ ≥ VISION_DIFF_THRESHOLD 0.03 as fallback) — the scene actually changed. Else → skip the VLM (cheap probe only). Sub-gates: self-motion defers while the head pans; window-dedup collapses near-identical frames. ⏳ 5-min REFRESH overrides (guards slow drift).
🚪2 sense-wake gate 👁 vision-snapshot vision-snapshot.ts another sense heard something salient since the last analysis → earns ONE vision look per audio episode (continuous audio ≠ continuous visual change). Else → no extra look.
🚪3 STT confidence gate 🎙 speech-watch speech-watch.ts the transcript isn't garbage (a repetition-loop, compression > 3.0, or no_speech > 0.6 AND logprob < -1.0). Garbage → rendered as [unclear speech, ~Ns], never as words the brain acts on; shaky → tagged [low-confidence].
🚪4 addressed gate 🧠 DockBrainSession brain/index.ts · conversation-state.ts the utterance was addressed — a tap, a wake phrase, or it landed inside an open conversation window. Else → skip:not-addressed (room chatter, no turn). Also skips: recording, garbage tier, <2 words, or queued while mid-turn. The decision is stamped on the record (🚪 → [→ TO YOU] vs [overheard]).
🚪5 attention gate 🧠 proactive perception/attention/gate.ts with no user speaking, cheap rules over a signals snapshot say raise a self-thought — and not within the cooldownMs window (don't nag) and not a dup of the last raise (key). Else → stay quiet.
gates vs triggers

A trigger (⏳/⚡ on the M3 nodes) is what starts a processor looking; a gate (🚪) is the condition that decides whether the result flows on. Vision is the clearest: it's triggered to probe every ⏳1.5s, but the expensive VLM only fires when the 🚪1 change-gate passes.

The memory arm

M4The ring is durable and self-compressing — one summarizer, two outputs

Raw snapshots are noisy (a small VLM misreads a mug as a "ruler" frame to frame, or straps on a hook as "a person on a pull-up bar") and internally contradictory (identity flickers "no one" while speech is heard). Nothing reasons over raw mush directly. The ring is now durable on disk (per-dock JSONL, retention.ts); the summarizer (Gemini) is the single quality-control point that fuses it into a coherent, reconciled picture — sent the window's keyframe images too, so it can see the scene and override a bad VLM line rather than propagate it. There is one summarizer, one checkpoint, producing two outputs (the background curator job — a second pipeline with its own checkpoint — was removed 2026-07-10):

This memory arm is the REMEMBER band of the single end-to-end diagram at the top of the page (each writer-job → its own store, drawn in context). The ASCII above is the edit-first source; the table below has the cadences + files.

  • Grounding — pulled synchronously when a turn is built: the last summary (stamped with staleness) + the salient tail. No network on the turn's critical path. This is what the model always sees.
  • Span-summary (①) — at trim, each aged-out clock-hour is compressed into a durable digest before the raw is dropped, so the timeline compresses but never has a hole. This is perception compressing its own tail, not a separate layer.
  • Facts (②) — the same trim pass extracts durable facts ("Guru prefers tea") into the memory store (append + a light semantic dedup), which is a generic fact primitive the LLM also writes via remember and reads via recall_memory.
separate jobs — each PULLS from the stream, WRITES to its own store

summarize()/stitch() are tools (pure functions — they never write). The writing is done by independent scheduled jobs, each on its own timer, each reading the perception stream and writing to a different store. why the summarizer is the one gate

Both durable outputs feed from a reconciled read, not the raw ring — deliberate. Consolidating raw per-utterance speech once produced 81% junk "speaker 0" beliefs. Digest first, then compress + remember. Crucially, the ego reads reconciled too (reconciledPerceptionSince) — never raw sensor lines, because a faithful reasoner reads raw contradictions as "my eyes are broken" and spirals. Note the trim job re-summarizes the raw span (not the rolling pulse), so facts derive from the full window, not a lossy 60 s snapshot.

M4bInside the summarizer — stitch (deterministic) then fuse (LLM)

"The summarizer" (summarizer.ts) is not one Gemini call — it's two stages, and most of the noise-handling is deterministic code, not the model. stitch() turns the four independent, time-ordered streams into one clean, annotated transcript; summarize() then sends that (plus keyframe images) to Gemini for the fusion. The model reasons over an already-cleaned timeline, not raw jitter.

where the QC actually lives

Most of "make sense of noisy streams" is the deterministic stitch() — spans, staleness, confidence tiers, addressed/overheard. The LLM only does the fusion on top of a transcript that's already been cleaned. And because stitch() is reused by grounding and the ego, that cleaning is consistent for every consumer, not re-implemented per reader. Graceful degradation: no API key ⇒ summarize() still returns the stitched transcript (with an error field); empty window ⇒ a canned "no perception data".

Where the planes cross

×Media understanding returns via the WS gateway

A processor calls ctx.emit(result); the hub stamps it and publishes to the perception topic twice — directed to the dock, plus an undirected copy for the console. This is the exact hand-off from the media plane back to the control plane:

perception/perception-processing-hub.ts#publishResult
// directed to the dock — its brain re-grounds on the fact
this.#bus.publish({ topic:'perception', kind:r.kind, payload:r, source:'station', to:r.dockId });
// undirected copy — the console + world-state aggregator
this.#bus.publish({ topic:'perception', kind:r.kind, payload:r, source:'station' });

That perception-topic message is now caught by the WebSocketGateway, whose one bus.on('*') subscription turns every bus message into an event frame for peers subscribed to that topic — the reverse of the inbound publish path:

core/websocket-gateway.tsbus '*' → outbound event frames
bus.on('*', (msg) => {
  const frame: EventFrame = { t:'event', topic:msg.topic, kind:msg.kind, payload:msg.payload, ts:msg.ts };
  for (const peer of this.#peers.values()) {
    if (!peer.topics.has(msg.topic)) continue;        // not subscribed → skip
    if (isDirected(msg)) { /* only the addressed dock: to / toAddr */ }
    peer.ws.send(json);
  }
});
full circle

Media entered as RTP on the media plane; understanding leaves as a JSON event frame on the control plane. The directed copy reaches only the dock it's about (its brain re-grounds the next turn); the undirected copy feeds the Perception Studio console. The Bus is the only place the two planes touch.

addressed vs overheard, persisted

The brain owns the authoritative addressed decision (tap / wake / conversation-window latch) — and now stamps it onto the speech snapshot (markSpeechAddressed). So the summarizer renders [→ TO YOU] vs [overheard — not to you], fact-extraction won't mine relationship facts from room chatter, and the ego stops reading a workout video's instructions as words spoken to it. Previously this decision was used and discarded — every downstream consumer treated "said to the dock" and ambient speech identically.

Once a turn runs, the session streams its output back out as directed frames (toAddr: dock/phone) — and this is where the pipeline ends, in more than one place:

the ends (plural)

There is no single "end." A moment fans into several sinks: speech + face UI (agent→phone), body motion (bodylink→ESP32), the durable memory the summarizer wrote at trim — span-summaries + facts (for future turns), and the console (the undirected perception copy). The pipeline's true output is the robot changing state in the world — not a published frame.

the audio enricher (merged 2026-07-12)

The durable audio path is the audio enricher (background-audio.ts · enrichAudio, Gemini flash), which replaced the old two bg-audio calls (per-utterance speech-details patch + per-impulse sound record). It is not one of the six registered stream-processors — it's a step on the audio path — so it's absent from the processor table but drawn as the 👂 node in the single diagram at the top. ONE debounced batch call per window (see the trigger on that node) does the accurate, in-context transcription + diarization + acoustic read, and lands ONE record kind: 'enriched' per segment — WHAT it contains (real in-room speech / played media / a non-speech sound) is the audioSource field, not a separate kind. Parakeet stays LIVE-ONLY (conversation/wake/addressed-latch); the enricher owns durable memory, and dropSupersededSpeech drops the overlapping parakeet record so nothing double-counts. Model is flash, not flash-lite — flash-lite hallucinated confident fake conversations from unintelligible audio; flash is honest ("[unintelligible]"/"[music]"). Full design + the flash decision: decision-traces/audio-enricher.md.
