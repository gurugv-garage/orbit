# Face harness — see what the dock is actually showing

> **Why this exists.** This dock has **no adb** (deploys are OTA), and the phone
> reported no UI state up. The dock's face was therefore **unobservable from
> off-device** — and two bugs shipped straight past a green test suite because of
> it:
>
> 1. a wake regression (the face sat asleep while someone talked to it), and
> 2. a "sweat bead" that still rendered as a **tear** — the exact bug it was
>    written to fix. Its geometry was reasoned about, never looked at.
>
> A drawing bug is invisible to every unit test in this repo. **The only real
> check is a picture.** This harness is the eye.

## The three hooks

All ride the **existing `face` cap** over the station↔phone WebSocket — no adb,
no new transport, works on the real OTA release build.

| Hook | What | Where |
|---|---|---|
| `face_probe` | what the face IS right now (read-only) | `DockTools.faceProbe()` |
| `face_force` | set a mood with **no LLM** | `RemoteBrain` → `tools.setFace` |
| `face_shot` | **JPEG of the screen** + state, same instant | `debug/ScreenCapture.kt` |

## REST surface

```bash
D=dock-redmi; S=http://localhost:8099/api/brain

# What is the face showing RIGHT NOW?
curl -s $S/$D/debug/face | jq

# Force a mood (no LLM, no turn) — then probe it
curl -s -X POST $S/$D/debug/face -H 'content-type: application/json' \
     -d '{"expression":"concerned"}'

# FILMSTRIP: 5 shots, 1s apart → var/debug/face/<runId>/{00..04}.jpg + film.json
curl -s -X POST $S/$D/debug/face/film -H 'content-type: application/json' \
     -d '{"count":5,"everyMs":1000,"label":"concerned"}' | jq
```

`GET /debug/face` returns the probe parsed into fields:

```json
{ "ok": true,
  "face": { "expression": "concerned", "state": "speaking", "speaker": "bot",
            "style": "aurora", "privacy": "false", "gaze": "0.10,-0.05",
            "userEmotion": "angry", "facePresent": "true" } }
```

### `expression` vs `userEmotion` — read this before using the output

These are **two different faces** and the whole system has a habit of confusing
them (see [face-behavior-spec.md](../poc-plans/face-behavior-spec.md) §Fix 4):

| field | whose face | set by |
|---|---|---|
| `expression` | **the ROBOT's** — pixels on the dock's screen | LLM, mirroring, timers, dev panel |
| `userEmotion` | **the HUMAN's** — what the camera reads on *you* | on-device FER |

The probe deliberately names them apart. `UserEmotion.Kind.Angry` and
`FaceExpression.Angry` are *identical member names in different enums*, mapped by
an identity function — which is why the dock can wear your anger and then
sincerely deny being angry. Never report these under one word.

## Why a filmstrip and not one screenshot

Every bug in this area is about **time**, and a single frame cannot see a
transition:

- a mood that won't decay (pinned `curious` for an hour)
- a face that won't wake (asleep while you talk)
- a transition that flickers or lands late

```bash
# Does a forced mood actually decay? Force, then film 30s.
curl -s -X POST $S/$D/debug/face -d '{"expression":"curious"}' -H 'content-type: application/json'
curl -s -X POST $S/$D/debug/face/film -d '{"count":6,"everyMs":5000,"label":"decay"}' -H 'content-type: application/json'
```

`film.json` records **`tMs` per sample** — real elapsed time from the first
shot. Samples are **sequential** (each waits for the previous result), so
`everyMs` is a *floor*: a slow RTT stretches it. **Trust `tMs`, never the nominal
interval.**

## Output — `var/debug/face/<runId>/`

```
var/debug/face/2026-07-15T09-12-33-000Z-concerned/
  00.jpg 01.jpg 02.jpg 03.jpg 04.jpg
  film.json     # per-sample: i, tMs, file, expression, state, userEmotion, …
```

`var/` is **gitignored and disposable** — delete the whole tree any time; the
harness recreates it. `label` in the request becomes a `runId` suffix, so runs
stay tellable apart.

## Gotcha: test hooks are exempt from the turn-staleness guard

`RemoteBrain.onToolCall` drops any tool-call whose `turnId` isn't the **active
turn** — correct for real tools (a late `set_face` from a cancelled turn must not
fire). But a probe is **turn-less**: it observes the dock while idle, between
turns, or mid-turn, and invents its own `turnId`.

The first live probe returned `"(no response from dock)"` for exactly this
reason — silently dropped by a guard doing its job. The three hooks are now
exempt by name. **If you add a fourth hook, add it to that set** or it will look
like the phone is broken.

## Deploying: use the script

```bash
# bump versionCode in node-dock/app/app/build.gradle.kts FIRST (it's the OTA gate)
orbit-station/scripts/ota-app-deploy.sh 47 dock-redmi
```

It handles all three silent-failure modes below (stage race, zero peers, stale
versionCode) and only exits 0 once the dock is **actually running** the build.
Doing it by hand cost me three wasted deploys in one day — every one looked
exactly like "the code is broken".

## Gotcha: the OTA build/stage race (hit 3×)

`POST /api/ota/app/build` returns when the **build state** leaves `building` —
but the artifact + `meta.json` are staged a moment LATER. Reading
`/api/ota/app/latest` right after the state flips returns the **previous** build,
which reads exactly like a failed build. It isn't.

Don't paper over it with `sleep N` (I tried 3s, then 6s — both raced). **Poll the
condition:**

```bash
for i in $(seq 1 12); do
  OFFER=$(curl -s $S/api/ota/app/latest | jq -r .build)
  [ "$OFFER" = "$WANT" ] && break; sleep 5
done
[ "$OFFER" = "$WANT" ] || exit 1     # only NOW is it a real failure
curl -s -X POST $S/api/ota/app/announce
```

Announcing before the artifact is staged silently offers the OLD build (seen:
`{"announced":0,"build":35}` while 36 was still staging — zero peers, no error).
**Always verify the offered build equals what you meant to ship, before
announcing.**

**And check `announced` in the reply.** `{"announced":0}` means it reached **zero
peers** — the dock was offline, or (the common case) it was mid-install of the
PREVIOUS build and had dropped its socket. The station reports this as success:
no error, nothing installs, and the next probe reads the old build and looks like
a code bug. Announcing back-to-back builds is the way to hit it. Wait for the
dock to be online at build N *before* announcing N+1:

```bash
[ "$(curl -s $S/api/ota/app/announce -X POST | jq -r .announced)" != "0" ] || echo "NOBODY HEARD IT"
```

## Driving a camera EMOTION (react-don't-mirror)

```bash
# 4 reads, 700ms apart — enough to clear the gate's ~2s hold
curl -s -X POST $S/$D/debug/face/emotion -H 'content-type: application/json' \
     -d '{"kind":"Angry","confidence":0.9}'
curl -s $S/$D/debug/face | jq -r '.face | .expression, .moodSource, .moodWhy'
# → concerned / react / "you look upset, so I'm concerned — I'm not angry myself"
```

Publishes onto the **same `PerceptionBus`** every real FER read flows through, so
it exercises the true path — confidence floor, hold time, and all — not a
shortcut past them. `times` defaults to 4 because **a single read correctly does
nothing**: the gate is a debounce, and that debounce IS the anti-flicker fix. The
camera streams ~1 Hz; so does this.

## Gotcha: "present" means two different things

Diagnosing "I reacted and nothing happened", the station said `present: true`
while the phone's probe said `facePresent=false`. Both were right:

| signal | means |
|---|---|
| `/api/perception/:dock` → `present` | the phone is **streaming video** |
| probe → `facePresent` | **ML Kit sees a face in the frame** |

A dock streaming a view of the ceiling is `present: true` with nobody in it. The
actual cause that time: **the camera was aimed at the ceiling** — no face, so no
FER read, so nothing for the gate to react to. `userEmotion` sat frozen at a
stale value and the `sleepy` face was *correct* (nobody in view for 90s).

**Grab the frame before blaming the code:**

```bash
curl -s "$S/../perception/frame/<peerId>" -o /tmp/cam.jpg   # then LOOK at it
```

`facePresent=false` + a stale `userEmotion` = the pipeline never ran. That is a
camera-aim or lighting problem, not a face-logic one.

## Limits — know these before trusting a result

- **Screenshots are lossy by design**: downscaled to 480px, JPEG q70, so a sample
  is <~40KB and rides a JSON WS frame. Fine for "is that a tear or a bead"; **not**
  for subtle colour/AA work.
- **Only 2 of 7 faces draw accents** (`Aurora`, `Puppy`). `Vader`/`Robot`/`Ghost`/
  `Owl`/`Dragon` never call `drawAccent` — filming `concerned` on those shows no
  droplet at all, and that is *correct*, not a failure.
- **`face_force` bypasses the LLM** — it proves the RENDERER, not that the brain
  would ever choose that mood. (It wouldn't choose most: 805 real inline tags are
  only `neutral`/`curious`/`happy`/`concerned`; `sad` and `angry` are **never**
  sent.)
- **The probe is read-only** and safe in release. `face_force`/`face_shot` mutate/
  capture — deliberately kept in `main` (not `src/debug`), because the OTA build
  is a **release** build and a debug-only hook could never run on the real dock,
  which is the only place the face renders.

## Removing this later

Three call sites: the `face_probe`/`face_force`/`face_shot` branches in
`RemoteBrain.onToolCall`, `DockTools.faceProbe()`, `debug/ScreenCapture.kt` +ked
its `MainActivity` binding, and the `/debug/face*` routes in `modules/brain/index.ts`.
`RpcResult.imageBase64` is additive and harmless.

**Recommendation: keep it.** The cost is a few dozen lines and one `@Volatile`
activity reference; the benefit is that the dock's face stops being a thing we
can only reason about. Both bugs above would have been caught in one command.
