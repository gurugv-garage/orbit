## 1. Goal and the core finding

**Goal:** orbit-station (the brain + control plane) and everything it needs runs on
a remote, always-on machine. The laptop can be closed; the dock keeps working 24/7.

**Core finding — it's deployable, and the latency is fine *if you pick the right
region*.** The hard blockers are not latency; they are:

1. **The perception sidecars are Apple-MLX-only** — they cannot run on a generic
   Linux VM as written (the models are portable; the MLX *engine* is not).
2. **WebRTC + security assume a LAN** — no STUN/TURN, no auth, plain `ws://`.
3. **Vision runs ungated 24/7** — the dominant cost driver, and the thing that
   decides GPU-vs-cloud economics.

Latency only "breaks" the conversation if you put the station on a **different
continent** from the dock. **An in-India VM for an India dock is essentially as
good as the laptop.** Region choice is the whole latency lever — and it points at
the same in-country VM the cost analysis favors anyway.

## 3. Latency — laptop (LAN) → remote VM

The honest question: when the station leaves the laptop (~1ms LAN) for a VM
(tens-to-hundreds of ms), **which links lengthen?** It depends on where each
endpoint sits. In the real "no laptop" setup the **dock (phone + ESP32) stays in
your house** and the **station + its sidecars move to the VM together**.

### Link-by-link (dock at home, station on VM)

| Link | LAN | On VM | Critical path? | Effect of the move |
|---|---|---|---|---|
| Mic audio → station (WebRTC, continuous) | ~1ms | one-way WAN | feeds STT | **One-way**, jitter-buffered. Adds geo one-way delay before STT starts |
| STT (sidecar co-located with station) | local | local *to VM* | yes | **No new RTT** — moves with the station |
| `speak` reply → phone (stream) | ~1ms | one-way WAN | **yes (human-facing)** | **One-way**. Adds geo one-way delay to time-to-first-word |
| `set_face` tool RPC (brain→phone→brain) | ~2ms | **full WAN RTT** | **yes — brain awaits it** | **The one true RTT amplifier.** +1 WAN RTT per call |
| `move` body command | ~1ms | one-way, fire-and-forget | no | In-proc→firmware, not awaited. No felt change |
| Body digest → phone (~1Hz) | ~1ms | one-way | no | Display-only, staleness-tolerant. No felt change |
| Vision frames → station | ~1ms | one-way WAN + **uplink bw** | no (background) | Not on speak-path; see §6 bandwidth |

### Human-facing path

`[you stop speaking] → mic one-way to VM → STT (unchanged) → brain+LLM (unchanged,
always was cloud) → first speak frame one-way back`. Added latency ≈ **one
geographic round-trip**, not several:

| Station location (dock in India) | Added one-way | Felt delay | Verdict |
|---|---|---|---|
| **In-India VM** (Mumbai/Delhi/Noida) | +10–30ms | +20–60ms | **Unnoticeable** ✅ |
| Same continent (e.g. Singapore) | +40–80ms | +80–160ms | Slightly perceptible |
| Cross-continent (US/EU VM) | +120–200ms | +250–400ms | **Noticeably laggy** ❌ |

**Decision implied: host in India.** It removes the latency question and is also the
cheap, accessible option (§4). The "different geo" case from the original question
is the only one that hurts — and it's the one to avoid.

### The two RTT caveats

- **`set_face` is the only awaited phone round-trip.** Same-region: negligible.
  Cross-geo: it stacks (a turn can make several) and gets ugly. *Cheap fix if ever
  needed: make `set_face` fire-and-forget instead of awaited — a small brain change.*
- Everything else interactive is **one-way or in-process at the station**, so it
  feels the same as on the laptop.

## 4. Cost model — India options (latency-weighted)

One dock, **24/7**. Two strategies: **self-hosted GPU (flat)** vs **hybrid cloud
(cheap CPU VM + APIs)**. Latency-weighted = prefer **in-India** so RTT stays low.

### 4a. The hosting box

**The recommended path needs NO GPU.** A GPU is only required if you choose to
**self-host the vision model** instead of calling a cloud vision API. The
recommendation (hybrid, §4e) sends vision to a cloud API (§4c–4d), so it runs on a
**plain CPU VM**. The GPU row exists purely so you can compare the self-host
alternative — it is not the default.

**Recommended box (Indian providers, INR billing, India DCs, ~10–40ms):**

| Box | Provider | ~Monthly | Notes |
|---|---|---|---|
| **Plain CPU VM** (4–8 vCPU) ✅ | Any Indian VPS / **E2E** / hyperscaler | **~₹1,500–4,000** | The hybrid path. **No GPU.** Runs station + faster-whisper STT + face |

**Only if you self-host vision (the GPU alternative — not recommended):**

| Box | Provider | ~Monthly | Notes |
|---|---|---|---|
| L4 24GB GPU | **Jarvislabs.ai** (Noida) | ~₹30,000 | Cheapest verified 24GB; per-minute; no-KYC. Charges USD via Stripe (no UPI) |
| A30 24GB GPU | Jarvislabs.ai | ~₹28,400 | Slightly cheaper |
| L4 24GB GPU | **E2E Networks** (Delhi/Mumbai/Chennai) | ~₹36,500 | True native UPI/INR + GST-clean. ⚠️ price hike 2026-07-01 — lock a committed plan before |

Why you'd *not* take the GPU: a cloud vision API at gated volume is ~₹0.4–1.5k/mo
(§4c), so the flat ~₹30k GPU only wins if your "interesting hours" are so high that
gated-cloud approaches it — unlikely for a home dock. The GPU also keeps the
**MLX→Linux vision port** on your plate (§6); the cloud-API path deletes that work.

### 4b. STT cost (cheap either way)

STT is **VAD-gated already** — only ~2–3 min of audio/hr actually hits transcribe,
not 60. So:

- **Self-host on the CPU VM** (faster-whisper-small): **₹0 marginal**, and keeps
  audio off the wire. *Preferred* — also the only sidecar left to port.
- Cloud STT (AssemblyAI async / gpt-4o-mini-transcribe): **~₹375–450/mo/dock**.
  A rounding error.

### 4c. Vision cost — the real driver, and why the gate matters

Today vision is **ungated**: a back-to-back loop, ~16,000 VLM calls/day even in an
empty room (`modules/perception/processors/vision-snapshot.ts`). A cheap **activity
gate** (§5) cuts that 90–95%. Cost by regime, **Gemini 2.5 Flash-Lite** (the
recommended vision API — see §4d):

| Regime | Calls/day | Calls/mo | Vision API/mo/dock |
|---|---|---|---|
| **Gated low** | ~500 | ~15k | **~₹400** |
| **Gated moderate** | ~2,000 | ~60k | ~₹1,500 |
| Ungated (today's behavior) | ~16,000 | ~480k | **~₹12,350** ❌ |

The gate is a **32× lever** — bigger than any provider or region choice.

### 4d. Which vision API

With 5 images/call, ~90% of each call's cost is **image tokens**, so the only number
that matters is how a provider tokenizes a 512px image:

| Vision API | per 1,000 calls | Note |
|---|---|---|
| **Gemini 2.5 Flash-Lite** ✅ | ~₹26 | first-party, **stable supply**, ~516 tok/image; key already wired |
| Qwen2.5-VL 3B (OpenRouter) | ~₹16 | cheaper but **serverless supply being retired**; less stable |
| GPT-4o-mini ⚠️ | ~₹185 | **avoid** — bills each image at ~2,833 tokens (~33×) |

Free extra lever: **resize frames to ≤384px** → Gemini drops to a flat 258
tok/image, ~halving vision cost again. A "room gist" doesn't need more.

### 4e. Totals (1 dock, 24/7)

| Setup | Box | Vision | STT | **Total/mo** | MLX port? | Latency (in-India) |
|---|---|---|---|---|---|---|
| **Hybrid, gated** ✅ | CPU VM ~₹3k | gated Gemini ~₹0.4–1.5k | local ₹0 | **~₹3,500–4,500** | STT only | unnoticeable |
| Self-host GPU, gated | L4 ~₹30k | flat (on GPU) | local ₹0 | **~₹30,000** | vision + STT | unnoticeable |
| Cloud, **ungated** | CPU VM ~₹3k | ~₹12k | ~₹0.4k | **~₹15,000** ❌ | unnoticeable | — |

**Recommendation: hybrid + gated, in-India CPU VM. ~₹3.5–4.5k/mo per dock, no GPU,
no Apple Silicon, lowest latency.** Self-hosted GPU only wins if "interesting hours"
are so high that gated-cloud approaches the flat ₹30k — unlikely for a home dock.

## 5. The vision gate (the enabling build)

Today: `while (stream live) { analyze() }` — no filter. Replace the loop body with a
tiered, cheapest-first pre-filter so the VLM only fires on frames worth analyzing.

| Tier | Check | Cost | Decision |
|---|---|---|---|
| 0 | **Motion / frame-diff** (mean pixel delta vs last frame) | CPU, sub-ms | no motion → skip (kills most empty-room ticks) |
| 1 | **Person present** — read the **face-api** detection already computed | ~free | motion but no person → skip/downgrade |
| 2 | **Novelty** — scene meaningfully different from last described? (downscaled hash/embed) | cheap CPU | "already described 4s ago" → skip |
| 3 | **VLM call** (Gemini Flash-Lite) | expensive | only frames passing 0–2 |

- Tier 0 alone ≈ 80% of savings, ~20 lines of image math.
- Tier 1 is nearly free — **face-api already runs on these frames**
  (`modules/perception/face/recognizer.ts`); read a signal you already pay for.
- Contained change in `modules/perception/processors/vision-snapshot.ts`; snapshot
  store, brain grounding, and per-turn on-demand vision all unchanged.
- Anticipated as the deferred **tiered-escalation pyramid** (§9 of
  [../perception-pipeline.md](../perception-pipeline.md)) — building with the grain.
- Latency: tiers 0–1 run locally <1ms; when vision fires, a cloud call (~300ms–1s)
  is **faster** than today's local 3B-on-laptop (~3–6s). Not on the speak-path.

## 6. The infrastructure work the move requires

These — not RTT — are what actually decide whether the move works.

1. **Port the speech sidecar off MLX.** `parakeet-mlx`/`mlx-whisper` →
   **faster-whisper** (CUDA *or* CPU; preserves the `avg_logprob`/`no_speech_prob`/
   `compression_ratio` hallucination knobs). Keep the same HTTP contract
   (`/transcribe`, `/health`) so the station needs **zero** changes. *Vision sidecar
   need not be ported — it becomes a cloud API call.*
2. **STUN/TURN for WebRTC.** The SFU defaults to LAN host candidates (`sfu.ts`).
   Set `STUN_URL`; for symmetric/CGNAT home NATs add a **TURN server** (coturn) —
   not yet wired, a small add.
3. **Security — the hub becomes internet-facing.** Add `wss://` (TLS; `npm run
   certs` exists, want a real cert) and **an auth token on the WS handshake**
   (does not exist today). **Non-optional before exposing the brain/body plane.**
4. **Uplink bandwidth (the sleeper).** The dock continuously uploads camera+mic
   over WebRTC. Audio (~40kbps Opus) is fine anywhere; **continuous video upload**
   is real on a home uplink and jitter-prone — degrades *perception* before RTT
   bothers *conversation*. Size/measure home upload; the gate (§5) does **not**
   reduce upload (frames still stream up), only VLM calls.
5. **Env-var rewiring** (all already supported):
   `STATION_WS`, `PERCEPTION_SIDECAR_URL`/`TEMPORAL_SIDECAR_URL` (if any sidecar
   off-box), `STUN_URL`, OTA artifact host reachable by the dock.
6. **OTA artifact host.** Served from the station's LAN IP today
   (`modules/ota/index.ts`); the dock must be able to reach the VM's address for
   firmware/app updates.

## 7. Migration steps (suggested order)

1. **Port the speech sidecar to faster-whisper**, verify same HTTP contract against
   the running station locally (no behavior change yet).
2. **Build the vision gate** (Tier 0+1), tune motion threshold on real footage,
   confirm VLM-call volume drops as expected.
3. **Switch vision to the cloud API** (Gemini Flash-Lite, frames ≤384px) behind the
   `TEMPORAL_*` seam; confirm grounding still reads well.
4. **Stand up the in-India CPU VM**; deploy station + faster-whisper + face there.
5. **Add wss + WS auth token**, then **STUN (+TURN if needed)**; point the dock's
   `STATION_URL` at the VM, verify WebRTC connects through NAT.
6. **Verify the full chain** end-to-end (per [perception-runbook.md](perception-runbook.md)):
   conversation turn, STT, vision gate firing, body motion, OTA reach.
7. **Measure** real "interesting hours" and uplink; revisit gated-low vs -moderate
   cost and decide whether GPU self-host is ever warranted.
