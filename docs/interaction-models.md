## Why this doc

Right now the dock mostly does **request → response**: you address it, it thinks,
it replies. We added one async shape — **background tasks** that run out-of-turn and
push updates back. But there are interaction shapes neither covers well (e.g. "follow
me while I move"), and there are open questions about **async tools, parallel tools,
and how a tool differs from a task**. This doc lays out the space so we can design
deliberately instead of bolting each new behavior onto request→reply.

The organizing question for any new behavior: **what is its lifecycle relative to a
turn?** Does it complete within the turn (sync tool), outlive the turn and report back
(task), or run *continuously alongside* conversation until a condition ends it
(a behavior/mode — the new shape)?

## The models (current + proposed)

> **M1–M6 are a local shorthand for this doc only** — just row numbers for cross-
> referencing the shapes below, not established terminology. (The more principled
> lens is the authoring axis further down.)

| # | Model | Lifecycle | Initiative | Built? |
|---|---|---|---|---|
| **M1** | **Request → response** | one turn: you ask, it replies | user | ✅ |
| **M2** | **In-turn tool call** | within a turn; tool runs, result feeds the same LLM loop | model | ✅ (sync / fire-and-forget) |
| **M3** | **Background task** | outlives the turn; runs as a separate process; pushes updates back as autonomous turns | model (on user ask) | ✅ ([tasks.md](tasks.md)) |
| **M4** | **Proactive / unprompted** | the dock starts a turn itself (a task signal, a perception gate) | dock | ✅ partial (task notify; gate is scoped) |
| **M5** | **Continuous behavior / mode** | runs *while* you can still converse; sensor-driven; ends on a condition or command ("follow me", "watch the door") | user starts, dock sustains | ❌ **the gap** |
| **M6** | **Async / parallel tools** | a tool that doesn't block the turn, or several tools at once | model | ❌ (see §Open questions) |

M1–M4 are variations on "a turn happens." **M5 is the genuinely different shape** —
a long-lived behavior concurrent with conversation. M6 is a plumbing question that
mostly serves M5/M3.

## The deepest cut: WHO authored the behavior, and WHEN

The sharpest distinction between a **tool** and a **task** isn't duration or where it
runs — it's **when the behavior's code was written**:

- A **tool** = **fixed, pre-coded behavior.** Someone authored `move()`,
  `send_to_slack()` at build time. The LLM only **selects** it and fills in args. The
  logic is frozen; the model can't change what the tool *does*.
- A **task** = **on-the-fly scripted behavior.** The LLM **writes the script** that
  then runs. The behavior is **dynamically generated at runtime** — code that didn't
  exist until the model composed it for this situation.

> **Tool = fixed code the model *calls*. Task = new code the model *writes*.**

Everything else about a task — it's a separate process, it's async, it reports back —
falls out of that one fact: *generated code needs somewhere to run, and runtime-
authored behavior is open-ended, so it lives outside the synchronous turn.*

This reframes the axes below: "duration / locality / async" are **consequences**, not
the root cause. The root axis is **authored-at-build-time vs generated-at-runtime**,
i.e. **how much novelty the behavior needs.** A spectrum:

```
  fixed ◀───────────────────────────────────────────────▶ generated
  TOOL              PARAMETERIZED TOOL          SKILL            TASK
  move()            move(path, speed)      a saved recipe   LLM writes a script
  (frozen logic)    (logic + knobs)       (composed once,   (novel each time)
                                            reused)
```

**Where does "follow me" (UC3/M5) sit?** Interesting — it's **fixed code, not
generated.** "Follow me" is a *controller* someone writes once (perception→body PID-
ish loop); the LLM just *starts* it. So M5 is **closer to a tool than a task** on the
authoring axis — a **long-lived, stateful tool that owns an actuator**, not a generated
script. That's a cleaner way to see it than "a third thing": **M5 = a tool whose
execution outlives the call and runs a control loop** (a "stateful tool" / "mode"),
whereas a task = generated code.

→ Re-framed taxonomy on the authoring axis:
  - **Tools** (fixed): instant (M2), or **long-lived/stateful** (M5 — "modes").
  - **Tasks** (generated): the LLM-scripted, out-of-process, open-ended path (M3).
  - The "in-process task" idea from the boundary cases (UC4/9) then splits cleanly:
    if the behavior is **fixed**, it's a stateful tool; if it's **generated**, it's a
    task — and a task *could* gain an in-process mode, but that's an optimization, not
    a new category.

### Two lenses, kept on purpose
We keep **both** views; they answer different questions and don't compete:
- **The M1–M6 table (temporal/control shape)** — answers *"how does this behave over
  time, and what plumbing does it need?"* (duration, body ownership, concurrency).
- **The authoring axis (fixed → generated)** — answers *"who wrote the behavior, and
  when?"* (a tool is selected; a task is composed). This is the cleaner line for the
  **tool-vs-task** question specifically.
A given behavior has a coordinate in *both*: e.g. "follow me" is **M5** on the shape
lens and **fixed/stateful-tool** on the authoring lens. Use whichever lens the
question calls for.

### OPEN: how the LLM relates to a running mode (UNDECIDED — exploring)
Once a stateful tool/mode is running, what's the LLM's role? Not yet decided; the
tradeoff:
- **LLM out of the per-tick loop** (it `start`s the mode; a fixed perception→body
  control loop runs on its own; the LLM only `stop`/reconfigures + narrates). Cheap,
  fast, conversation stays free — but the moment-to-moment logic is frozen (can't
  adapt mid-run without a reconfigure).
- **LLM in the loop each tick** (every adjustment routes through the model). Maximally
  adaptive, but slow, costly, and it ties up the turn lane — wrong for a real-time
  control loop.
- **Hybrid** (likely): a cheap loop runs the ticks; the LLM gets *occasional* digests
  (like a task's `notify`) and can nudge/stop/reconfigure, but isn't on the critical
  path. Mirrors "LLM sets policy, the loop runs" — but the exact cadence + handoff is
  the thing to design. **Leave open until we design the mode primitive.**

## Start from the use cases

### UC1 — "What's 2+2?" / "What's on my calendar?"
Plain **request→response** (M1), maybe with an **in-turn tool** (M2) for the calendar
read. The tool is synchronous: the LLM waits for the result, then speaks. ✅ works.

### UC2 — "Render this video / research X and tell me when it's done"
**Background task** (M3). Kicks off a separate process, the turn ends ("I'll let you
know"), and later a `notify`/`finish` lands an autonomous turn that speaks the update.
Intermittent updates already exist via `notifyAgent()`. ✅ works.

### UC3 — "Follow me" (the motivating new case)
You start walking; the dock should **keep you in frame / move with you** *until you
stop or say "stop"* — while you can **still talk to it** ("turn left here"). This is
**M5**: a continuous, perception-driven behavior, concurrent with conversation.
- NOT request→response: there's no single reply; it's an ongoing control loop.
- NOT quite a task: a task is a separate OS process that reports back; "follow me"
  needs **tight, low-latency access to live perception + the body**, in-process, and
  it coexists with normal turns rather than just pushing updates.
- Needs: a **behavior** that (a) subscribes to a perception stream (your position),
  (b) drives the body each tick, (c) is interruptible/cancellable by voice or
  condition, (d) leaves the conversation free the whole time.

### UC4 — "Watch the door and tell me if someone comes"
Also **M5**, but the loop is mostly *idle-watching* + a trigger → then an M4 proactive
turn ("someone's at the door"). Long-lived, condition-armed, low duty cycle. Could be
a task (M3) if latency is loose, or a behavior (M5) if it should react fast and maybe
move (pan to look).

### UC5 — "Take a photo, post it to Slack, and dim the lights" (in one breath)
Several actions from one turn. Today they run **sequentially as fire-and-forget tools**.
Question: should independent actions run **in parallel** (M6)? For fire-and-forget
actuation it barely matters (each acks instantly); it matters when a tool actually
*waits* for a result (a slow API), where serial latency stacks up.

### UC6 — "Read me the news while I cook" (a long monologue you can interject into)
The dock talks for minutes; you can cut in ("skip that", "go back"). It's M1 output
but **long-running + interruptible mid-stream** — closer to a behavior (a playback
loop) than a single reply. Tap-to-interrupt (built) is the seed of this.

### UC7 — "Remind me to stretch every 30 min" (a recurring trigger)
A **scheduled/repeating** trigger that fires an M4 proactive turn. Not continuous —
it sleeps, wakes, speaks, sleeps. A cron-shaped task, not a control loop.

### UC8 — "Dance" / "do your happy wiggle" (a self-contained motor routine)
A fixed **choreography** — runs to completion, no perception feedback, not
interruptible-by-design (or trivially so). Today this is a fire-and-forget `move`/
gesture (M2). A *bounded* behavior with no sensing — the degenerate end of M5.

### UC9 — "Keep my plant in frame for a timelapse" (passive tracking, no body urgency)
Continuous perception (M5-ish) but **slow** — adjust occasionally, latency-loose. This
could live as an in-process task (M3 variant) just as well as a behavior. A boundary
case that tests whether M5 and "in-process task" are really distinct.

## The distinguishing axes (how to classify ANY behavior)

Rather than argue each case, classify on the axes that actually drive the plumbing.
M5 earns its place only if real cases cluster on these — otherwise it's the wrong cut.

| Axis | Question | Cheap end | Expensive end |
|---|---|---|---|
| **Duration** | within a turn, or outlives it? | one turn | open-ended |
| **LLM in the loop?** | does each step need the model? | yes (it's a turn) | no (a control loop the LLM only starts/stops) |
| **Feedback** | does it react to live perception each tick? | none (open-loop) | tight closed loop |
| **Body ownership** | does it hold the actuator over time? | no | yes (needs an arbiter) |
| **Latency** | how fast must each tick be? | seconds+ | ~frame-rate |
| **Concurrent w/ conversation?** | can you still talk to it while it runs? | no (owns the lane) | yes (parallel subsystem) |
| **Locality** | where does it run? | in the turn | in-process loop / separate process |

Plot the use cases:

| | Dur | LLM/tick | Feedback | Body | Latency | Concurrent | → model |
|---|---|---|---|---|---|---|---|
| UC1 ask | turn | yes | none | no | — | n/a | **M1** |
| UC2 render | long | no | none | no | loose | yes | **M3 task** |
| UC3 follow | long | **no** | **tight** | **yes** | **fast** | **yes** | **M5** |
| UC4 watch door | long | no | armed-trigger | maybe | medium | yes | M5 / M3 edge |
| UC6 news | long | no | none | no | loose | yes (interject) | M1-stream / M5-lite |
| UC7 remind | repeating | no | none | no | loose | yes | M3 cron |
| UC8 dance | short | no | none | yes (briefly) | — | maybe | M2 / bounded-M5 |
| UC9 timelapse | long | no | loose | yes (slow) | loose | yes | **M3 vs M5 boundary** |

**What the table says:** the cases that demand **{no-LLM-per-tick + tight feedback +
body ownership + fast latency + concurrent}** all collapse to one cell — that's UC3,
and it's genuinely unlike a task. So **M5 is a real cut**, but its defining trait is
narrower than "continuous": it's **a tight closed-loop controller that owns an
actuator, runs without the LLM per tick, and coexists with conversation.** Everything
that *doesn't* need tight feedback + body ownership (UC4 watch, UC7 remind, UC9
timelapse) can be an **in-process task** — so the task model may just need an
**in-process runner + a perception subscription**, not a whole new primitive.

→ Refined hypothesis: **two changes, not one.**
  (a) give **tasks an in-process mode** (+ perception subscribe) — covers UC4/7/9;
  (b) add **M5 only for tight body-control loops** (UC3) — the part tasks genuinely
      can't be, because of latency + actuator arbitration.

This is worth more poking before we commit either.

## What we have today (the contracts)

- **Tools are synchronous + fire-and-forget.** The brain publishes a directed
  `tool-call`; the device acks **instantly after dispatch** (it doesn't wait for the
  physical action). The result feeds back into the same LLM loop within the turn.
  ([server-brain-impl.md](decision-traces/server-brain-impl.md) §tool-call.)
- **Tasks are the async escape hatch.** A separate process (`child`/`tmux`),
  connected over the WS `tasks` topic, with a state machine (`running`/`stuck`/
  terminal) and signals (`notify`/`finish`/`errored`/`ask`→`provide_input`) that land
  **autonomous turns**. ([tasks.md](tasks.md).)
- **No parallel tool fan-out, no awaited async tool.** The loop dispatches tool calls
  and continues; nothing today issues N tools concurrently and joins them.
- **Continuous behaviors don't exist** as a first-class thing. The closest is a task,
  but tasks are out-of-process and update-oriented, not tight control loops.

## Open questions (to resolve before building M5/M6)

1. **Is an "async tool" just a task?** Partly. Both outlive the synchronous moment.
   The difference is *coupling + latency*: a task is out-of-process, coarse-grained,
   reports back occasionally; an async tool would be **in-process, awaited (or
   polled) within/around a turn**, fine-grained. "Follow me" wants neither exactly —
   it wants a **behavior** that owns the body + a perception subscription for as long
   as it runs. → Leaning: **introduce M5 (a behavior/mode primitive)** rather than
   stretch "tool" or "task" to cover it.

2. **Parallel tool calls — worth it?** Only when tools *wait* (slow reads). For
   fire-and-forget actuation, serial is fine (instant acks). → Likely **defer** until
   we have awaited tools that actually stack latency.

3. **Who owns the body during M5?** A "follow me" behavior and a normal `move` tool
   both drive the body. Need an **arbiter**: a behavior holds the body; a conflicting
   turn either queues, preempts, or is told "busy following." (Mirrors the
   conversation-state priority model — windows/sources with priority.)

4. **How does conversation coexist with a running behavior?** During "follow me" you
   can still talk. So a behavior must run **without holding the turn lane** — like a
   task, but in-process. The conversation state machine stays as-is; the behavior is a
   parallel subsystem the brain can start/stop via a tool.

5. **How does a behavior end?** Conditions: a voice command ("stop"), a predicate
   (you stopped moving for N s), a timeout, loss of the tracked subject, or a
   higher-priority interrupt. Same shape as the listening-window expiry/priority.

6. **Does M5 need the LLM in the loop every tick?** No — that's too slow/expensive.
   A behavior is a **cheap control loop** (perception → body) that runs without the
   LLM; the LLM only *starts/stops/reconfigures* it (and narrates). This is the key
   design line: **LLM sets policy, the behavior runs the loop.**

## Sketch: the "behavior" primitive (M5) — NOT built, for discussion

```
  user: "follow me"
      │  (request→response turn)
      ▼
  LLM calls a tool:  start_behavior("follow", { subject: "user" })
      │
      ▼
  Behavior runs IN-PROCESS, concurrent with conversation:
      every tick:  perception(subject position) → arbiter(body) → move
      ends on:     "stop" | subject lost | timeout | preempt
      │
      ├─ you can still talk → normal turns run alongside
      └─ on end → optional proactive turn ("okay, stopped following")
```

Contrast with a task: a behavior is **in-process, tight-loop, body-owning,
LLM-out-of-loop**; a task is **out-of-process, coarse, update-oriented**. The brain
starts both with a tool call; the difference is what runs after.
