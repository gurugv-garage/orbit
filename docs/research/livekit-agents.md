# LiveKit Agents as a conversation-orchestration base

Can we replace orbit's hand-rolled speech-to-speech orchestration (barge-in,
turn-taking, tool interleave, waiting/thinking states) with LiveKit Agents as a
hardened, Apache-2.0 base we plug our own STT/LLM/TTS/body into?

**TL;DR (after three source passes + a Pipecat head-to-head):** Adopt **LiveKit
Agents**, hosted **in-process in orbit-station via agents-js** (the JS port is
SOLID at Python parity — the "agents-js trails" worry was wrong), with the station
brain plugged in as an `llmNode`. The orchestration core is genuinely decoupled
from LiveKit's WebRTC server (custom audio I/O + node overrides are first-class),
and its barge-in / false-interruption machinery — pause-then-resume-or-kill — is
**structurally the fix for our recurring "pauses then CONTINUES" bug**
([[barge-clap-onset-fix]]) and its most mature feature. **Why LiveKit over
Pipecat:** Pipecat is more transport-agnostic and its local turn model is
friction-free (BSD, bundled), but it **hard-cancels on barge-in with no resume** —
exactly the behavior orbit needs and LiveKit alone ships. The one hard adoption
cost is the **AEC-safe audio downlink** (station synth → new track → phone decodes
into the existing `renderTtsPcm` loopback, *not* a second sink). Steal the
false-interruption arbiter into the current brain now regardless; steal Pipecat's
Smart-Turn + ✓/○/◐ turn-gate ideas for the [[should-respond-gate]] work.

## Things to note / takeaways

Each: claim — confidence (why) — **verdict** — cite.

1. **Their barge-in decouples pause from the stop-decision by design — exactly
   our unsolved bug.** An interruption *pauses* playout and arms a
   `false_interruption_timeout` (2.0s default); if no real transcript commits, it
   *resumes* (`AgentFalseInterruptionEvent(resumed=True)`); a real transcript
   commits a turn which *kills* the paused speech. Pause ≠ stop, arbitrated
   explicitly. — **high** (read the arbiter both directions) — **steal**
   (design), **watch** (adopt) — LK `voice/agent_activity.py:4117-4191` vs orbit
   `orbit-station/server/src/modules/brain/index.ts:745-748` (our "non-stop final
   still resumes" is the *symptom* their arbiter prevents;
   [docs/rca/2026-07-21-barge-stop-continues.md](../rca/2026-07-21-barge-stop-continues.md)).

2. **Interrupted assistant turns are truncated to what was actually heard.** On
   interruption the stored chat message is replaced with the playout-synced
   partial transcript + `interrupted=True` (realtime path truncates by
   `audio_end_ms`). We store the full intended text. — **high** — **steal** — LK
   `voice/agent_activity.py:2768-2803, 3288-3296` (orbit has no equivalent; brain
   history keeps the full `speak` text).

3. **Orchestration is decoupled from the WebRTC server — custom audio I/O is
   first-class.** `AgentSession.start(room=...)` is optional; if you set
   `session.input.audio` / `session.output.audio`, `RoomIO` is skipped entirely.
   `AudioInput` = async-iterator of frames; `AudioOutput` = implement
   `capture_frame`/`flush`/`clear_buffer` + `on_playback_finished(pos,
   interrupted, synchronized_transcript)`. — **high** (read the setters +
   skip-path) — **steal** — LK `voice/agent_session.py:832-866`,
   `voice/io.py:41-72, 131-290`.

4. **A custom `llm_node` can wrap our existing station brain — no pi/LLM object

## Supporting analysis

### How LiveKit orchestrates a turn (the part we'd be buying)

Three objects, all under `voice/`:

- **`AgentSession`** (`voice/agent_session.py:255`) — the facade: holds
  STT/LLM/TTS/VAD, chat history, user/agent state, the event emitter, and the
  public verbs `say()`, `generate_reply()`, `interrupt()`, `commit_user_turn()`.
- **`AgentActivity`** (`voice/agent_activity.py:166`) — the engine: owns the
  speech queue, the LLM→TTS pipeline, the realtime path, and **all interruption
  logic**. This is the hardened part.
- **`AudioRecognition`** (`voice/audio_recognition.py:228`) — the ear: pumps
  audio into VAD+STT, accumulates the turn transcript, runs end-of-utterance
  detection, and calls back into the engine via a `RecognitionHooks` protocol.

State vocabulary (`voice/events.py:311-312`): `UserState =
speaking|listening|away`, `AgentState =
initializing|idle|listening|thinking|speaking`. The turn flow:

1. VAD start-of-speech → user "speaking".
2. VAD end / STT final → `_run_eou_detection`.
3. Turn-detector (optional): if EOT probability is low, stretch endpointing
   delay from `min_delay` (0.5s) to `max_delay` (3.0s); then fire end-of-turn.
4. `on_end_of_turn` → interrupt current speech → `Agent.on_user_turn_completed`
   hook → generate reply (or reuse a **preemptive** generation started at interim
   time).
5. LLM streams (agent "thinking") → per-segment TTS → agent "speaking" on first
   played frame.
6. After playout: "thinking" again if tools pending, else "listening".

This is a **single, explicit FSM** with one owner — the exact opposite of our
distributed coupling (see below), and the structural reason their barge-in is
cleaner.

### Orbit today: no single orchestrator

Our coordination is split across (map cited to real source):

- `brain/index.ts` (`brainModule`) — the turn-loop closure: STT events → admit
  gate (`onAddressedFinal:721`, `trigger.via:915-928`) → dispatch; owns the
  **barge-hold registry** (`:637-665`) and **busy-queue** drain (`:674`).
- `brain/conversation-state.ts` — the per-dock FSM (`idle/listening/thinking/
  speaking/followup`), window admit rules, `SPEAK_MAX_MS` cap; single owner of
  the addressed decision.
- `brain/session.ts` (`DockBrainSession`) — per-dock executor: LLM step loop
  (`#runTurn:1010`), supersede/merge, TTS emit (`#speak:1805`), tool events
  (`:1518-1600`), autonomous queue.
- `brain/busy-queue.ts`, `brain/speech-gate.ts`, `brain/stop-intent.ts`,
  `brain/sentence.ts` — the satellite policies.
- `perception/index.ts` + `processors/speech-watch.ts` + `vad-endpoint.ts` — STT,
  VAD/energy floors, echo-gate, addressed shadow tap.
- `node-dock/.../agent/RemoteBrain.kt` — phone-side **pure renderer** of
  control frames + local stop + face-state; `tts/DockTts.kt` — on-device TTS +
  keepalive.

Barge/echo-gate/busy-queue/motion-speech timing is coupled across the index
closure, the perception echo-gate, `speech-gate.ts`, and the phone keepalive —
which is precisely why bugs in it keep needing RCAs
([[barge-clap-onset-fix]], [[busy-queue-black-hole]],
[[verify-user-perceivable-outcomes]]).

### What adoption would actually require

**Updated 2026-07-22 after a second source pass** (agents-js in-process viability
+ the station→phone downlink reality). Two findings reshaped this:

- **agents-js is SOLID for in-process use** — not the "trailing second-class port"
  the first pass implied. Every feature the decision gated on is present at Python
  parity in the cloned cut (`@livekit/agents` 1.5.4, HEAD `687ddf7`): room-optional
  manual audio I/O (`agent_session.ts:664,804` — `room?` optional, RoomIO branch
  skipped when no room), the pause-first false-interruption arbiter with a
  `resumed` flag (`agent_activity.ts:4448-4489`, `startFalseInterruptionTimer` →
  resume|kill → always emit `AgentFalseInterruption`), a **string-yielding
  `llmNode`** wired straight into TTS (`io.ts:22` `LLMNode` returns
  `ReadableStream<ChatChunk | string | FlushSentinel>`; consumed at
  `generation.ts:607-609`), the onnxruntime-node local turn-detector + streaming
  audio-EOT + dynamic endpointing (`plugins/livekit/src/turn_detector/*`,
  `voice/turn_config/endpointing.ts`), and async-tool `update()`/`foreground()`/
  filler (`run_context.ts:125-205`). No "not implemented" stubs in the voice
  orchestration path; the false-interruption/endpointing/EOU paths have dedicated
  tests. **So the brain can host LiveKit's orchestration IN-PROCESS in Node — no
  Python sidecar, no IPC.**
- **The downlink is the real cost, and AEC is the trap — but the trap is about
  playout routing, not synthesis location.** There is **no station→phone audio
  track today** — all station→phone comms are bus frames; the SFU is
  ingest-from-producer + fan-out-to-browser-viewers only and cannot originate a
  track; the phone's inbound-track handler is an empty stub (`MediaStreamer.kt:148`;
  `sfu.ts:35-54,113-130,245-246`; `tap.ts` observe-only). The AEC constraint:
  whatever the phone's speaker plays MUST also be handed to the on-device software
  AEC as the reference signal — which today happens because TTS PCM goes through
  `renderTtsPcm` → the WebRTC AEC *loopback* that both plays it out AND feeds it
  as the reference (`WebRtcAudio.kt:118-124,375-431`). **Synthesis can move to the
  station freely** — where the bytes are made is irrelevant. What breaks AEC is
  letting the downlink *play through a different sink* (e.g. naive
  `peerConnection.onTrack` auto-render): that sink is not the AEC reference, so the
  mic captures it and the always-on STT transcribes the dock talking to itself.
  Fix = route the downlink into the SAME loopback (see step 3).

Given both, the plan in rough effort order:

1. **Custom `llmNode` wrapping the station brain** — the brain's turn loop yields
   plain strings into LiveKit's TTS stream. Low-medium; the node contract accepts
   strings (`io.ts:22`). Body/motion tool-calls surface as LiveKit async tools.
2. **Station-side `AudioInput`** at the STT tap — wrap the existing 16 kHz Int16
   PCM (`speech-watch.ts:14,26-27`) as LiveKit `AudioFrame`s. Low risk; PCM
   already exists. This is also where the local turn-detector then earns its keep.
3. **`AudioOutput` = the downlink, done the AEC-safe way.** Move TTS synthesis
   station-side into a pause-capable `AudioOutput` (**must** pass
   `{ pause: true }` to `super()` or the false-interruption arbiter is silently
   off — `io.ts:124`, gate `agent_activity.ts:4438`); stream the synthesized PCM
   **down a new track**; on the phone, **decode it back to 16 kHz PCM-16 and feed
   the existing `renderTtsPcm` loopback** (`WebRtcAudio.kt:294-304`) rather than
   playing it through a second sink — so AEC, the speaking-clock, keepalives, and
   face/mood timing all keep working. Net-new on both ends (server-originated SFU
   track + dock-as-subscriber signaling + on-phone inbound decode path). This is
   the bulk of the work.
4. **Keep driving the bus control contract regardless.** `speak`/`tts-hold`/
   `cancelled`/`turn-status` (`RemoteBrain.kt:414-493,610-656,834-896`) drive
   face, gestures, and the echo-gate — none of that lives in the audio bytes, so
   the `AudioOutput` adapter must still emit them from LiveKit's playout/interrupt
   callbacks. One simplification falls out: with synthesis station-side, the
   station knows its own speaking edges from the `AudioOutput` clock and can drive
   the `SPEAK_ON_WINDOW` echo-gate directly instead of waiting for the phone's
   `speech-status` round-trip (`perception/index.ts:764-785`).
5. **Body/motion + addressed-gate stay ours.** LiveKit has no notion of them; the
   `[move]` anchor / speech-gate / addressed-classifier / trigger-provenance
   ([[turn-trigger-provenance]]) interleave with LiveKit's speech queue via
   `Agent` hooks + tool concurrency. This is where "plug it in" stops being
   plug-and-play — but it's additive, not a rewrite.

### License posture

All **code** in both repos is Apache-2.0 and self-hostable end-to-end (no Cloud
needed for the framework). Two carve-outs: (1) the turn-detector **model
weights** are under the proprietary `MODEL_LICENSE` — legal only *inside* the
LiveKit Agents framework, so adopting the framework keeps them usable but
cherry-picking the endpointer does not; (2) adaptive-interruption and
streaming-EOT-v1 assume LiveKit Cloud inference. The fully-local, fully-open
subset (VAD + STT-endpoint or text-turn-detector modes, custom plugins, manual
io) is enough to run a dock.

### Pipecat comparison (2026-07-22)

Source-read of `pipecat-ai/pipecat` 1.6.0 (`08e8715`, BSD-2-Clause, Python) as a
head-to-head. Pipecat's model is a **linear pipeline of `FrameProcessor`s** —
`transport.input() → STT → user context-aggregator → LLM → TTS →
transport.output() → assistant aggregator`. Reputation says turn-taking is
"emergent"; **that's now stale** — v1.6.0 has a real owned turn state machine
(`UserTurnController`, `turns/user_turn_controller.py:32`) with explicit
`_user_turn` state, a stop-timeout watchdog, and turn-lifecycle events. It's just
*embedded in the context-aggregator processor* and frame-driven, rather than
sitting above the media graph like LiveKit's `AgentActivity`.

Where each wins, on the axes that matter for orbit:

- **Bring-your-own transport / audio — Pipecat wins.** A transport is literally
  two `FrameProcessor`s (`base_transport.py:117,126`); push raw PCM in via
  `push_audio_frame(InputAudioRawFrame)` (`base_input.py:195`), consume
  `OutputAudioRawFrame` out (`base_output.py:580`), no WebRTC anywhere. A
  `LocalAudioTransport` already ships. This is a cleaner seam for orbit's
  "we own the phone↔station WS + PCM" reality than LiveKit's `AudioInput/Output`
  (which are also fine, but LiveKit's whole framing assumes a room even when
  bypassed).
- **Wrapping our brain — comparable, both clean.** Pipecat: subclass
  `LLMService`, consume `LLMContextFrame`, emit
  `LLMFullResponseStartFrame` → `LLMTextFrame`… → `LLMFullResponseEndFrame`
  (`services/openai/base_llm.py:550-575`). LiveKit: override `llmNode`, yield
  plain strings. Both let the pi loop stay the "LLM" with no rewrite. LiveKit's
  string-yield is marginally lower-ceremony; Pipecat's frame contract is more
  explicit.
- **Turn-model licensing — Pipecat wins decisively.** Pipecat's default semantic
  endpointer **Smart Turn v3** is BSD, **audio-input**, ONNX/CPU, **bundled in the
  pip package with zero download** (`audio/turn/smart_turn/local_smart_turn_v3.py`,
  model `data/smart-turn-v3.2-cpu.onnx`). LiveKit's turn-detector is Apache *code*
  but **text-input** with **separately-downloaded, framework-licensed weights**.
  No gated-repo friction with Pipecat (contrast [[local-speaker-id-diarization-test]]).
- **Barge-in / false-interruption resume — LiveKit wins, and this is the big
  one.** LiveKit has the **pause-then-resume-or-kill arbiter**: a barge-in pauses
  playout, and if it turns out to be noise (no committed transcript) it *resumes*.
  **Pipecat has no resume** — `InterruptionFrame` is a **hard cancel** that
  *discards the buffered playout* (`base_output.py:551-578,739-740`); its answer to
  false positives is **prevention** (gate the turn open on min-words / smart-turn /
  an LLM-completion verdict), not recovery. Orbit already runs a *polite-pause →
  resume* barge-in ([[barge-in-polite-pause]], [[barge-clap-onset-fix]]); on
  Pipecat we'd rebuild resume *against the grain* of a framework that throws the
  paused audio away. On LiveKit it's the framework's most mature feature.
- **Things Pipecat ships that orbit is separately building.** A local semantic
  endpointer (Smart Turn) and an **LLM turn-completion gate** (the LLM emits
  `✓ / ○ / ◐`; only `✓` finalizes the turn — `user_turn_strategies.py:103-154`) —
  direct analogs to our [[should-respond-gate]] and [[stt-eou-poc-parked]] work,
  worth studying even if we don't adopt Pipecat.
- **Maturity caveat — both churn, Pipecat harder.** Pipecat is mid-refactor toward
  2.0 (449 commits/30 days; the entire interruption API + `PipelineTask` were just
  renamed/removed; 15 core files carry `deprecated::`). Docs lag source badly —
  adoption means reading source, not tutorials. LiveKit's core is steadier.

**Net:** Pipecat is the more transport-agnostic framework with the friction-free
local turn model, and its pipeline seam fits orbit's PCM-over-WS reality slightly
more naturally. **But orbit's single most valuable want — a *hardened* barge-in
that pauses-then-resumes instead of cutting the bot on room noise — is exactly
what LiveKit has and Pipecat lacks by design.** Since barge-in is the source of
our recurring RCAs, that one axis outweighs Pipecat's transport elegance. Pick
LiveKit for the barge-in maturity; steal Pipecat's Smart-Turn and ✓/○/◐ turn-gate
ideas regardless of which framework (or neither) we adopt.

### Recommendation

**Updated 2026-07-22 (after the Pipecat comparison).** Framework choice:
**LiveKit over Pipecat** — the deciding axis is barge-in. Orbit's recurring pain
is a barge-in that must pause-then-resume rather than cut the bot on room noise;
LiveKit ships exactly that arbiter, Pipecat hard-cancels by design and would have
us rebuild resume against the grain. Pipecat's edges (more transport-agnostic,
BSD/bundled audio Smart-Turn model) are real but don't outweigh a mature,
built-in version of our #1 want. With the constraints the user relaxed (willing to
move audio station-side, accept the model license, adopt whichever runtime works),
the call then sharpens to a topology — and **agents-js in-process wins**:

- **Topology: LiveKit agents-js hosted in orbit-station, brain as `llmNode`.** The
  JS port is SOLID in-process, so this keeps everything in the Node process, no
  IPC, no second runtime. LiveKit owns the STT→turn→barge→TTS FSM (the one owned
  state machine we lack today); the brain plugs in as an `llmNode`; body/motion
  and the addressed-gate stay ours as `Agent` hooks + async tools. A Python
  sidecar is the fallback only if a JS integration snag appears — not the default.
- **The one hard piece is the AEC-safe downlink** (step 3 above): station-side
  synthesis → new downlink track → phone decodes → existing `renderTtsPcm`
  loopback. Do NOT let WebRTC render the downlink through a second sink; that
  reintroduces self-transcription. This is the critical path and the main risk.
