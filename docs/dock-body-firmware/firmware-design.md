# dock body ‚Äî firmware design (as-built)

The firmware that runs on the dock body's ESP32 (`node-dock/body-firmware/dock_body_v0`).
Native **ESP-IDF** (v5.5.x) via **PlatformIO**, C. This doc is the **software
architecture**; the physical build (boards, power, wiring) is
[hardware-design.md](hardware-design.md).

The body is a **pure station client**: it holds exactly one WebSocket ‚Äî to
orbit-station ‚Äî and speaks the `bodylink` protocol on it. There is no
phone‚Üîbody link and no on-device brain; the LLM loop and all motion decisions
live in orbit-station. The firmware's job is narrow and reflex-like: **receive
motion targets, drive servos smoothly, report state, stay connected, self-update.**

## 2. Boot sequence (`main.c`)

1. **NVS init** ‚Äî required by `esp_wifi`'s bookkeeping (even with `WIFI_STORAGE_RAM`);
   also holds the runtime dock binding (¬ß5).
2. **Servos** ‚Äî `servo_init_all()` brings PWM up and parks all 4 at center
   (1500 ¬µs) so the lines are settled before anything else.
3. **Motion runtime** ‚Äî `bl_motion_init()` (parks each *advertised* part at its
   home) + the **100 Hz `motion_task`**, which runs *regardless of connectivity*
   so the body settles transitions and holds pose even fully offline.
4. **Wi-Fi STA** ‚Äî on `GOT_IP`, the callback calls `station_link_start()` (the
   body's only socket).
5. `app_main` idles forever; all real work is on FreeRTOS tasks.

Compile flags for bring-up: `-DBL_SERVO_SWEEP` (wiring smoke test on boot),
`-DBL_DISABLE_WIFI` (servos-only, no link).

## 3. The station link (`station_link.c`)

The body is a WebSocket **client** of orbit-station (`/ws`). One socket, held for
the device's lifetime, with aggressive keepalive so a dead link is noticed and
re-dialed (this was a real bug ‚Äî see ¬ß8).

### 3.1 Handshake (hello v2)

On connect the body sends a `hello` frame identifying itself:

```jsonc
{ "t":"hello", "role":"device",
  "id":"body-<mac>",         // hardware instance = full 6-byte MAC; the station's binding key
  "dock":"<name>", "component":"body",   // omitted when UNCLAIMED (see ¬ß5)
  "kind":"dock-body-fw" | "dock-body-fw-c3",   // per-arch OTA target (¬ß6)
  "caps":["servo"],          // the motion executor routes by cap, never by name
  "label":"...", "build":<n> }            // build = the OTA gate (¬ß6)
```

Then it subscribes to the topics it needs:
`["bodylink","config","ota","station"]`. After the station's `welcome` it
publishes its capability **profile** (the parts + param specs it accepts).

### 3.2 Topic vocabulary (`bodylink`)

| Direction | kind | payload |
|---|---|---|
| station ‚Üí body | `command` | `set_target` ‚Äî `{ parts: { <part>: { pulse_width_us, duration_ms, velocity_us_per_sec_cap } } }` |
| station ‚Üí body | `ping` | `{ seq }` ‚Äî conn-health probe; body echoes immediately |
| body ‚Üí station | `profile` | capability profile (on connect) |
| body ‚Üí station | `state` | per-part current pulse widths (every 2 s) |
| body ‚Üí station | `applied` | ack of a command that actually changed state |
| body ‚Üí station | `heartbeat` | `{ ts, build, rssi, heap_free, reconnects }` (every 10 s) |
| body ‚Üí station | `event` / `error` | async notices (clamp, stale, out-of-range) |
| body ‚Üí station | `pong` | `{ seq }` ‚Äî echo of a `ping` |

The station is the **single master**: the brain's move tools and the console
sliders both route through the station's motion executor, which sends `command`
frames here. The body never initiates motion.

### 3.3 Heartbeat + link health

Every 10 s the body sends a `heartbeat` carrying the OTA `build` (keeps the
station's version view fresh without a reconnect) **plus cheap link-health
metrics**: `rssi` (live signal to the AP), `heap_free`, and `reconnects` (WS
connects since boot ‚Äî a rising value = a flapping link). The station folds these
onto the peer and surfaces them on the dock card / body console ‚Äî so "why is it
offline?" is a glance, not a debugging session (the lesson from the C3 range
debug, ¬ß8).

On demand, the console can also fire a **conn-health ping burst**: the station
sends N `ping { seq }` frames; the body echoes each as `pong { seq }` as fast as
it can. The **station** counts echoes (real packet-loss %) and times each
round-trip (RTT). Measuring from the station is deliberate ‚Äî it's the side that
can count, and it tests the exact station‚Üîbody link (not body‚ÜîAP). The body side
is intentionally trivial: reflect the seq, nothing else.

### 3.4 Staleness tripwire

Once the executor has driven the body at least once this connection, 30 s of
command silence *while connected* means the executor is broken ‚Äî the body emits
one `event:"stale"` and **holds pose** (never auto-homes; the brain homes
explicitly). Disarmed while offline.

## 4. Motion model (`bodylink_motion.c`)

A `set_target` names an absolute pulse width per part + a `duration_ms`. The
100 Hz tick interpolates from the current position to the target:

- **Ease-in/out (smoothstep)** ‚Äî an S-curve, so the servo accelerates off the
  start and decelerates into the target instead of slamming 0‚Üífull‚Üí0. Removes
  endpoint jerk (and the current spike that jerk causes on the C3's power rail).
  Peak speed is 1.5√ó the linear average.
- **Velocity cap** ‚Äî `velocity_us_per_sec_cap` is an **un-bypassable safety
  floor**. If a requested duration is too short for the travel (e.g. a full sweep
  in `duration_ms:0`), the firmware *stretches* it so peak speed stays under the
  cap ‚Äî protecting the gear train and the C3 power rail regardless of what the
  station sends. (Station-side pacing agrees with this cap; both must match or
  fast sequences collapse.)
- **Step-and-settle, preemptive** ‚Äî a new target captures the current position as
  the new start and restarts the clock; last-write-wins. `duration_ms:0` = snap.
- **Idempotency** ‚Äî a target that already matches (and is settled or moving there)
  is skipped, so `set_target` heartbeat-spam is cheap.

Clamp / out-of-range emits are published back as `event`s (diagnostics; the
executor pre-clamps so these only fire on profile drift).

## 5. Runtime dock binding

The body's dock is **not compiled in** ‚Äî it's learned from the station and
persisted to NVS (survives an app-partition reflash; the binding key is the full
MAC). See `docs/modules/runtime-dock-binding.md` for the full protocol.

- **Unclaimed** (empty NVS) ‚Üí hello omits `dock`/`component`; the station resolves
  the binding (or a console claim) and pushes it back in a `welcome`.
- **Claim** (`welcome{dock:"x"}`) ‚Üí persist to NVS + **reboot** for a clean reset
  (the one fail-proof way to guarantee no stale in-memory state carries into the
  new dock). The echoing welcome then matches ‚Üí no reboot loop.
- **Unclaim** (`welcome{dock:null}`) ‚Üí clear NVS + hold center (idle, no reboot);
  redial UNCLAIMED so we don't re-seed the old binding.

## 6. Dual-chip build + OTA

**Two boards, one tree.** The IDF target selects the servo PWM backend at compile
time (`servo.c`: S3‚ÜíMCPWM, C3‚ÜíLEDC) and the OTA `kind` (`station_link.c`:
S3‚Üí`dock-body-fw`, C3‚Üí`dock-body-fw-c3`). Per-chip `sdkconfig.defaults.<target>`
(merged by chip name) carry flash size + partition table + RF cal. PlatformIO
envs: `seeed_xiao_esp32s3` (default) and `esp32c3_mini`.

**OTA** (`station_ota.c`; full design in `docs/ota.md`):
- `BL_FW_BUILD` (monotonic int in `include/version.h`) is the **only** version on
  the wire and the OTA gate: station's build > device's build ‚Üí offer. Sent in
  hello *and* every heartbeat (self-healing version view).
- Dual-slot partitions + **rollback**: a freshly-OTA'd image boots in
  pending-verify; the bootloader auto-reverts unless the firmware calls
  `esp_ota_mark_app_valid_cancel_rollback()` ‚Äî which it only does *after*
  rejoining Wi-Fi AND reconnecting to the station (proof the new image works).
- First flash is cabled (`pio run -t upload`); every update after is network OTA
  from the station.
- Architecture-specific: an S3 (Xtensa) and C3 (RISC-V) image are never
  interchangeable, so each `kind` has a separate station artifact.

## 8. Notable failure modes fixed (history worth keeping)

- **WS link wedging on a half-open socket.** A blocked send (`portMAX_DELAY`) on a
  half-open socket held the single WS client lock, starving the reconnect/PONG
  watchdog ‚Äî the board stayed pingable but its WS was dead for minutes. Fixed with
  `CONFIG_ESP_WS_CLIENT_SEPARATE_TX_LOCK=y` + bounded send timeouts.
- **C3 Wi-Fi TX-power / rail-sag / range.** See hardware-design.md ¬ß4 and
  `progress.md` ¬ß"ESP32-C3 SuperMini Wi-Fi" ‚Äî a firmware TX-power cap
  (`BL_C3_TX_POWER_Q` in `wifi_sta.c`) paired with a hardware 3V3 bulk cap.
- **Servo thrash = unsoldered board**, not firmware ‚Äî intermittent contact chops
  PWM. (bring-up gotcha.)
