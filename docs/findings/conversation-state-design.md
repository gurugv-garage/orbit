## The founding problem (why this exists)

"Is the dock idle / listening / thinking / speaking / awaiting follow-up?" used to
have **three uncoordinated owners on two machines** — loose session flags
(`#speaking` cleared only by a phone frame → lost frame = stuck forever), a separate
`addressedLatch` Map, and the phone's `ListeningArbiter` acting as a second brain.
Three sources of truth drifted; every fix patched one desync and the next appeared
elsewhere.

## The principle (held ever since)

**The STATION owns conversation state. The phone is I/O.**
The phone reports raw events (tap, palm, tts start/end, connect, face
arrival/leave) and renders the `conversation` frames the station emits. It decides
nothing. The addressed decision IS the state machine: *an utterance is addressed
iff a listening/followup window is open when it lands* — no separate latch.

## The state machine (one per dock, on the `DockBrainSession`)

```
  IDLE ──tap/palm/wake──▶ LISTENING ──addressed utterance──▶ THINKING ──tts──▶ SPEAKING
   ▲   ◀── tap-off ─────────┘  ▲                               │                  │
   │                           │ STT interims HOLD the window  │ no speech →      │ tts-end
   │                           │ (speechInFlight, rolling)     │ settle → IDLE    ▼
   ├──── followup expires ◀────┴──────────────────── FOLLOWUP ◀───────────────────┘
   │
   └──── DISMISS ("stop"/"shut up"/"not talking to you") — from ANY mode → IDLE
```

Modes: **idle · listening · thinking · speaking · followup** (followup = auto
re-listen after every reply, `FOLLOWUP_MS`). All timings live in `ConvCfg`
(env-tunable, documented in-code).

### Window openers
- **tap** — toggle: opens from idle, closes an open window, INTERRUPTS a
  thinking/speaking reply (abort + fresh listening window).
- **palm / wake** (`tapOpen`) — open-only, never toggles off ("a palm always means
  listen to me"). Wake ("hey orbit") = `tapOpen` + a **canned spoken ack** (no LLM
  — Addendum 3 WI-4); the enricher's out-of-band wake fallback requires the robot's
  NAME in what it heard (Addendum 5.2).
- **followup** — opens after EVERY reply, trigger-agnostic (user/task/self).
- **face-arrival** — exists but default-OFF (`CONV_FACE_ARRIVAL=1` to enable);
  deliberate triggers only.

### Window holds — ONE ear (Addendum 8)
The **parakeet pipeline owns the window**: each STT interim is speech-in-flight
evidence → `speechInFlight()` holds an open window `INTERIM_HOLD_MS` (rolling).
Inbound phone `vad` frames are **ignored** — the phone's Silero VAD previously
shaped the window on its own clock and closed it UNDER in-flight utterances (the
"listening but ignored" class). Silero survives app-side as a UX sensor only
(instant face perk-up, screen wake). The transcriber that stamps the utterance
keeps its own door open, so the two-ear disagreement is structurally impossible.

### Window closers
- natural expiry (`LISTEN_MS` / `FOLLOWUP_MS`, prune-driven, ticked while timed);
- **tap-off** (deliberate; clamps the addressed grace to now);
- **spoken dismissal** — "stop / never mind / shut up / I'm not talking to you /
  go away" in ANY engaged mode → `dismiss()`: abort the turn, close every window,
  clear the busy queue (traced), → idle. Contrast **pause** ("wait" / "hold on")
  → abort + OPEN a listening window (stop talking and listen). Both via the
  narrow `classifyStopIntent` lexicon (content sentences pass through).
- safety prunes: `SPEAK_MAX_MS` (lost tts-end), `THINK_MAX_MS` (wedged turn),
  `noSpeechSettle` (a turn that ended silently → idle now, not 60s later);
- `reconcileConnected` (a fresh phone is idle).

### The addressed decision (`utteranceEnded`)
Window open now + `GRACE_MS` for the tap↔transcript ordering race; plus the
started-while-open interval rescue (`[openedAt − GRACE, lastWindowUntil]`).
**Deliberately NO trailing start-grace**: a startedAt landing after the close is
overheard — test A1h pins live numbers where a +600ms-late start was room speech;
indistinguishable by timestamp from a genuine late stamp. The interim hold removes
the early-close cause instead.

## What hangs off the transitions (the wave's additions)

- **Settle chokepoint**: entering `followup` or `idle` fires `onSettled` → the
  busy-queue drain. Everything heard while thinking/speaking is queued and either
  runs as ONE combined turn at settle (`drain:ran`) or is visibly traced
  (`skip:stale` / `skip:dismissed` / `skip:recording`) — zero silent outcomes
  (Addendum 3 WI-1 + review fixes).
- **Overheard framing**: turns born from the followup window or the busy drain
  carry `trigger.via` and may STAY SILENT if clearly not addressed — silence emits
  no reply, so no new followup window, so an ambient chain self-terminates
  (Addendum 5.1; the dock once spent 82 turns inside a meeting).
- **Audible interrupt**: `#interruptSpeech()` — tap/palm/dismiss abort the active
  turn OR, post-turn, send a bare `cancelled` frame that silences the phone's
  draining TTS queue (app build 31+; before that the speaker played out its buffer
  regardless — Addendum 5.4).

## Per-session, no persistence (unchanged decision)

State lives on the session object. Reconnect within the same open session →
re-send the current `conversation` frame (`resendConversation`); new session →
fresh idle state. No resume machinery, on purpose.

## Resolved questions (from the original proposal)

1. *Where does the priority arbiter live?* → Station owns state; the phone renders
   `conversation` frames instantly and keeps only sub-100ms UX reflexes local
   (Silero-driven face perk-up — sensor, not decider).
2. *Big-bang or staged?* → Staged, as planned: latch folded into the session first,
   then the phone became a renderer.
3. *Keep the `ListeningArbiter`?* → Its priority model became the station's
   `WindowSource` priorities (face < followup < tap); the phone-side arbiter is gone.
